* Overview

The daemon is a long-running background HTTP server that manages one or more
/repositories/. A repository is a directory tree on the filesystem whose
entries are tracked in a local SQLite database. The daemon exposes a REST API
consumed by the CLI and GUI.

The daemon is local-only (binds to =127.0.0.1= by default) and handles no
authentication; access control is left to the OS.

** Non-goals

- No built-in remote protocol: the daemon is local-only and makes no network
  calls. Cross-machine synchronisation is handled at the CLI/script level (e.g.
  loading a remote repo over an SSH-mounted filesystem into the local daemon).
- No authentication or multi-user access control.
- No conflict resolution between concurrent daemon instances on the same repo.

** Key invariants

- Every filesystem entry in a watched repository has exactly one corresponding
  metadata entry in the database.
- When a file or directory is deleted, its metadata entry is preserved; the
  =mf_path= field is set to =Nothing= rather than deleting the row.
- =set_field= replaces *all* rows for =(uuid, field_name)=, collapsing multi-map
  fields to a single value.
- A metadata entry whose =db_ids= contains more than one repository UUID (a
  /link entry/) can only be created or modified when every listed repository is
  simultaneously loaded in the same daemon instance.
- The =version= counter on a metadata entry is managed exclusively by the
  daemon; it is incremented on every write to =fields= and cannot be set
  directly by users.
- A repository database must be held under an exclusive SQLite lock for the
  entire lifetime of the daemon connection (=PRAGMA locking_mode = EXCLUSIVE=).
  This prevents a second daemon instance from loading the same repo
  simultaneously, which would cause version drift and watcher conflicts (e.g.
  one daemon's watcher updating =mf_path= and bumping =version= while the other
  is committing a sync).
- =TreeRef= references must form a forest (no cycles per =field_name=) with a
  maximum depth of 1000. The daemon rejects any write that would create a cycle
  or exceed this depth. The check walks the parent chain iteratively using the
  tree cache (fetching missing nodes from the DB step by step as needed).

* Data Model

** Repository

Persisted as =.metafolder/config.json= inside the watched directory.

| Field       | Type   | Description                            |
|-------------+--------+----------------------------------------|
| =repo_uuid= | UUID   | Stable identifier for this repository  |
| =name=      | string | Name of the repository                 |
| =port=      | int    | The port the daemon listen to          |
| =version=   | string | Config schema version                  |
| =root=      | path   | Absolute path to the watched directory |

** Metadata

An entry represents one tracked object (typically a file, but can be any
user-created record such as a tag, collection, or cross-repo link).

| Field     | Type       | Description                                              |
|-----------+------------+----------------------------------------------------------|
| =uuid=    | UUID       | Stable identifier for this entry                         |
| =db_ids=  | list<UUID> | Repositories that own this entry (usually one)           |
| =version= | u64        | Monotonic write counter, managed by the daemon           |
| =fields=  | list       | Ordered list of =Field= values                           |

Fields form a *multi-map*: multiple fields with the same name are allowed.

=db_ids= normally contains a single repository UUID (the owning repo). When it
contains two UUIDs the entry is a /link entry/ shared between two repositories
(see [[*Cross-repo Synchronisation]]).

By default, queries return only entries whose =db_ids= equals exactly
={current_repo}=. Entries with multiple =db_ids= are excluded unless explicitly
requested.

** Field and Value

A field is a =(name, value)= pair. Value is one of:

| Variant    | Rust type       | Semantics                                    |
|------------+-----------------+----------------------------------------------|
| =Nothing=  | —               | Explicit absence ("this field does not apply") |
| =String=   | =String=        | UTF-8 text                                   |
| =Int=      | =i64=           | 64-bit signed integer                        |
| =Float=    | =f64=           | 64-bit float                                 |
| =Bool=     | =bool=          |                                              |
| =DateTime= | =String=        | ISO-8601 datetime (=YYYY-MM-DDTHH:MM:SSZ=)   |
| =Ref=          | =UUID=         | Reference to another entry's UUID (same repo)              |
| =TreeRef=      | =(UUID?, String)= | Position in a named tree: =(parent_uuid, name)=             |
| =RefBase=      | =UUID=         | Reference to a repository UUID                             |
| =ExternalRef=  | =(UUID, UUID)= | Cross-repo reference: (repo UUID, entry UUID)              |

A =TreeRef= stores the position of an entry in a named tree as =(parent_uuid, name)=,
kept in the =value_uuid= and =value_name= columns of the =field= table
(see [[*SQLite Schema]]):

- =value_uuid=: UUID of the parent entry (zero UUID =x'00...00'= for root nodes).
- =value_name=: the name component contributed by this entry (e.g. filename, tag name).

Multiple independent trees can coexist within the same repository by using different
field names (e.g. =mf_path= for the filesystem tree, =parent= for a tag hierarchy).

Rename (same parent, new name): update =value_name= of one row — O(1).
Move (different parent): update =value_uuid= of one row — O(1).

Path-to-UUID resolution is accelerated by the *Tree Cache* (see [[*Tree Cache]]).

=Nothing= is distinct from the field simply not existing:
- field present with =Nothing= value → "this field does not apply" (IsAbsent)
- field absent entirely → "unknown" (IsUnknown)

All values serialize to JSON as ={"type": "<variant>", "value": <json_value>}=.

** File fingerprint

Used by reconcile and the watcher to detect identity across renames/moves.
*mtime is never used as a criterion* because it can change without the content
changing (e.g. =touch=, copies without =--preserve=).

Fingerprint is computed lazily in cascading steps — each step is only reached
when the previous one is insufficient:

1. *File size* (from =stat=, free during directory walk) — eliminates most
   non-matches without opening any file.
2. *Partial hash* — xxHash3 of the first 4 KB + last 4 KB of the file.
   Computed on first open; stored in the database alongside the entry.
3. *Full hash* — xxHash3 of the entire file content.
   Computed only when size and partial hash both match and a definitive
   identity check is required (e.g. deduplication confirmation).

Hash functions are provided by the =xxhash-rust= crate. For content-based
similarity between files that do not share an exact hash (e.g. slightly
modified files), the =fast-tlsh= crate provides a TLSH locality-sensitive hash
that can be used as an additional signal in the similarity phase (see
[[*File Similarity]]).

Files and directories are both tracked as first-class entries. Directories can
carry arbitrary metadata fields (e.g. =sync_method=) that files can inherit via
the =value_uuid= chain of their =mf_path= TreeRef field.

The root directory entry has =mf_path= with =value_uuid = x'00...00'= and
=value_name = ""=. All other entries reference their direct parent directory via
=value_uuid=. The root is looked up by
=WHERE field_name = 'path' AND value_uuid = x'00...00' AND value_name = ''=.

The stored fields for a file entry are:

| Field              | Value type | Protected | Description                                                                                                |
|--------------------+------------+-----------+------------------------------------------------------------------------------------------------------------|
| =mf_path=             | TreeRef    | yes       | Position in filesystem tree; =Nothing= if explicitly deleted (IsAbsent); stale TreeRef if location unconfirmed; absent if never assigned (IsUnknown) |
| =mf_type=          | String     | yes       | ="file"= or ="dir"=                                                                                        |
| =mf_size=          | Int        | yes       | File size in bytes at last watcher/reconcile update                                                        |
| =mf_mtime=         | DateTime   | yes       | Last content modification time (from =stat=)                                                               |
| =mf_btime=         | DateTime   | yes       | Creation time (from =statx= =btime=); absent if filesystem does not support it                             |
| =mf_permissions=   | String     | yes       | Unix permission bits as octal string, e.g. ="0644"=                                                        |
| =mf_uid=           | Int        | yes       | Owner user ID                                                                                              |
| =mf_gid=           | Int        | yes       | Owner group ID                                                                                             |
| =mf_mime=          | String     | yes       | MIME type from =libmagic= content analysis; absent if not yet computed                                     |
| =mf_partial_hash=  | String     | yes       | Hex xxHash3 of first+last 4 KB; absent if not yet computed or invalidated by =Modify=                      |
| =mf_full_hash=     | String     | yes       | Hex xxHash3 of full content; absent if not yet computed                                                    |

The =mf_path= TreeRef stores =(value_uuid, value_name)= for each entry.

The three states of the =mf_path= field have distinct semantics:
- =mf_path= present (valid TreeRef) → the entry is at that location.
- =mf_path= stale (TreeRef whose path no longer exists on disk) → location
  uncertain; reconcile will attempt to find a match via fingerprint or
  similarity and return it as a candidate.
- =mf_path= =Nothing= (IsAbsent) → file was explicitly removed (watcher
  =Remove= / =Rename(From)= event); no search needed. *Only the watcher sets
  =mf_path= to =Nothing=; reconcile never does.*
- =mf_path= absent (IsUnknown) → no path was ever assigned.

** Protected fields

All =mf_*= fields are managed by the daemon (watcher and reconcile) and are
*protected*. Attempting to write a protected field via any API write endpoint
(=PATCH=, =POST .../fields=, =PUT .../fields/:id=, =DELETE .../fields/:id=,
batch =set=) without the =force= flag returns =400 Bad Request=.

To override, add ="force": true= at the top level of the request body:

#+BEGIN_SRC json
{"name": "mf_permissions", "value": {"type": "string", "value": "0755"}, "force": true}
#+END_SRC

*Warning*: daemon operations (watcher events and reconcile) always overwrite
=mf_*= fields unconditionally, regardless of whether the value was previously
set with =force=. Using =force= is an explicit acknowledgement that the value
may be overwritten by the daemon at any time.

Typical legitimate use-cases for =force=: restoring a backed-up value after an
erroneous =chmod -R=; manually correcting a MIME type misdetected by =libmagic=;
resolving an =mf_path= conflict that the watcher cannot handle automatically.

Direct lookup by path uses the tree cache (see [[*Tree Cache]]); on cache
miss, the daemon resolves each component iteratively via the
=(field_name, value_uuid, value_name)= index.

Inherited property lookup (e.g. nearest ancestor with =sync_method=) is performed
in two steps: first, walk the parent chain via the tree cache (fetching missing
nodes from the DB step by step) to collect the ordered list of ancestor UUIDs;
then issue a single query =WHERE metadata_uuid IN (...)= to retrieve the desired
field across all ancestors at once, keeping the closest result.

** File Similarity

Used by reconcile as a fallback when fingerprint matching fails — either because
no =mf_partial_hash= is stored for the orphaned entry, or because the file was both
moved and modified (so hashes no longer match).

*Orphaned entry*: an entry whose =mf_path= TreeRef points to a filesystem location
that no longer exists on disk (stale path), and that was not matched by
fingerprint during the current reconcile run.

*Unmatched new file*: a file present on disk that has no corresponding database
entry and was not claimed by fingerprint matching.

Similarity between an orphaned entry and an unmatched new file is computed from
four signals:

| Signal           | Formula                                                                  | Weight |
|------------------+--------------------------------------------------------------------------+--------|
| =name_sim=       | Trigram Jaccard similarity of basename without extension                 |    0.5 |
| =ext_match=      | 1 if extensions are identical, 0 otherwise                               |    0.2 |
| =size_proximity= | =max(0, 1 − |sz_a − sz_b| / max(sz_a, sz_b))=; 0 if either size unknown |    0.2 |
| =path_sim=       | Longest common directory prefix length / max directory depth             |    0.1 |

Final score = weighted sum of the four signals (range [0, 1]).

*Candidate selection with MinHash banding* (crate: =probabilistic-collections=):

Repositories can contain millions of files; O(n²) pair evaluation is
impractical. Instead, a MinHash banding scheme is used to find candidate pairs
in O(n) expected time:

1. Each filename (basename without extension) is decomposed into character
   trigrams. A MinHash signature of =k= hash values is computed over the trigram
   set (=k= ≈ 128 in practice).
2. The signature is split into =b= bands of =r= rows each (=k = b × r=). For
   each band, the =r= hash values are combined into a single bucket key. Two
   files are candidate pairs if and only if they share at least one bucket —
   i.e. their MinHash signatures are identical on at least one band.
3. The probability that two files with Jaccard similarity =s= become candidates
   is =1 − (1 − sʳ)ᵇ=. Choosing =b= and =r= controls the recall/precision
   trade-off independently of =n=.

After candidate selection via banding, the full four-signal score is computed
only for surviving pairs. The size range pre-filter (range scan on a
size-sorted list, O(n log n)) is applied first to further reduce the candidate
set before banding.

Only pairs with =score ≥ threshold= are included in the reconcile response.

* SQLite Schema

Each repository uses its own =.metafolder/db.sqlite= file (WAL mode, foreign
keys enabled). Exception: when a repository is loaded from a network filesystem
(e.g. SSHFS), the daemon switches to =DELETE= journal mode (=PRAGMA
journal_mode=DELETE=) at connection time. WAL requires POSIX shared memory
(=.wal-shm=) which network filesystems do not support. This switch is applied
per-connection and does not modify the database permanently — the next local
connection resumes WAL mode normally. A benchmark should be run once sync is
implemented to verify that DELETE journal mode is acceptable for large syncs.

** UUID encoding

All UUIDs (entry UUIDs and repository UUIDs) are stored as =BLOB= (16 bytes,
raw binary). This applies uniformly to every UUID column in the schema.

The HTTP API encodes UUIDs as 32-char lowercase hex strings (no hyphens) for
readability and interoperability.

Rationale: 16-byte BLOBs enable fast binary comparison in SQLite indexes and
are the most compact representation. Hex encoding in the API is standard and
unambiguous.

The zero UUID (=x'00000000000000000000000000000000'=) is reserved as a sentinel
for =TreeRef= root nodes and must never be assigned to a real entry. UUID v4
has 122 random bits (2¹²² ≈ 5×10³⁶ possible values); the probability of
collision among 1 billion entries is ≈ 10⁻¹⁹ — negligible in practice.

#+BEGIN_SRC sql
CREATE TABLE metadata (
    uuid     BLOB    PRIMARY KEY NOT NULL,  -- 16-byte UUID
    version  INTEGER NOT NULL DEFAULT 0     -- incremented by daemon on every field write
);

-- One row per owning repository (usually one; two for link entries).
CREATE TABLE metadata_db (
    metadata_uuid  BLOB NOT NULL REFERENCES metadata(uuid) ON DELETE CASCADE,
    db_id          BLOB NOT NULL,           -- 16-byte repo UUID
    PRIMARY KEY (metadata_uuid, db_id)
);

CREATE INDEX idx_metadata_db ON metadata_db(db_id);

CREATE TABLE field (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    metadata_uuid   BLOB    NOT NULL REFERENCES metadata(uuid) ON DELETE CASCADE,
    field_name      TEXT    NOT NULL,
    value_type      TEXT    NOT NULL,  -- nothing | string | int | float | bool |
                                       -- datetime | ref |
                                       -- tree_ref | refbase | externalref
    value_text      TEXT,              -- string, datetime
    value_int       INTEGER,           -- int, bool (0/1), duration (ms)
    value_real      REAL,              -- float
    value_uuid      BLOB,              -- ref/externalref: entry UUID; tree_ref: parent UUID (zero UUID for root)
    value_ref_repo  BLOB,              -- externalref only: repo UUID
    value_name      TEXT               -- tree_ref: name component contributed by this entry
);

CREATE INDEX idx_field_entry        ON field(metadata_uuid, field_name);
CREATE INDEX idx_field_reverse      ON field(field_name, value_uuid, value_ref_repo)
    WHERE value_type IN ('ref', 'externalref');
CREATE UNIQUE INDEX idx_field_tree  ON field(field_name, value_uuid, value_name)
    WHERE value_type = 'tree_ref';
#+END_SRC

* HTTP API

The daemon listens on =127.0.0.1:7523= by default (configurable via =--port=).
All request and response bodies are JSON. Errors return a JSON object
={"error": "<message>"}= with an appropriate HTTP status code.

** Metadata CRUD

#+BEGIN_SRC
GET    /repos/:repo_uuid/metadata
POST   /repos/:repo_uuid/metadata
GET    /repos/:repo_uuid/metadata/:metadata_uuid
DELETE /repos/:repo_uuid/metadata/:metadata_uuid
PATCH  /repos/:repo_uuid/metadata/:metadata_uuid
POST   /repos/:repo_uuid/metadata/:metadata_uuid/fields

#+END_SRC

*** =GET /repos/:repo_uuid/metadata=

Returns all metadata UUIDs in the repository.

Response body: =["uuid1", "uuid2", ...]=

*** =POST /repos/:repo_uuid/metadata=

Creates a new metadata record with the given fields.

Request body:
#+BEGIN_SRC json
{
  "fields": [
    {"name": "path", "value": {"type": "tree_ref", "value": {"parent": "<parent_uuid>", "name": "bar.mp3"}}},
    {"name": "rating", "value": {"type": "int", "value": 5}}
  ]
}
#+END_SRC

For =TreeRef= fields, the client supplies =(parent_uuid, name)= directly.
The daemon validates that the given =parent_uuid= either is =null= (creating a
root node) or refers to an existing entry that has a =TreeRef= field of the same
name. The filesystem root (=value_uuid = x'00...00'=, =value_name = ""=) is
created at =repo init= time and is unique per field name (enforced by the
=idx_field_tree= UNIQUE index). Other roots can be created freely.

A metadata record may have multiple =mf_path= fields (multi-map), indicating the
record is accessible at several locations (e.g. a file with copies in different
directories). =set_field= on =mf_path= replaces all existing =mf_path= rows with a
single new one.

Response body: full metadata object with =id= on each field (see =GET .../metadata/:uuid= below).

*** =GET /repos/:repo_uuid/metadata/:metadata_uuid=

Returns a single metadata record with all its fields.

Response body:
#+BEGIN_SRC json
{
  "uuid": "...",
  "db_ids": ["<repo_uuid>"],
  "version": 3,
  "fields": [
    {"id": 42, "name": "path", "value": {"type": "tree_ref", "value": {"parent": "<parent_uuid>", "name": "bar.mp3"}}},
    {"id": 43, "name": "rating", "value": {"type": "int", "value": 5}}
  ]
}
#+END_SRC

Each field carries an =id= (the internal DB row id) that uniquely identifies
it within the repository. This id is used to target individual fields in
multi-map operations (see below).

*** =DELETE /repos/:repo_uuid/metadata/:metadata_uuid=

Deletes the metadata record and all its fields. Returns =204 No Content=.

*** =PATCH /repos/:repo_uuid/metadata/:metadata_uuid=

Replaces *all* existing rows for a given field name with a single new value
(=set_field= semantics — collapses multi-map to one entry).

Request body:
#+BEGIN_SRC json
{"name": "rating", "value": {"type": "int", "value": 9}}
#+END_SRC

Writing a protected field name without =force= returns =400=. See [[*Protected fields]].

Response body: updated full metadata object.

*** =POST /repos/:repo_uuid/metadata/:metadata_uuid/fields=

Appends one field value without touching existing rows for that name (multi-map
append). Use this to add a second =tag= without removing the first.

Request body:
#+BEGIN_SRC json
{"name": "tag", "value": {"type": "string", "value": "jazz"}}
#+END_SRC

Writing a protected field name without =force= returns =400=. See [[*Protected fields]].

Response body: updated full metadata object.

*** =PUT /repos/:repo_uuid/metadata/:metadata_uuid/fields/:field_id=

Replaces the single field row identified by =field_id=. Returns =404= if the id
does not belong to this metadata record.

Request body:
#+BEGIN_SRC json
{"value": {"type": "string", "value": "blues"}}
#+END_SRC

If the targeted field row is protected, omitting =force= returns =400=. See [[*Protected fields]].

Response body: updated full metadata object.

*** =DELETE /repos/:repo_uuid/metadata/:metadata_uuid/fields/:field_id=

Removes the single field row identified by =field_id=. Returns =404= if the id
does not belong to this metadata record.

If the targeted field row is protected, the request body must contain
={"force": true}=; otherwise returns =400=. Returns =204 No Content= on success.

** Query

#+BEGIN_SRC
POST /repos/:repo_uuid/query
#+END_SRC

Executes a query and returns matching entries.

Request body:
#+BEGIN_SRC json
{
  "query": <Query>,
  "select": ["path", "rating"]
}
#+END_SRC

=select= is optional:
- Absent → response is =["uuid1", "uuid2", ...]=
- =["field1", "field2", ...]= → response is a list of metadata objects
  containing only the requested fields.
- ="*"= → response is a list of full metadata objects (all fields).

Example response with =select=:
#+BEGIN_SRC json
[
  {
    "uuid": "...",
    "version": 3,
    "fields": [
      {"id": 42, "name": "path", "value": {"type": "tree_ref", "value": {"parent": "<parent_uuid>", "name": "bar.mp3"}}},
      {"id": 43, "name": "rating", "value": {"type": "int", "value": 5}}
    ]
  }
]
#+END_SRC

Fields absent from the entry (not present in the DB for that entry) are simply
omitted from the response — they are not included as =Nothing=.

** Reconcile

#+BEGIN_SRC
POST /repos/:repo_uuid/reconcile
#+END_SRC

Walks the repository root on the filesystem and synchronises the database.

Request body (all fields optional):
#+BEGIN_SRC json
{
  "threshold": 0.5
}
#+END_SRC

=threshold= (default 0.5, range [0, 1]) controls the minimum similarity score
for a pair to appear in =candidates=. A higher value reduces noise; a lower
value surfaces more speculative matches.

*Steps:*

1. Load all entries with a non-Nothing =mf_path= TreeRef from the DB; separate
   them into /active/ (path exists on disk) and /orphaned/ (path no longer
   exists on disk, i.e. stale path).
2. Walk the filesystem; collect all file and directory paths.
3. *Fingerprint phase* — for each orphaned entry that has a stored =mf_size= (and
   optionally =mf_partial_hash=), search unmatched filesystem paths for a
   fingerprint match (mf_size → mf_partial_hash). Matches → update =mf_path= TreeRef.
4. *Similarity phase* — for each still-unmatched orphaned entry and each still-
   unmatched new path, compute the similarity score (see [[*File Similarity]]).
   Pairs with =score ≥ threshold= are collected as candidates. New files that
   appear as a candidate for at least one orphaned entry are *not* auto-created
   as new entries; they wait for explicit confirmation.
5. *Create* — new files not claimed by fingerprint or similarity → create entry
   with =mf_path= TreeRef and all =mf_*= fields (=mf_size=, =mf_mtime=,
   =mf_permissions=, =mf_uid=, =mf_gid=, =mf_btime= if available,
   =mf_type=, =mf_mime= if computed); create missing parent directory entries
   if needed.
   For fingerprint-matched or similarity-confirmed entries, refresh their =mf_*=
   fields from the new file's =stat= result.
6. Orphaned entries not matched by fingerprint and not present in any candidate
   pair keep their stale =mf_path= TreeRef unchanged. *Reconcile never sets =mf_path=
   to =Nothing=.*

*Directory identification*: directories have no fingerprint and are matched only
by their path string. A directory entry whose path no longer exists on disk
is left with its stale path (like any other orphan); it may appear in candidates
if a new directory with a similar name exists.

*Cascading Nothing*: whenever any entry's =mf_path= is set to =Nothing= (by the
watcher or manually), all entries whose =value_uuid=
(directly or transitively, for =field_name = 'path'=) points to this entry are
also set to =Nothing= in the same transaction. Descendants are collected
iteratively using the tree cache (fetching missing nodes from the DB step by
step), then cleared in a single batched UPDATE. No entry can have a valid path
if one of its ancestors is gone.

All creates and updates execute inside a single transaction.

*Confirming a candidate*: the client sends a =PATCH= request to update the
orphaned entry's =mf_path= field to the chosen new location. The new file is then
considered matched; if the client calls =reconcile= again without the candidate
having been confirmed, the same candidate will be returned again.

Response body:
#+BEGIN_SRC json
{
  "created": 2,
  "moved": 1,
  "candidates": [
    {
      "entry_uuid": "abc...",
      "stale_path": "/music/jazz/old.mp3",
      "matches": [
        {"path": "/archive/old_renamed.mp3", "score": 0.82},
        {"path": "/backup/file.mp3",         "score": 0.51}
      ]
    }
  ]
}
#+END_SRC

* File Watcher

The daemon watches each loaded repository root with inotify (Linux). *The
watcher is the only component that sets =mf_path= to =Nothing=*; reconcile and
manual writes leave =mf_path= stale rather than setting it to =Nothing=. Events
outside =.metafolder/= are processed as follows:

| inotify event   | Action                                                                                                   |
|-----------------+----------------------------------------------------------------------------------------------------------|
| =Create=        | Look up parent UUID via tree cache; create entry with =mf_path= TreeRef and all =mf_*= fields               |
| =Remove=        | Set =mf_path= to =Nothing= (entry preserved)                                                                |
| =Rename(Both)=  | Update =mf_path= TreeRef: =value_name= (rename) or =value_uuid= (move) — O(1)                              |
| =Rename(From)=  | Set =mf_path= to =Nothing= (rename target not yet known)                                                    |
| =Rename(To)=    | Create entry if no matching fingerprint; else update =mf_path= TreeRef and refresh =mf_*= fields            |
| =Modify=        | Refresh =mf_size=, =mf_mtime=, =mf_permissions=, =mf_uid=, =mf_gid= from =stat=; delete =mf_partial_hash= and =mf_full_hash= (content changed); =mf_mime= is also deleted and recomputed lazily |

=Modify= events are *debounced*: the daemon waits until no further =Modify=
event has been received for the same file within 200 ms before processing.
This avoids redundant stat/DB writes during a long sequential write.

Parent directory lookup on =Create= or =Rename(To)= uses the tree cache
(see [[*Tree Cache]]); on cache miss, falls back to iterative indexed queries
on =(field_name, value_uuid, value_name)=.

Events inside =.metafolder/= are always ignored.

* Tree Cache

The daemon maintains one in-memory tree per loaded repository for resolving path
strings to entry UUIDs without issuing recursive SQL queries. The cache is shared
across all =TreeRef= field names within a repository; the field name forms the
first level of the tree.

** Path string format

A path string is the ordered list of =value_name= components from the root down
to an entry, joined by =/=. There is no trailing =/=.

The root's own =value_name= determines whether paths in the tree have a leading
=/=:

| Tree type                    | Entry                     | Path string        |
|------------------------------+---------------------------+--------------------|
| filesystem (root: name=""=, parent=NULL)   | root           | =""=               |
| filesystem                   | child ="projects"= of root | ="/projects"=      |
| filesystem                   | child ="rust"= of above   | ="/projects/rust"= |
| tags (root: name="tag1", parent=NULL)     | root ="tag1"= | ="tag1"=           |
| tags                         | child ="tag2"= of ="tag1"=| ="tag1/tag2"=      |

For the filesystem tree the root has =value_name = ""=, so all paths naturally
start with =/=. For other trees (e.g. tags) the roots have non-empty names and
there is no leading =/=. Multiple roots are allowed within the same field name.
The uniqueness of the filesystem root =(field_name='path', value_uuid=x'00...00',
value_name='')= is enforced by the =idx_field_tree= UNIQUE index.

** Structure

#+BEGIN_SRC
root (virtual)
├── "path"
│   └── [root entry, parent=NULL]
│       └── "music"  → uuid_music
│           └── "jazz"  → uuid_jazz
│               └── "file.mp3"  → uuid_file
└── "parent"
    └── "animaux"  → uuid_animaux
        └── "félins"  → uuid_felins
#+END_SRC

Each node stores:
- =children=: map of name → child node
- =uuid=: the entry UUID at this position (=None= for the virtual root and field-name nodes)
- =last_used=: logical clock updated on every traversal of this node
- =heap_handle=: reference into the LRU heap (leaf nodes only)

** Eviction

A *min-heap of leaf nodes* (nodes with no children), keyed by =last_used=,
drives eviction when the cache exceeds its configured size limit (default: 100 MB).

On eviction of a leaf:
1. Remove the leaf node from the tree and the heap.
2. If its parent has no remaining children, the parent becomes a new leaf —
   insert it into the heap with its own =last_used=.

Only leaves are evicted; internal nodes are kept as long as they have children.
Because every traversal updates =last_used= on all nodes along the path, a newly
promoted leaf always carries the correct recency timestamp.

** Operations

| Operation              | Complexity       | Notes                                                        |
|------------------------+------------------+--------------------------------------------------------------|
| Path → UUID (hit)      | O(depth)         | Traverse tree; update =last_used= on each node              |
| Path → UUID (miss)     | O(depth × log n) | Iterative indexed DB queries; populate tree on the way      |
| Rename (same parent)   | O(1)             | Update label of one node                                     |
| Move (different parent)| O(1)             | Detach node, reattach under new parent; descendants follow   |
| Evict leaf             | O(log k)         | Heap pop; k = number of leaves in cache                     |

** Startup

The cache starts empty; entries are populated lazily on first access. No
pre-loading from the database is required.

* Query Language

Queries are represented as a recursive JSON structure (the =Query= enum). The
CLI provides a DSL that compiles to this representation.

** Variants

| Variant             | JSON =type= key       | Description                                    |
|---------------------+-----------------------+------------------------------------------------|
| IsPresent           | =is_present=          | Field exists and is not =Nothing=              |
| IsAbsent            | =is_absent=           | Field exists and is =Nothing=                  |
| IsUnknown           | =is_unknown=          | Field does not exist at all                    |
| Eq / Neq            | =eq= / =neq=          | Equality / inequality comparison               |
| Lt / Lte / Gt / Gte | =lt= / =lte= / ...    | Ordered comparison                             |
| And                 | =and=                 | All operands must match                        |
| Or                  | =or=                  | At least one operand must match                |
| Not                 | =not=                 | Operand must not match                         |
| Follows             | =follows=             | =Ref= or =TreeRef= field: direct referent matches condition    |
| FollowsTransitive   | =follows_transitive=  | =TreeRef= only: entry is a descendant of the entry at the given path |
| Matches             | =matches=             | String field matches a regex pattern                           |

** Compilation

Queries are compiled to a CTE chain in SQL. Each query node becomes one CTE;
the final SELECT is filtered via =metadata_db= to isolate the repository and
exclude link entries by default.

Exception: =FollowsTransitive= on a =TreeRef= field is executed as a hybrid
operation — the daemon first collects the descendant UUID set iteratively via
the tree cache, then injects it as an =IN= clause into the surrounding SQL query
rather than emitting a CTE.

=Follows= on a =Ref= field takes a condition (sub-query) as right-hand side and
matches entries whose =value_uuid= points to an entry satisfying that condition.
On a =TreeRef= field, the right-hand side is a path string (e.g. ="/music/jazz"=)
resolved to a UUID via the tree cache; it matches entries whose direct parent
(=value_uuid=) is that entry. DSL syntax: =field -> "path"=.

=FollowsTransitive= is restricted to =TreeRef= fields; applying it to a =Ref=
field is rejected with =400=. The right-hand side is a path string (e.g.
="/projects"=) that is resolved to a UUID via the tree cache. All descendants of
that entry are then collected iteratively via the tree cache (fetching missing
nodes from the DB step by step), and the resulting UUID set is used to filter
the query results.

=Matches= uses a SQLite user-defined function =REGEXP= (registered at connection
setup) backed by the =regex= crate. On a =TreeRef= field, =MATCHES= applies the
regex to =value_name= (the single name component).

* Cross-repo Synchronisation

The daemon provides low-level primitives for cross-repo sync. Orchestration
(deciding what to link, in which order, with which method) lives in CLI
commands and user scripts.

** Design principles

- The daemon never touches files. File transfers are always performed by the
  CLI command or script.
- Sync intentions (which entries to link, with what method) are expressed as
  queries in the sync script, not stored in the database.
- Both repos must be loaded in the same daemon instance to create or modify
  link entries or to commit a sync.
- For cross-machine sync, the recommended approach is to mount the remote
  repository root over SSH (e.g. SSHFS) and load it into the local daemon.
  The daemon itself makes no network calls.

*** Directory linking during sync

Links are always UUID-based. For directory entries, the initial link is
established by matching reconstructed path strings across repos:

1. A directory exists in both repos with the same reconstructed path string
   (assembled from =value_name= components via =value_uuid= chain) →
   create a link between them.
2. A directory exists in repo A but has no corresponding entry (same path) in
   repo B → create the directory entry in repo B, then link it.
3. A linked directory is later moved on one side → the link persists by UUID;
   the path divergence is detectable as a version change and treated as a
   field conflict on the =mf_path= field.

** Link entries

A link entry is a metadata entry with =db_ids = [repo_A_uuid, repo_B_uuid]=.
It records that one entry in repo A corresponds to one entry in repo B.

All fields are managed by the sync script, not by the daemon:

| Field              | Type        | Description                                              |
|--------------------+-------------+----------------------------------------------------------|
| =link_a=           | ExternalRef | (repo A UUID, entry UUID) — self-contained cross-repo ref |
| =link_b=           | ExternalRef | (repo B UUID, entry UUID) — self-contained cross-repo ref |
| =snapshot=         | Ref         | Shared metadata entry capturing the common state at last sync |
| =version_a=        | Int         | =E_A.version= recorded at last sync                     |
| =version_b=        | Int         | =E_B.version= recorded at last sync                     |
| =last_sync_method= | String      | Method used during the last sync (informational)         |

Link entries are stored identically in both repos (same UUID, same fields,
same =version=). They are hidden from normal queries (=db_ids ≠ {current_repo}=)
but can be queried explicitly.

** Version-based change detection

After a successful sync of entries =E_A= (repo A) and =E_B= (repo B), the
sync script writes the current versions of both entries into the link:

- =version_a= ← =E_A.version= at time of sync
- =version_b= ← =E_B.version= at time of sync

On the next sync, for each linked pair:

| =E_A.version= vs =version_a= | =E_B.version= vs =version_b= | Meaning          | Action          |
|------------------------------+------------------------------+------------------+-----------------|
| equal                        | equal                        | No changes       | Nothing to do   |
| greater                      | equal                        | Only A changed   | Propagate A → B |
| equal                        | greater                      | Only B changed   | Propagate B → A |
| greater                      | greater                      | Both changed     | Conflict        |

Conflict resolution is left to the sync command (user prompt or declared
policy in the script).

When a link is created for the first time, if =E_A= and =E_B= already have
divergent fields, this is treated as an immediate conflict to resolve before
=version_a=, =version_b=, and =snapshot= can be recorded.

The =snapshot= entry has =db_ids = [repo_A, repo_B]= and captures the common
field state after each successful sync. It is updated at every sync-commit
alongside =version_a= and =version_b=. During conflict resolution (both sides
changed), the sync command computes:

- Fields changed only in A = diff(E_A, snapshot)
- Fields changed only in B = diff(E_B, snapshot)
- Fields changed in both   = conflict → manual resolution

Fields changed on only one side are propagated automatically.

*** Ref and TreeRef fields in the snapshot

For non-Ref fields, E_A and E_B hold identical values after sync, so the
snapshot is unambiguous. For =Ref= and =TreeRef= fields, the same logical
reference has different local UUIDs in each repo.

The snapshot always stores these values from the perspective of =link_a='s
repository (repo A). This is a fixed canonical choice that does not change
over time:

- =Ref=: stores repo A's local UUID.
- =TreeRef=: stores =(value_uuid_A, name)= using repo A's local UUID for the
  parent. The =name= component is portable (identical across repos).

Consequences:
- =diff(E_A, snapshot)=: direct field comparison, no translation needed.
- =diff(E_B, snapshot)=:
  - For =Ref=: translate the snapshot UUID A→B via the link table.
  - For =TreeRef=: translate =value_uuid= A→B via the link table;
    =value_name= can be compared directly without translation.
  - Translation found → compare translated value with E_B's value.
  - Translation not found (referenced entry not linked) → conservative
    treatment: flag as potential conflict and warn.

** Watcher suspension during sync

The file watcher must be suspended on both repos during the execution phase
of a sync (and similarly for rollback and other multi-step operations).
Without suspension, watcher events (e.g. a file rename) would increment
=version= on an entry mid-sync, corrupting change detection.

The daemon exposes two distinct watcher control modes:

*** pause / resume (buffered)

The daemon continues reading the inotify file descriptor but queues events
in an in-memory buffer rather than processing them. The kernel inotify
queue has a fixed limit (=/proc/sys/fs/inotify/max_queued_events=, default
16384); continuing to read prevents it from overflowing.

On resume, buffered events are replayed against the current database state:

- Events for paths already consistent with the database (e.g. sync updated
  the metadata before resuming) → no-op.
- Events for paths that diverge from the database (user-driven changes
  during the pause) → processed normally (path update, =version= bump, etc.).

Used by: sync execution phase, rollback, and other multi-step operations
where filesystem changes should not interrupt the operation but must not
be silently discarded.

Degraded modes:
- *Buffer overflow*: daemon detects overflow, falls back to =reconcile= on
  resume.
- *Crash or SIGKILL*: buffer lost; =reconcile= required before retrying.

*** stop / start (clean slate)

The daemon stops reading the inotify file descriptor entirely. No events
are buffered. On start, a fresh watcher is registered with no backlog to
replay.

Used by: bulk imports or any operation where the user explicitly wants all
filesystem changes during the stopped window to be ignored, and will run
=reconcile= manually afterward if needed.

** Entry deletion propagation

- =mf_path= set to =Nothing= (file deleted, entry preserved) → link is kept;
  the entry still exists and its metadata remains meaningful.
- Entry fully deleted → the sync command propagates the deletion to the
  corresponding entry in the other repo, then deletes the link.

** Link deletion

When a link entry is deleted, its associated =snapshot= entry must be deleted
at the same time (both repos loaded). The snapshot has no other owner and
becomes unreachable once the link is gone.

** Link discovery on repo load

When a repo is loaded into the daemon, the daemon scans its =metadata_db=
table for entries whose =db_ids= includes the newly loaded repo's UUID and at
least one other already-loaded repo's UUID. These are cross-repo link (and
snapshot) entries.

For each such entry, the daemon runs the crash-recovery check (see below):
compare the =version= of each copy across repos and overwrite the older copy
with the newer one if they diverge.

** Link entry consistency after crash

A sync-commit writes to the link entry in both repos sequentially. If the
daemon crashes between the two writes, the two copies of the link entry
diverge.

Recovery is automatic: when both repos are loaded, the daemon compares the
=version= system counter of each copy of every link entry (same UUID, both
repos). The copy with the higher =version= is authoritative and overwrites
the other. This works because the daemon guarantees that a higher =version=
always reflects a more recent completed write within a single repo.

** Two-phase sync process

Sync is split into two distinct phases:

1. *Planning* (watcher running): query both repos, compute all operations to
   perform (file transfers, metadata propagations, conflicts). Produce a plan
   that includes, for each repo involved, the =PRAGMA data_version= value at
   planning time. The user can inspect the plan before proceeding.

2. *Execution* (watcher paused): validate that =PRAGMA data_version= still
   matches the value recorded at planning time for each repo. If any DB has
   changed — even by a single write — the plan is invalidated and the user
   must re-plan. This is intentional: detecting *what* changed is complex,
   and even a single change could mean the plan would overwrite recent
   modifications. If valid, execute operations in order: file transfers first
   (all idempotent), then metadata commits in batches.

After execution: resume watcher + run =reconcile= to catch any filesystem
changes that occurred during the paused window.

** Sync intentions

Sync intentions are declared in the sync script as a mapping from sync method
to a query. Example:

#+BEGIN_SRC
rsync  ←  tag = "sync_pc_b"
git    ←  path →* "/projects"
rsync  ←  tag = "image" AND tag = "family"
#+END_SRC

At sync time the script:

1. Evaluates each query on both repos to determine which entries should be
   linked (and with what method).
2. Compares the result with the existing link entries to identify links to
   create, delete, or update.
3. Resolves any intent conflicts between the two repos (e.g. different methods
   declared on each side).
4. Executes file transfers for file entries (using the declared method).
5. Propagates changed fields for all linked entries (metadata sync).
6. Calls the sync-commit primitive to set common versions and record
   =last_sync_method= on each updated link.

** Ref translation during sync

=Ref= and =TreeRef= fields store local UUIDs that must be translated when
writing to the target repo.

Sync order for a given repo pair:

1. Update the link table first (create, delete, update =last_sync_method=).
2. Sync entry fields:
   - Any UUID value that is =NULL= / =Nothing= is written as-is — no translation
     needed.
   - For each =Ref= field: look up the link table to find the corresponding
     UUID in the target repo.
   - For each =TreeRef= field: translate =value_uuid= A→B via the link
     table; =value_name= is portable and written as-is.
   - Translation found → write the translated value.
   - Translation not found → skip the field and emit a warning. Since the link
     table is complete at this point, a missing translation means the
     referenced entry is genuinely out of sync scope.

=Ref= values have no foreign-key constraint on =value_uuid= (see schema). The
system tolerates dangling refs: =Follows= on a dangling =Ref= simply returns no
results. Circular references (A → B → A) are likewise permitted; =Follows= is
non-recursive so cycles are not a concern.

** Required daemon primitives

The following endpoints are needed to support sync scripts (exact HTTP shapes
TBD):

- List link entries for a given pair of loaded repos.
- Detect link candidates by matching fingerprint (=mf_partial_hash= / =mf_full_hash=)
  across two loaded repos.
- Create / delete a link entry (both repos must be loaded).
- Pause / resume the file watcher (buffered mode).
- Stop / start the file watcher (clean-slate mode).
- Query sync status: for each link, return the change/conflict state derived
  from =version= comparisons.
- Sync-commit: write =version_a=, =version_b=, and =last_sync_method= to the
  link entry in both repos.

* Performance concerns (sync)

Questions to resolve before implementing sync:

1. *Sequential writes without batching*: the sync-commit writes to the link and
   snapshot in both databases, one record at a time. Over SSHFS each write is
   slow. Accumulating changes in batch transactions reduces this cost at the
   price of more re-work after a crash (at most one batch to redo). The optimal
   batch size is to be determined by benchmark. For file entries, batches should
   be smaller than for pure metadata entries, since each batch corresponds to
   file transfers (idempotent, but slow to re-transfer if the batch is large).

2. *Watcher pause duration*: resolved by buffering (see §Watcher suspension
   during sync). The pause covers only the execution phase; events are buffered
   rather than lost and replayed intelligently on resume. =reconcile= remains
   the fallback in case of overflow or crash.

* Open Questions

** Full-text / regex index
=MATCHES= currently performs a full table scan with the REGEXP UDF (~650 ms for
50 k entries). Options: FTS5 virtual table, or a precomputed hash index for
exact-match patterns.

** Pagination
=GET /metadata= and =POST /query= return all results in one response. For very
large repositories this may be unacceptable. A cursor-based pagination API
should be designed.

** Watcher reliability on edge cases
Behaviour when the watched root is unmounted, moved, or deleted while the
daemon is running is undefined. The daemon should detect this and mark the
repository as unavailable.

** Daemon lifecycle
No mechanism yet for cleanly unloading a repository (stopping its watcher,
releasing its DB connection) without restarting the whole daemon.
