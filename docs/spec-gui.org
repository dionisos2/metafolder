#+TITLE: Metafolder — GUI Specification
#+OPTIONS: toc:2 num:t

* Overview

The GUI is a graphical interface to the Metafolder daemon. It exposes the full
CLI surface in visual form: repository management (init/load/list), entry
CRUD, query execution, reconcile, and event log inspection. It communicates
with an already-running daemon over HTTP and does not manage the daemon's
lifecycle.

Four cross-cutting design principles govern the GUI:
- *Keyboard-first*: every command is reachable from the keyboard; all
  keybindings are configurable.
- *Command-driven*: a persistent status bar shows notifications; a command
  input accepts named commands and shell invocations.
- *Scriptable*: the GUI exposes its own local HTTP API so that external scripts
  can drive panels and wait for user input.
- *Themeable*: appearance is controlled by a CSS file loaded at startup.

The GUI is built with *Tauri*: a Rust backend communicating with a system
WebView frontend. The JavaScript/TypeScript frontend framework is not yet
decided.

* Concepts

** Workspace

The complete state of one tab: panel count (1 or 2), each panel's current
panel type and its internal state (scroll position, query draft, display mode,
etc.), the panel width ratio, the command input content, and the workspace
variables. Switching tabs swaps workspaces atomically; no state is lost. Each
workspace is independent.

** Panel

A resizable content area belonging to a workspace. Each workspace displays one
or two panels side by side (single panel at creation; the user can open a
second). Panels are separated by a draggable divider; their relative width
ratio is part of the workspace state.

** Panel type

The content/functionality kind displayed inside a panel. Each panel type is
implemented as a directory containing an =index.html= entry point; built-in
types ship as default examples in the user's config directory so they can be
inspected and modified. The panel type name is the directory name.

Built-in panel types:

| Name           | Description                                                  |
|----------------+--------------------------------------------------------------|
| =entry-list=   | Metadata entries filtered by an embedded query; primary selection source |
| =entry-detail= | All fields of the selected entry; follows =selected_entry=   |
| =file=         | Preview of the selected entry's file (image, audio, video, text); follows =selected_entry= |
| =message=      | Persistent log of status bar messages, shell output, and reconcile results |
| =log=          | Event log (revisions and operations)                         |
| =repos=        | Repository list, init, load                                  |

Users can add custom panel types by creating a new directory under
=~/.config/metafolder-gui/panel-types/<name>/=. Custom types appear alongside
built-in types in the panel type selector.

** Display mode

The visual representation used within a panel type that supports several
layouts. Stored as part of the panel's state in the workspace.

The =entry-list= panel type supports at minimum:
- =table= — tabular list with configurable columns.
- =grid= — thumbnail grid, useful for image collections.

** Workspace variables

A reactive, per-workspace key-value store shared across all panels in the
workspace. Panel types read and write these variables to communicate without
being directly coupled to each other.

Standard variables:

| Key                | Type                          | Description                        |
|--------------------+-------------------------------+------------------------------------|
| =active_repo=      | =string \vert null=           | UUID of the active repository      |
| =selected_entry=   | =\{uuid, repo\} \vert null=   | Currently highlighted entry        |
| =selected_entries= | =string[]=                    | UUIDs of all selected entries      |

Panel types may define their own variables using namespaced keys (e.g.
=my-panel:current-node=).

** Command

A named, invocable operation with an optional parameter string. Commands are
registered by panel types (built-in or custom) and are available in the
command input with autocomplete. Every user-visible behaviour is a command.
Keybindings map key combos to command invocation strings (command name plus
optional parameters).

Each command has a *scope* that determines when it is active (see [[*Keybinding]]).
Built-in commands default to =global=; commands registered by a panel type
default to =local=.

** Keybinding

A mapping from a key combo (e.g. =ctrl+k=, =alt+x=, =g g= for a two-key
sequence) to a command invocation string (e.g. =entry-list:set-mode grid=).
Keybindings are loaded from a configuration file and can be edited at runtime.
No keybinding is hard-coded; the defaults ship in the default config file.

Each keybinding has an optional =when= clause that restricts when it fires.
Four scope levels exist, in decreasing priority:

| Scope     | =when= value          | Active when                                               |
|-----------+-----------------------+-----------------------------------------------------------|
| =always=  | =always=              | Everywhere, including inside text inputs. Use sparingly.  |
| =editing= | =editing=             | A text input is focused (command input or field editor).  |
| =local=   | =<panel-type-name>=   | The named panel type is focused; no text input focused.   |
| =global=  | absent                | No text input focused; no panel-type restriction.         |

When multiple bindings match the same key, =always= wins over =editing=/=local=,
which win over =global=.

Config file examples:

#+BEGIN_SRC toml
"escape"  = { command = "editing:unfocus",          when = "always"     }
"ctrl+a"  = { command = "editing:goto-line-start",  when = "editing"    }
"j"       = { command = "entry-list:next",           when = "entry-list" }
"ctrl+t"  = { command = "tab:new"                                        }
#+END_SRC

The =editing= scope covers all text inputs uniformly: the command input and
every field editor in =entry-detail=, so text-editing keybindings are defined
once and work everywhere.

Panel types may suggest default keybindings via =metafolder.addKeybinding=;
these are applied only when the user has no conflicting binding configured.

** Status bar

A read-only line at the very bottom of the window, always visible and never
focused. Displays status messages, error notifications, the name of the last
executed command (dim text), and a spinner during long operations. Content is
global (shared across all workspaces). Messages auto-clear after a
configurable timeout (default 5 s).

** Command input

A single-line text input that appears just above the status bar when activated
(configurable keybinding, default suggestion: =:=). Dismissed with =Escape=.
Content (text and cursor position) is per-workspace: switching tabs restores
the exact state that was active in that workspace, including a partial command
being typed.

Accepts:
- A command name (with inline completion; =Tab= accepts the first suggestion)
  optionally followed by space-separated parameters.
- A shell command prefixed with =!= — executed as a subprocess; stdout/stderr
  goes to the =message= panel type and to the terminal from which the GUI was
  launched.

** Field editor

An inline editor for a single field value, adapting its widget to the type:

| Value type    | Widget                                         |
|---------------+------------------------------------------------|
| =String=      | Text input                                     |
| =Int=         | Integer input                                  |
| =Float=       | Float input                                    |
| =Bool=        | Checkbox / toggle                              |
| =DateTime=    | Date-time picker (ISO-8601 output)             |
| =Nothing=     | Read-only label "absent"                       |
| =Ref=         | UUID text input (lookup by path available)     |
| =TreeRef=     | Parent UUID + name, or path string             |
| =RefBase=     | UUID text input                                |
| =ExternalRef= | Repo UUID + entry UUID                         |

** Script session

An external process communicating with the GUI over the GUI API. A script
session can set panel types, display messages in the status bar, activate the
command input with a prompt, and block waiting for user key input. Only one
script session may hold the input-wait lock at a time.

** Repo selector

A persistent UI element that sets =active_repo= in the current workspace.
Populated from =GET /repos=.

* GUI

** Connection to the daemon

The GUI connects to a daemon URL configured by the user (default
=http://127.0.0.1:7523=), editable via the =daemon:set-url= command.

On startup (and whenever the URL changes) the GUI calls =GET /health=. If the
daemon is unreachable or returns a non-200 response, a prominent error banner
is shown and all daemon-dependent commands are disabled. The GUI itself
(panels, status bar, command input, keybindings, panel type system, and the
GUI API) remains functional.

No authentication: the daemon is local-only and requires none.

** Tab bar

A row of tabs at the very top of the window. Each tab represents one
workspace. Clicking a tab (or pressing its keybinding) switches workspaces
atomically: panels, their panel types, the panel divider position, and the
command input are all restored.

Tabs have a short user-editable name (default: ="Tab 1"=, ="Tab 2"=, etc.).
Tab state (names, order, workspace contents) is persisted across sessions.

| Command           | Description                                               |
|-------------------+-----------------------------------------------------------|
| =tab:new=         | Open a new workspace tab (default panel type: entry-list) |
| =tab:close=       | Close the active tab (not available when only one remains)|
| =tab:rename=      | Rename the active tab (prompts via command input)         |
| =tab:next=        | Switch to the next tab (wraps around)                     |
| =tab:prev=        | Switch to the previous tab (wraps around)                 |
| =tab:goto-N=      | Switch directly to tab N                                  |

Default keybinding suggestions: =alt+t= for =tab:new=, =alt+w= for
=tab:close=, =alt+1= … =alt+9= for =tab:goto-N=.

** Layout and panels

Each workspace's area (below the tab bar) is divided into:
- *Main area*: one or two panels side by side.
- *Command input*: a single line just above the status bar, visible only when
  active.
- *Status bar*: a single read-only line at the very bottom, always visible.

At startup one panel is shown. =panel:split= opens a second panel mirroring
the first. =panel:close= closes the focused panel (not available when only
one remains). The focused panel receives keyboard navigation; focus switches
via =panel:focus-next=. The focused panel has a visible indicator (border
highlight — controlled by the theme).

** Status bar

Always-visible read-only line at the bottom. Shows:
- Status messages after commands (e.g. ="Entry deleted."=).
- Error messages from failed operations.
- The last executed command name (dim text).
- A spinner during long-running operations (reconcile, shell commands).

Content is global (not per-workspace).

** Command input

Hidden by default; activated by a configurable keybinding (default: =:=).
Shows a prompt character and a text field. Dismissed with =Escape=. Content
is per-workspace.

Behaviours:
- Typing a command name + =Enter= → invokes the command (=Tab= completes).
- Parameters follow the command name, separated by spaces.
- =!shell-command= + =Enter= → runs in background; output goes to the
  =message= panel type and the launching terminal.
- When a script calls =POST /gui/prompt=, the command input activates with the
  script-supplied prompt text.

** Keyboard navigation

Every command is reachable from the keyboard. No command requires the mouse.

Within =entry-list=:
- =Up= / =Down= move the selection (updates =selected_entry=).
- =Enter= or =Right= opens the entry in the other panel (or opens the other
  panel if it was closed), changing its panel type to =entry-detail= (or =file=
  if the entry has a known file path). See [[*Cross-panel selection]].
- =Space= toggles selection for multi-select (updates =selected_entries=).

Within =entry-detail=:
- =Tab= / =Shift-Tab= move focus between field rows.
- =Enter= on a field starts inline editing; =Enter= again confirms, =Escape=
  cancels.
- =d= (configurable) deletes the focused field row (with confirmation).

Keybindings are stored in =~/.config/metafolder-gui/keybindings.toml=
(see [[*Keybinding]] for the =when= clause syntax and the four scope levels).
The =config:open= command opens a settings view where they can be edited
interactively; changes take effect immediately.

** Style and theming

The GUI loads =~/.config/metafolder-gui/style.css= at startup and applies it
globally to the WebView. The GUI ships a default stylesheet the user copies
and modifies. The =config:open= settings view shows the active CSS path and
offers a reload button; the GUI also auto-reloads on file change via inotify.

** Panel type system

*** Directory layout

Panel types live under =~/.config/metafolder-gui/panel-types/=:

#+BEGIN_SRC
~/.config/metafolder-gui/panel-types/
  entry-list/
    index.html        ← entry point
  entry-detail/
    index.html
  file/
    index.html
  message/
    index.html
  log/
    index.html
  repos/
    index.html
  my-custom-panel/    ← user-defined
    index.html
#+END_SRC

Built-in types ship as pre-populated examples. The user can edit them freely;
the GUI does not overwrite them on update (the update ships new defaults to a
separate location the user can diff against).

*** The =metafolder= API

The GUI injects a =window.metafolder= object into every panel type's JS
context. It provides three namespaces:

**** =metafolder.daemon=

Typed async wrappers around the daemon HTTP API. Examples:

#+BEGIN_SRC js
await metafolder.daemon.query(repo, query, select)
await metafolder.daemon.getEntry(repo, uuid)
await metafolder.daemon.setField(repo, uuid, name, value)
await metafolder.daemon.reconcile(repo)
// ... mirrors the full daemon REST API
#+END_SRC

**** =metafolder.workspace=

Reactive per-workspace key-value store.

#+BEGIN_SRC js
metafolder.workspace.set('selected_entry', { uuid, repo })
const entry = metafolder.workspace.get('selected_entry')
metafolder.workspace.onChange('selected_entry', ({ uuid, repo }) => { ... })
#+END_SRC

Standard keys: =active_repo=, =selected_entry=, =selected_entries=.
Panel types may use namespaced custom keys (e.g. =my-panel:state=).

**** =metafolder.commands= and =metafolder.addKeybinding=

#+BEGIN_SRC js
// scope defaults to 'local' for panel-type-registered commands
metafolder.commands.register('my-panel:change-mode', {
  label: 'My Panel: Change display mode',
  scope: 'local',    // 'always' | 'editing' | 'local' | 'global'
  handler: ({ mode }) => { ... }
})

metafolder.commands.register('my-panel:global-action', {
  label: 'My Panel: Global action',
  scope: 'global',
  handler: () => { ... }
})

// Suggest default keybindings (applied only if no conflict in user config).
// The optional third argument overrides the when clause; defaults to the
// command's scope.
metafolder.addKeybinding('my-panel:change-mode list', 'ctrl+l')
metafolder.addKeybinding('my-panel:change-mode grid', 'ctrl+g')
metafolder.addKeybinding('editing:my-text-action',    'ctrl+e', 'editing')
metafolder.addKeybinding('my-panel:urgent',           'ctrl+u', 'always')
#+END_SRC

Registered commands appear in the command input autocomplete alongside
built-in commands. The autocomplete shows global commands always, plus local
commands for the currently focused panel type.

**** =metafolder.statusBar=

#+BEGIN_SRC js
metafolder.statusBar.message('Sync complete.', 5000)  // timeout_ms or null
#+END_SRC

** Repository management

*** Repo list

Calls =GET /repos= at startup and on explicit refresh. Results populate the
Repo selector and the =repos= panel type.

*** Init repo

A form with =root= (required) and =metafolder= (optional, for external
database). Submitting calls =POST /repos/init=. On success the repo is added
to the Repo selector and selected as active. On =409= an error message appears
in the status bar.

*** Load repo

A form with mutually exclusive =root= or =metafolder= inputs. Submitting calls
=POST /repos/load=. On success the repo is added and selected.

** Entry management

*** =entry-list= panel type

Displays entries for the active repo (=active_repo= workspace variable)
filtered by an embedded query. The query editor is shown/hidden by the
=entry-list:edit-query= command; it uses the same DSL as the CLI. An empty
query matches all entries.

Default display mode: =table= with columns =mfr_path= (resolved path),
=mfr_type=, =version=. Columns are configurable. The =entry-list:set-mode=
command switches between display modes.

Supports sorting by column, multi-selection (=selected_entries=), and full
keyboard navigation.

When the user moves the selection, =entry-list= writes the highlighted entry
to =workspace.selected_entry=. Other panels update reactively (see
[[*Cross-panel selection]]).

*** =entry-detail= panel type

Calls =GET .../metadata/:uuid= for the entry in =selected_entry=. Displays all
fields. Each field row shows name, value, an edit button, and a delete button.

Keyboard: =Tab= moves between rows, =Enter= edits, =d= deletes (with
confirmation).

At the bottom:
- /Add field/ → =POST .../metadata/:uuid/fields=
- /Set field/ → =PATCH .../metadata/:uuid=
- /Delete entry/ → confirmation → =DELETE .../metadata/:uuid=

Protected =mfr_*= fields are read-only by default; an advanced toggle exposes
a =force= checkbox.

*** =file= panel type

Displays a preview of the file at the path stored in =selected_entry='s
=mfr_path= field, using the HTML5 =<img>=, =<audio>=, or =<video>= element
depending on the file type. Falls back to a plain-text view for text files.
Shows a "no preview available" message for unsupported formats.

*** Cross-panel selection

Selecting an entry in =entry-list= (keyboard or mouse) sets =selected_entry=
in the workspace. Any other panel displaying =entry-detail= or =file=
subscribes to this variable and updates its content automatically.

*Double-click* (or =Enter= / =Right= keybinding) in =entry-list=: if the other
panel is closed, it is opened; its panel type is set to =entry-detail= (or
=file= if the entry has a valid =mfr_path=).

*Edit guard*: before =entry-detail= replaces its content with a new entry, it
checks whether any field is being edited. If so, an alert is shown:
="Unsaved changes — discard and switch entry? [Confirm / Cancel]"=. The switch
proceeds only if the user confirms.

*** Create entry

=entry:create= opens a dialog with an empty field list. The user adds fields
via the Field editor. Submitting calls =POST .../metadata=. On success the new
entry is inserted into the active =entry-list= and selected.

*** Batch set

When =selected_entries= is non-empty, =entry:batch-set= opens a dialog for
field name and value and calls =POST .../set= with an =Or= of the selected
UUIDs.

** Message view

The =message= panel type is a scrollable, append-only log. It receives:
- All messages posted to the status bar (with timestamp).
- stdout/stderr of shell commands run via =!=.
- Reconcile progress and results.
- Any message posted by a script via =POST /gui/message= (also shown in the
  status bar as a transient notification).

The log is cleared by the =message:clear= command. It is not persisted across
sessions.

** Reconcile

=reconcile:run= calls =POST .../reconcile=. A spinner appears in the status
bar. On completion:
- Status bar: ="Reconcile: N created, M updated, P unchanged."=
- Full output (per-file actions and candidate stale matches) is appended to the
  =message= panel type.

** Event log

*** Log view (=log= panel type)

Calls =GET .../log= and displays revisions in reverse chronological order.
Columns: =rev_id=, timestamp, operation count, =status=.

Selecting a revision (=Enter=) expands it to show individual operations.

*** Revision commands

- =log:rollback= → confirmation → =POST .../rollback=. Refreshes =entry-list=
  afterwards.
- =log:mark-checkpoint= [v2] → =PATCH .../log/revisions/:rev_id=.

*** Prune log

=log:prune= opens a dialog (target revision or "keep last N revisions") and
calls =POST .../log/prune=.

** Scripting / GUI API

The GUI exposes a local HTTP API on =127.0.0.1:7524= (configurable via
=--gui-port=). External scripts can drive the GUI: set panel types, display
messages, wait for user input.

All endpoints are available as long as the GUI is running, regardless of
daemon connectivity.

*** =PUT /gui/panels/:id/view=

Sets the panel type of panel =:id= (=1= or =2=) in the active workspace.
Optional =?tab=N= query parameter targets a specific workspace by 1-based
index. If panel =2= does not exist, it is opened automatically.

Request body examples:

#+BEGIN_SRC json
{"type": "entry-list"}
{"type": "entry-detail"}
{"type": "file", "path": "/absolute/path/to/image.jpg"}
{"type": "message"}
{"type": "log"}
{"type": "my-custom-panel", "state": { ... }}
#+END_SRC

Response: =200 OK=, body ={}\=.

*** =GET /gui/panels/:id/view=

Returns the current panel type descriptor of panel =:id=, including a =status=
field (="loading"= / ="ready"=).

*** =POST /gui/message=

Posts a message to the status bar (transient) and appends it to the =message=
panel type log.

Request body:
#+BEGIN_SRC json
{"text": "← left  → right  Esc to stop", "timeout_ms": null}
#+END_SRC

=timeout_ms=: null → status bar message persists until the next message;
integer → auto-clears after that many milliseconds.

*** =POST /gui/input=

Waits for the user to press one of the listed keys. *Blocks* until a match or
timeout. The GUI enters a modal input-wait state; non-listed keys are
suppressed and noted in the status bar.

Only one =POST /gui/input= at a time; concurrent calls return =409=.

Request body:
#+BEGIN_SRC json
{"keys": ["left", "right", "escape"], "timeout_ms": null}
#+END_SRC

Responses: ={"event": "key", "key": "left"}= / ={"event": "timeout"}= /
={"event": "closed"}=.

*** =POST /gui/prompt=

Activates the command input with a custom prompt. Blocks until =Enter= or
=Escape=. Mutually exclusive with =POST /gui/input= (same lock).

Request body: ={"prompt": "Enter a rating (1-5): ", "timeout_ms": null}=

Responses: ={"event": "confirm", "text": "4"}= / ={"event": "cancel"}=.

*** =GET /gui/status=

Returns the current GUI state.

#+BEGIN_SRC json
{
  "tabs": [
    {
      "index": 1, "name": "Tab 1", "active": true,
      "panels": [
        {"id": 1, "type": "entry-list", "focused": true},
        {"id": 2, "type": "file"}
      ]
    }
  ],
  "active_repo": "<repo_uuid>",
  "daemon_connected": true,
  "input_lock": false
}
#+END_SRC

*** Example: image comparison script

#+BEGIN_SRC bash
#!/usr/bin/env bash
GUI="http://127.0.0.1:7524"

while true; do
  A=$(find /photos -name "*.jpg" | shuf -n1)
  B=$(find /photos -name "*.jpg" | shuf -n1)

  curl -s -X PUT $GUI/gui/panels/1/view \
       -H "Content-Type: application/json" \
       -d "{\"type\":\"file\",\"path\":\"$A\"}"
  curl -s -X PUT $GUI/gui/panels/2/view \
       -H "Content-Type: application/json" \
       -d "{\"type\":\"file\",\"path\":\"$B\"}"
  curl -s -X POST $GUI/gui/message \
       -H "Content-Type: application/json" \
       -d '{"text":"← left  → right  Esc to stop"}'

  RESP=$(curl -s -X POST $GUI/gui/input \
              -H "Content-Type: application/json" \
              -d '{"keys":["left","right","escape"]}')
  KEY=$(echo "$RESP" | jq -r .key)

  [ "$KEY" = "escape" ] && break

  WINNER=$( [ "$KEY" = "left" ] && echo "$A" || echo "$B" )
  # record winner as metadata via daemon …
done
#+END_SRC

* Open questions

** Frontend framework

Tauri is decided. The JavaScript/TypeScript frontend framework (Svelte, React,
Vue, vanilla JS, etc.) is not yet chosen. A lightweight option that compiles
to vanilla JS with no runtime is preferred.

** =metafolder= API injection mechanism

Panel types need access to =window.metafolder=. Two options:
- Each panel type is an =<iframe>=; the GUI communicates via =postMessage=.
- Panel types are loaded as web components or modules in the same document,
  sharing the JS runtime directly.
The choice affects isolation (iframes are sandboxed) vs. simplicity.

** Media codec support

The =file= panel type uses HTML5 =<audio>= / =<video>=; supported formats
depend on the system WebView (WebKit2GTK on Linux). H.264 may require
=gst-plugins-bad= or =gst-libav= on some Arch configurations.

** Panel type update strategy

Built-in panel types ship as editable examples. When the GUI updates, new
default versions must not silently overwrite user edits. A diff/merge workflow
is needed (exact mechanism TBD).

** Pagination

=GET /metadata= and =POST /query= return all results in one response.
=entry-list= must use virtual scrolling or wait for daemon-side cursor
pagination.

** Path display

=TreeRef= values are =(parent_uuid, name)= pairs. Displaying a full path
requires walking the parent chain. Resolution strategy (eager vs. lazy) is TBD.

** GUI API port persistence

Whether the GUI API port (default 7524) is written to a well-known file so
scripts can discover it without hard-coding is not yet decided.

** Concurrent script sessions

Only one =POST /gui/input= at a time. Whether additional sessions queue or
fail immediately is not specified for v1.
