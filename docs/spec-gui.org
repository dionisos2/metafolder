#+TITLE: Metafolder — GUI Specification
#+OPTIONS: toc:2 num:t

* Overview

The GUI is a graphical interface to the Metafolder daemon. It exposes the full
CLI surface in visual form: repository management (init/load/list), entry
CRUD, query execution, reconcile, and event log inspection. It communicates
with an already-running daemon over HTTP and does not manage the daemon's
lifecycle.

Four cross-cutting design principles govern the GUI:
- *Keyboard-first*: every action is reachable from the keyboard; all shortcuts
  are configurable.
- *Command-driven*: a persistent command zone (bottom bar) accepts named
  commands and shell invocations, as in Emacs.
- *Scriptable*: the GUI exposes its own local HTTP API so that external scripts
  can drive panels and wait for user input.
- *Themeable*: colours, fonts, and layout parameters are configurable via a
  style file; if a web renderer is used, a CSS file is the natural mechanism.

The GUI framework is not yet decided; this spec focuses on behaviour and UX.

* Concepts

** Workspace

The complete state of one tab: panel count (1 or 2), each panel's current
view and its internal state (scroll position, selection, query draft, etc.),
the panel width ratio, and the command zone content and mode (including any
partial command being typed). Switching tabs swaps workspaces atomically;
no state is lost. Each workspace is independent: the same view can be open
simultaneously in two workspaces without interference.

** Panel

A resizable content area belonging to a workspace. Each workspace displays
one or two panels side by side (a single panel at workspace creation; the
user can open a second panel). Each panel independently displays one /view/.
Panels are separated by a draggable divider; their relative width ratio is
part of the workspace state.

** View

The content type displayed inside a panel. Built-in views:

| View name      | Description                                    |
|----------------+------------------------------------------------|
| =entry-list=   | Tabular list of metadata entries               |
| =entry-detail= | All fields of a single entry                   |
| =query=        | Query builder + results                        |
| =reconcile=    | Reconcile trigger and result                   |
| =log=          | Event log (revisions and operations)           |
| =repos=        | Repository list, init, load                    |
| =file=         | Preview a file from disk (image, text, …)      |
| =html=         | Render an arbitrary HTML string (scripting use)|

Scripts can display =file= and =html= views via the GUI API (see [[*Scripting / GUI API]]).

** Command zone

A single-line text input anchored to the bottom of the window, always visible.
Its content and mode are part of the workspace state: switching tabs restores
the exact text and cursor position that were active in that workspace (including
a partial command being typed). It has two modes:

- *Status mode* (default): displays status messages, notifications, and the
  last executed command. Not focused; does not capture keystrokes.
- *Input mode*: focused by a configurable keybinding (default =:= or =M-x=
  equivalent). Accepts command names with completion, shell commands prefixed
  with =!=, and free-text prompts sent by scripts.

** Action

A named, invocable operation (e.g. =repos:init=, =panel:split=,
=query:execute=, =entry:delete=). Every user-visible behaviour is an action.
Actions can be invoked by keybinding or by typing the action name in the
command zone.

** Keybinding

A mapping from a key combo (or sequence) to an action. The full keybinding map
is loaded from a configuration file and can be edited at runtime. No built-in
shortcuts are hard-coded; the defaults ship in the default config file.

** Repo selector

A persistent UI element that sets the /active repository/. All entry, query,
reconcile, and log operations target the active repository. Populated from
=GET /repos=.

** Entry list

A tabular view of metadata entries. Each row is one entry; columns are field
names chosen by the user or returned by a query. Rows are selectable; opening
a row navigates to the entry detail view (in the same panel or the other).

** Entry detail panel

Displays all fields of one entry. Supports inline field editing, adding new
fields, and deleting individual field rows.

** Field editor

An inline editor for a single field value, adapting its widget to the type:

| Value type    | Widget                                         |
|---------------+------------------------------------------------|
| =String=      | Text input                                     |
| =Int=         | Integer input                                  |
| =Float=       | Float input                                    |
| =Bool=        | Checkbox / toggle                              |
| =DateTime=    | Date-time picker (ISO-8601 output)             |
| =Nothing=     | Read-only label "absent"                       |
| =Ref=         | UUID text input (lookup by path available)     |
| =TreeRef=     | Parent UUID + name, or path string             |
| =RefBase=     | UUID text input                                |
| =ExternalRef= | Repo UUID + entry UUID                         |

** Query builder

A visual tool for constructing a =Query= value. Supports the full =Query=
enum (boolean combinators, predicates, comparisons, =Matches=, =Follows=,
=FollowsTransitive=). The user may also type a raw DSL expression (same syntax
as the CLI) in a text field; it is parsed client-side and reflected in the
builder tree.

** Script session

An external process communicating with the GUI over the GUI API. A script
session can set panel views, display messages in the command zone, and block
waiting for user key input. Only one script session may hold the input-wait
lock at a time (subsequent sessions' =POST /gui/input= calls receive =409= until
the lock is released).

* GUI

** Connection to the daemon

The GUI connects to a daemon URL configured by the user (default
=http://127.0.0.1:7523=). The URL is editable in a settings panel or via the
=daemon:set-url= action.

On startup (and whenever the URL changes) the GUI calls =GET /health=. If the
daemon is unreachable or returns a non-200 response, a prominent error banner
is shown and all daemon-dependent operations are disabled until the connection
is re-established. The GUI itself (panels, command zone, keybindings, and the
GUI API) remains functional so that scripts can still interact with it.

No authentication: the daemon is local-only and requires none.

** Tab bar

A row of tabs is displayed at the very top of the window, above all panels.
Each tab represents one workspace. Clicking a tab (or pressing its keybinding)
switches to that workspace; the window content — panels, their views, the panel
divider position, and the command zone — is replaced atomically with the stored
workspace state.

Tabs have a short user-editable name (default: ="Tab 1"=, ="Tab 2"=, etc.).
The active tab is visually highlighted (exact appearance controlled by the
theme).

Tab-related actions:

| Action            | Description                                               |
|-------------------+-----------------------------------------------------------|
| =tab:new=         | Open a new workspace tab (default view: entry-list)       |
| =tab:close=       | Close the active tab (not available when only one remains)|
| =tab:rename=      | Rename the active tab (prompts via command zone)          |
| =tab:next=        | Switch focus to the next tab (wraps around)               |
| =tab:prev=        | Switch focus to the previous tab (wraps around)           |
| =tab:goto-N=      | Switch directly to tab number N (e.g. =tab:goto-1=)      |

Default keybinding suggestions: =Alt+T= for =tab:new=, =Alt+W= for
=tab:close=, =Alt+1= … =Alt+9= for =tab:goto-N=. All are configurable.

The GUI starts with one workspace tab. Tab state (names, order, and workspace
contents) is persisted across sessions in the config file.

** Layout and panels

Each workspace's window area (below the tab bar) is divided vertically into:
- *Main area*: one or two panels side by side.
- *Command zone*: a single line at the very bottom.

At startup only one panel is shown. The =panel:split= action opens a second
panel, mirroring the first panel's current view. The =panel:close= action
closes the focused panel (not available when only one panel is open). Panel
widths are separated by a draggable divider; the ratio is persisted across
sessions in the config file.

The /focused panel/ is the one that receives keyboard navigation. Focus
switches between panels via a configurable keybinding (default =Tab= or a
two-key chord). The focused panel has a visible focus indicator (e.g. a
highlighted border — exact appearance controlled by the theme).

** Command zone

*** Status mode

The command zone displays:
- Brief status messages after actions (e.g. "Entry deleted.", "Reconcile: 3 created.").
- Error messages from failed operations (e.g. "Error 409: repository already exists.").
- The name of the last executed action (dim text).

Messages auto-clear after a configurable timeout (default 5 s).

*** Input mode

Activated by a configurable keybinding (default suggestion: =:=). In input mode:

- Typing a word and pressing =Enter= invokes the matching action (completion
  is shown inline; =Tab= accepts the first suggestion).
- Typing =!= followed by a shell command and pressing =Enter= executes the
  shell command. Its stdout/stderr appears in the command zone and the
  terminal from which the GUI was launched. The GUI does not block during
  execution; a spinner appears in the command zone.
- When a script calls =POST /gui/prompt=, the GUI enters input mode with the
  script-supplied prompt text; the user's response is returned to the script.
- =Escape= cancels and returns to status mode.

*** Action names (non-exhaustive default set)

| Action              | Description                                    |
|---------------------+------------------------------------------------|
| =daemon:set-url=    | Change daemon URL                              |
| =repos:init=        | Open init-repo form                            |
| =repos:load=        | Open load-repo form                            |
| =tab:new=           | Open a new workspace tab                       |
| =tab:close=         | Close the active tab                           |
| =tab:rename=        | Rename the active tab                          |
| =tab:next=          | Switch to the next tab                         |
| =tab:prev=          | Switch to the previous tab                     |
| =tab:goto-N=        | Switch to tab N (replace N with a number)      |
| =panel:split=       | Open second panel in the active workspace      |
| =panel:close=       | Close focused panel                            |
| =panel:focus-next=  | Move focus to the other panel                  |
| =view:entry-list=   | Switch focused panel to entry list             |
| =view:query=        | Switch focused panel to query builder          |
| =view:log=          | Switch focused panel to event log              |
| =view:repos=        | Switch focused panel to repos view             |
| =entry:create=      | Open create-entry dialog                       |
| =entry:delete=      | Delete currently selected entry                |
| =query:execute=     | Execute the current query                      |
| =reconcile:run=     | Trigger reconcile on active repo               |
| =config:open=       | Open settings (keybindings, style, daemon URL) |
| =quit=              | Exit the GUI                                   |

** Keyboard navigation and shortcuts

*** Principle

Every UI action is reachable from the keyboard. No action requires the mouse.
Mouse interaction is supported everywhere but is never the only path.

Within the entry list:
- =Up= / =Down= (or =k= / =j= in a Vim-style config) move the selection.
- =Enter= or =Right= opens the entry detail view.
- =Left= or =Backspace= navigates back.
- =Space= toggles selection for multi-select operations.

Within the entry detail:
- =Tab= / =Shift-Tab= move focus between field rows.
- =Enter= on a field starts inline editing; =Enter= again confirms, =Escape=
  cancels.
- =d= (or a configured key) deletes the focused field row (with confirmation).

*** Configuration

Keybindings are stored in a TOML/JSON configuration file (path configurable,
default =~/.config/metafolder-gui/keybindings.toml=). Each entry maps a key
combo (e.g. =ctrl+k=, =alt+x=, =g g= for sequences) to an action name.

The =config:open= action opens a settings view where keybindings can be edited
interactively; changes take effect immediately without restarting.

** Style and theming

The GUI loads a style configuration at startup (default
=~/.config/metafolder-gui/style.css= or a structured theme file, depending on
the chosen framework):

- If a web renderer is used (e.g. Tauri/WebView): the style file is CSS,
  applied globally to the rendered HTML. The user can override any property.
- If a native renderer is used (e.g. egui, Iced): the style file is a
  structured format (TOML/JSON) with named variables: background colour,
  foreground colour, accent colour, font family, font size, border radius,
  panel divider width, etc.

In both cases the GUI ships a default style file that the user copies and
modifies. The =config:open= settings view shows the path to the active style
file and offers a /reload/ button (or the GUI auto-reloads on file change via
inotify).

** Repository management

*** Repo list

The GUI calls =GET /repos= at startup and on explicit refresh. Results
populate the Repo selector and the =repos= view, listing each repository with
its =repo_uuid=, =name=, and =root= path.

*** Init repo

A form with two fields:
- =root= (required) — absolute path to the directory to watch.
- =metafolder= (optional) — override for the =.metafolder/= location.

Submitting calls =POST /repos/init=. On success the new repo is added to the
Repo selector and selected as active. On =409= an error message is shown in
the command zone.

*** Load repo

A form with two mutually exclusive inputs:
- =root= — standard form.
- =metafolder= — external database form.

Submitting calls =POST /repos/load=. On success the repo is added and selected.

** Entry management

*** Entry list view

Lists entries for the active repo. Default columns: =mfr_path= (resolved path
string), =mfr_type=, =version=. The user can add/remove columns (arbitrary
field names, fetched via a query with =select=).

Supports: sorting by column, a free-text filter (=Matches= query on
=mfr_path= server-side), multi-selection, and full keyboard navigation.

*** Create entry

The =entry:create= action opens a dialog pre-populated with an empty field
list. The user adds fields using the Field editor. Submitting calls
=POST .../metadata=. On success the new entry is inserted and selected.

*** Entry detail view

Calls =GET .../metadata/:uuid=. Each field row shows name, value (rendered
by type), an edit button, and a delete button. Keyboard: =Tab= moves between
rows, =Enter= edits, =d= deletes (with confirmation).

At the bottom:
- /Add field/ → =POST .../metadata/:uuid/fields=
- /Set field/ (replaces all rows for a name) → =PATCH .../metadata/:uuid=
- /Delete entry/ → confirmation → =DELETE .../metadata/:uuid=

Protected =mfr_*= fields are read-only by default. An advanced toggle exposes
a =force= checkbox that sends ={"force": true}= in the request body.

*** Batch set

Available when one or more entries are selected. A dialog prompts for field
name and value; submitting calls =POST .../set= with the active query (or an
=Or= of the selected entry UUIDs).

** Query

*** Query builder view

A panel that lets the user construct a query visually (tree of nodes: leaf
predicates, combinators, graph traversals) or by typing a DSL expression.
Pressing the =query:execute= action calls =POST .../query= with =select: "*"=;
results replace the entry list in the focused (or other) panel.

*** Batch set from query results

A /Set field on results/ button (or action =query:set-field=) opens the
batch-set dialog and calls =POST .../set= with the current query object.

** Reconcile

The =reconcile:run= action calls =POST .../reconcile=. A spinner appears in
the command zone while the request is pending. On completion a result summary
is shown:
- Command zone message: ="Reconcile: N created, M updated, P unchanged."=
- A detailed result view opens in the focused panel listing candidate matches
  for stale =mfr_path= entries, each with an /Accept/ or /Ignore/ action [v2].

** Event log

*** Log view

Calls =GET .../log= and displays revisions in reverse chronological order.
Columns: =rev_id=, timestamp, operation count, =status=.

Selecting a revision (keyboard: =Enter=) expands it to show individual
operations.

*** Revision actions

- /Rollback to here/ (action =log:rollback=) → confirmation → =POST .../rollback=.
  After success the entry list is refreshed.
- /Mark as checkpoint/ [v2] → =PATCH .../log/revisions/:rev_id=.

*** Prune log

The =log:prune= action opens a dialog (target revision or "keep last N
revisions count") and calls =POST .../log/prune=.

** Scripting / GUI API

The GUI exposes a local HTTP API on =127.0.0.1:7524= (configurable via
=--gui-port=). This API allows external scripts to drive the GUI: set panel
content, display messages, and wait for user input. Scripts use any HTTP
client; the API returns JSON.

All endpoints are available as long as the GUI is running, regardless of
daemon connectivity.

*** =PUT /gui/panels/:id/view=

Sets the view displayed in panel =:id= (=1= or =2=) in the active workspace.
An optional =tab= query parameter targets a specific workspace by its 1-based
index (e.g. =?tab=2=); defaults to the active tab. If panel =2= does not
exist yet in the target workspace, it is opened automatically.

Request body — examples by view type:

#+BEGIN_SRC json
{"type": "entry-list"}
{"type": "entry-detail", "uuid": "<metadata_uuid>"}
{"type": "file",  "path": "/absolute/path/to/image.jpg"}
{"type": "html",  "html": "<h1>Hello</h1><p>World</p>"}
{"type": "query", "query": <Query>}
#+END_SRC

Response: =200 OK=, body ={}=.

If the requested panel does not exist and cannot be opened (e.g. panels
are at maximum), returns =400=.

*** =GET /gui/panels/:id/view=

Returns the current view descriptor of panel =:id=.

Response body mirrors the =PUT= request body, plus a =status= field
(e.g. ="loading"= / ="ready"=).

*** =POST /gui/message=

Displays a message in the command zone (status mode).

Request body:
#+BEGIN_SRC json
{"text": "← left image better, → right image better, Esc to stop",
 "timeout_ms": null}
#+END_SRC

=timeout_ms=: null → message persists until the next message or explicit
clear; integer → auto-clears after that many milliseconds.

Response: =200 OK=, body ={}=.

*** =POST /gui/input=

Waits for the user to press one of the listed keys and returns it. This call
*blocks* until a matching key is pressed or the timeout expires. The GUI
enters a modal input-wait state: only the listed keys (plus =Escape= which
always cancels) are forwarded; all other keystrokes are suppressed and shown
as ignored in the command zone.

Only one =POST /gui/input= can be in flight at a time. A second concurrent
call returns =409 Conflict= with body ={"error": "input lock held"}=.

Request body:
#+BEGIN_SRC json
{
  "keys": ["left", "right", "escape"],
  "timeout_ms": null
}
#+END_SRC

Key names follow the same format as the keybinding config (e.g. ="left"=,
="right"=, ="up"=, ="down"=, ="enter"=, ="escape"=, ="a"=, ="ctrl+z"=).

=timeout_ms=: null → wait indefinitely; integer → return after that many ms.

Response (key pressed):
#+BEGIN_SRC json
{"event": "key", "key": "left"}
#+END_SRC

Response (timeout):
#+BEGIN_SRC json
{"event": "timeout"}
#+END_SRC

Response (GUI closing during wait):
#+BEGIN_SRC json
{"event": "closed"}
#+END_SRC

*** =POST /gui/prompt=

Activates the command zone in input mode with a custom prompt and waits for
the user to type text and press =Enter= (or =Escape= to cancel). Blocks like
=POST /gui/input=; mutually exclusive with it (same input lock).

Request body:
#+BEGIN_SRC json
{"prompt": "Enter a rating (1-5): ", "timeout_ms": null}
#+END_SRC

Response (confirmed):
#+BEGIN_SRC json
{"event": "confirm", "text": "4"}
#+END_SRC

Response (cancelled via =Escape=):
#+BEGIN_SRC json
{"event": "cancel"}
#+END_SRC

*** =GET /gui/status=

Returns the current GUI state: tab list, active tab, panels, active repo,
daemon connectivity, input lock status.

Response body:
#+BEGIN_SRC json
{
  "tabs": [
    {
      "index": 1,
      "name": "Tab 1",
      "active": true,
      "panels": [
        {"id": 1, "view": {"type": "entry-list"}, "focused": true},
        {"id": 2, "view": {"type": "file", "path": "/photos/a.jpg"}}
      ]
    },
    {
      "index": 2,
      "name": "Query workspace",
      "active": false,
      "panels": [
        {"id": 1, "view": {"type": "query"}, "focused": true}
      ]
    }
  ],
  "active_repo": "<repo_uuid>",
  "daemon_connected": true,
  "input_lock": false
}
#+END_SRC

*** Example: image comparison script

#+BEGIN_SRC bash
#!/usr/bin/env bash
# Randomly pick two images, ask the user to choose, record the preferred one.

GUI="http://127.0.0.1:7524"
DAEMON="http://127.0.0.1:7523"
REPO="<repo_uuid>"

while true; do
  A=$(find /photos -name "*.jpg" | shuf -n1)
  B=$(find /photos -name "*.jpg" | shuf -n1)

  curl -s -X PUT $GUI/gui/panels/1/view \
       -H "Content-Type: application/json" \
       -d "{\"type\":\"file\",\"path\":\"$A\"}"
  curl -s -X PUT $GUI/gui/panels/2/view \
       -H "Content-Type: application/json" \
       -d "{\"type\":\"file\",\"path\":\"$B\"}"
  curl -s -X POST $GUI/gui/message \
       -H "Content-Type: application/json" \
       -d '{"text":"← left  → right  Esc to stop"}'

  RESP=$(curl -s -X POST $GUI/gui/input \
              -H "Content-Type: application/json" \
              -d '{"keys":["left","right","escape"]}')
  KEY=$(echo "$RESP" | jq -r .key)

  [ "$KEY" = "escape" ] && break

  WINNER=$( [ "$KEY" = "left" ] && echo "$A" || echo "$B" )
  # record winner as metadata via daemon …
done
#+END_SRC

* Open questions

** Framework choice

No GUI framework has been selected. The scripting use case (=file= and =html=
views, custom styles) favours a web renderer (Tauri/WebView) because file
preview and HTML rendering are native to the browser engine, and CSS theming
requires no extra work. A native renderer (egui, Iced) would need explicit
implementations for image decoding, HTML rendering, and a theme DSL.

** Layout (sidebar vs. tabs)

The two-panel model is specified. Whether the panels are complemented by a
persistent sidebar (repo/folder tree) or a tab bar for view switching is not
decided. A sidebar would occupy space from one panel; tabs would overlay the
panel header.

** File preview formats

The =file= view should support at minimum JPEG/PNG images and plain text.
Additional formats (PDF, SVG, video thumbnails, audio waveforms) are deferred
to v2.

** Pagination

=GET /metadata= and =POST /query= return all results in one response. For
large repos the entry list must either use virtual scrolling or request the
daemon's (not yet designed) cursor-based pagination.

** Path display

=TreeRef= values are =(parent_uuid, name)= pairs. Displaying a full path
requires walking the parent chain. The GUI must decide whether to resolve
paths eagerly (one extra request per entry) or lazily (on hover/expand).

** GUI API port persistence

Whether the GUI API port (default 7524) is persisted to a well-known file
(e.g. =~/.run/metafolder-gui.port=) so that scripts can discover it without
hard-coding the port is not yet decided.

** Concurrent script sessions

The input lock allows only one =POST /gui/input= at a time. Whether multiple
scripts can coexist (one holds the lock, others queue or fail immediately) is
not specified for v1.
