#+TITLE: Metafolder — GUI Specification
#+OPTIONS: toc:2 num:t

* Overview

The GUI is a graphical interface to the Metafolder daemon. It exposes the full
CLI surface in visual form: repository management (init/load/list), metarecord
CRUD, query execution, reconcile, and event log inspection. It communicates
with an already-running daemon over HTTP and does not manage the daemon's
lifecycle.

Four cross-cutting design principles govern the GUI:
- *Keyboard-first*: every command is reachable from the keyboard; all
  keybindings are configurable.
- *Command-driven*: a persistent status bar shows notifications; a command
  input accepts named commands and shell invocations.
- *Scriptable*: the GUI exposes its own local HTTP API so that external scripts
  can drive panels and wait for user input.
- *Themeable*: appearance is controlled by a CSS file loaded at startup.

The GUI is built with *Tauri* (v2): a Rust backend communicating with a
system WebView frontend. The shell frontend is *Svelte 5* (Vite build);
panel types are plain HTML/JS with no build step. The Rust backend owns all
canonical state (workspaces, layout, variables, keybindings, command
registry); the shell is a thin reflection updated through Tauri events, so
the scripting HTTP API and the window always agree.

* Concepts

** Workspace

A self-contained state container. Each workspace holds: the state of every
panel type that has been loaded in it during the session (scroll position,
query draft, display mode, etc.), the workspace variables, and the status bar
history. Panel types retain their state for the lifetime of the session —
switching to a different panel type and back restores the previous state.
Workspace state is not persisted across sessions; the GUI starts fresh each
time.

The =active_repo= of a workspace is set when the workspace is created and
cannot be changed. To work with a different repository, a new workspace is
created (see [[*Tab bar]]). This prevents =selected_metarecord= and other workspace
variables from becoming invalid due to a repo switch.

** Panel slot

One of two fixed content areas in the window (left and right). Each slot is
assigned at most one workspace. An unassigned slot remains visible but shows a
centred "No workspace selected" message; the user can assign a workspace by
clicking a tab or using =tab:new=. A workspace can be assigned to zero, one,
or both slots simultaneously.

When both slots display the same workspace, each slot independently selects
which panel type to show. This enables, for example, =metarecord-list= on the left
and =file= on the right within one workspace, sharing its variables
(=selected_metarecord=, etc.).

The two slots are separated by a draggable divider; the width ratio is part of
the window state (not per-workspace). The focused slot has a visible indicator
(border highlight — theme-controlled) and receives keyboard input.

** Panel type

The content/functionality kind displayed inside a panel. Each panel type is
implemented as a directory containing an =index.html= entry point; built-in
types ship as default examples in the user's config directory so they can be
inspected and modified. The panel type name is the directory name.

Built-in panel types:

| Name             | Description                                                  |
|------------------+--------------------------------------------------------------|
| =metarecord-list=     | Metadata metarecords filtered by an embedded query; primary selection source |
| =metarecord-detail=   | All fields of the selected metarecord; follows =selected_metarecord=   |
| =file=           | Preview of the file at the active path from =selected_paths=        |
| =file-manager=   | Filesystem tree navigator; browse and add files/dirs to the DB |
| =workspace-info= | Current workspace variables — useful for debugging and scripting |
| =message=        | Persistent log of status bar messages, shell output, and reconcile results |
| =log=            | Event log (revisions and operations)                         |
| =repos=          | Repository list, init, load, unload, per-repo running tasks   |
| =treeref=        | TreeRef forest explorer; pick a TreeRef field, navigate roots→leaves |
| =ref-list=       | Metarecords whose chosen Ref field points to the =treeref= selection |
| =help=           | In-app help: static pages with a grep search box (see [[*Help]]) |

Users can add custom panel types by creating a new directory under
=~/.config/metafolder-gui/panel-types/<name>/=. Custom types appear alongside
built-in types in the panel type selector.

Each panel type retains its full state for the lifetime of the session, even
when not displayed. Switching to another panel type and back restores the
previous state unchanged.

Panel *construction* is separate from the *first display*: loading the
iframe registers commands, keybinding suggestions and listeners, but a
panel's expensive work — the initial query of =metarecord-list=, the directory
listing of =file-manager=, the log fetch, streaming the previewed file —
is deferred to the first actual display via =metafolder.whenVisible(fn)=
(=metafolder.visible= exposes the current state; while hidden, repeated
triggers re-arming the same callback coalesce into one run on show). The
shell pre-instantiates every installed panel type once at startup, hidden,
for the startup workspace: all panel commands are registered from the
start at the cost of construction only, without daemon traffic.

** Display mode

The visual representation used within a panel type that supports several
layouts. Stored as part of the panel type's state in the workspace.

The =metarecord-list= panel type supports at minimum:
- =table= — tabular list with configurable columns.
- =grid= — thumbnail grid, useful for image collections.

For large result sets, =metarecord-list= uses the daemon's cursor pagination
(=limit= / =cursor= / =sort= on =POST /query=, see the Pagination section of
[[file:spec-data-model.org][spec-data-model.org]]), fetching further pages as
the user scrolls; a footer shows =loaded/total= rows (the total from the
daemon's =count=). The default page size comes from the GUI config
(=[page-size].metarecord-list= in =config.toml=, default 100) — the smallest of
the list panels, as each row needs several daemon round-trips (TreeRef path
resolution, ref-target metarecords) and parsing. It is further overridable per
workspace: =metarecord-list:set-page-size <n>= persists it in the
=metarecord-list:page-size= workspace variable (also settable by scripts);
invalid stored values fall back to the config default.

Orphaned metarecords — a metarecord with =mfr_path= whose file no longer
exists on disk, either =mfr_path = Nothing= (the watcher saw the deletion)
or a stale TreeRef (left in place by reconcile, see
[[file:spec-file-tracking.org][spec-file-tracking.org]]) — are made visible:
=metarecord-list= colours the row/card (theme variable =--mf-orphan=, purple
by default) with a tooltip giving the reason; =metarecord-detail= shows a
one-line note under the metarecord head. A metarecord with no =mfr_path=
field at all is not a file metarecord and is never marked.

** Workspace variables

A reactive, per-workspace key-value store shared across all panel types
loaded in the workspace. Panel types read and write these variables to communicate without
being directly coupled to each other.

Standard variables:

| Key                | Type                          | Description                        |
|--------------------+-------------------------------+------------------------------------|
| =active_repo=      | =string \vert null=           | UUID of the active repository      |
| =selected_paths=   | =string[]=                    | Absolute paths of the highlighted filesystem metarecord (usually one; multiple if the metarecord has several =mfr_path= values) |
| =selected_metarecord=   | =\{uuid, repo\} \vert null=   | Metadata metarecord for the highlighted path (null if untracked) |
| =selected_metarecords= | =string[]=                    | UUIDs of all selected metarecords      |
| =selected_treeref= | =\{repo, field, uuid, path\} \vert null= | TreeRef node selected in the =treeref= panel (consumed by =ref-list=) |
| =pick_request=     | =\{caller_ws, token\} \vert null= | Set on a *picker* workspace at creation; marks it as a value picker linked back to =caller_ws= (see [[*Value picker]]) |
| =pick_result=      | =\{token, uuid\} \vert \{token, cancelled\} \vert null= | Delivered to the *calling* workspace when a picker confirms (=uuid=) or cancels; the requesting form matches =token= and consumes it |

Panel types may define their own variables using namespaced keys (e.g.
=my-panel:current-node=).

** Value picker

Filling a =ref= or =tree_ref= field with a metarecord UUID by hand is
impractical. Instead, a field-editing form opens a *picker*: a normal,
throwaway workspace the user drives with the existing panels (lists, the
=treeref= explorer, …), then confirms a selection that is handed back to the
form. No new selection UI is duplicated into the forms; the picker reuses every
panel, keybinding and future navigation improvement.

The mechanism is a thin protocol over [[*Workspace variables][workspace
variables]], with the only cross-workspace step performed in the GUI backend
(panels can read/write only their own workspace's variables):

1. *Start* — the calling panel invokes the =metafolder.pick.start(spec)= API
   (structured arguments; not a string command). =spec= fully describes the
   picker's initial state — the caller decides everything, because only it
   knows what its picker is for: the active repo, a per-slot panel type with
   seed variables, a workspace =name=, and a =prompt= shown in the picker's
   status bar. The backend creates a workspace, stores
   =pick_request = \{caller_ws, token\}= on it, seeds the requested panels into
   *both* slots (the picker takes the whole window), focuses it, and posts the
   prompt. The =token= is an opaque value the caller mints to match the result
   to the exact widget that requested it.

2. *Navigate* — the picker behaves like any workspace; highlighting an entry
   publishes =selected_metarecord= as usual.

3. *Confirm / cancel* — =pick:confirm= reads the focused picker's
   =pick_request= and its current =selected_metarecord=, closes the picker,
   refocuses =caller_ws=, and writes =pick_result = \{token, uuid\}= into it
   (the single cross-workspace write). =pick:cancel= writes
   =\{token, cancelled: true\}= instead. Closing a picker by hand is a cancel.

The result is *always a metarecord UUID*. The form decides where it lands: for
=ref= it is the value; for =tree_ref= it fills the =parent= slot, leaving the
=name= for the user to type (e.g. picking =/document/test= (uuid =123=) for a
new field =plop= yields =\{123, ""\}=, then the user types =plop= →
=\{123, "plop"\}= → path =/document/test/plop=).

Seeding is configurable per field name, see [[*Picker seeds]]. The calling
panel reads the mapping through the API and folds it into =spec=, so the seed
policy stays the caller's decision.

** Command

A named, invocable operation with an optional parameter string. Commands are
registered by panel types (built-in or custom) and are available in the
command input with autocomplete. Every user-visible behaviour is a command.
Keybindings map key combos to command invocation strings (command name plus
optional parameters).

Every registered command is listed in the autocomplete and invocable by
name, whatever panel is focused: invocations are dispatched to the panel
type that registered the command (in the focused workspace), so acting on
an unfocused — or hidden — panel is legitimate (e.g. the file panel's
arrow keys drive =metarecord-list:next= / =metarecord-list:prev= in the other slot).
Focus-dependent behaviour belongs to keybindings (=when=, see
[[*Keybinding]]), not to commands. Panel commands are registered when the
panel type is instantiated; the shell pre-instantiates every installed
panel type once at startup (hidden, see [[*Panel type]]), so all command
names are known from the start. The registry survives iframe removal
(workspace closed): a later dispatch re-instantiates the owner lazily.

Discoverability rule: every interactive control's action is reachable as a
registered command, and every edit zone has a command that focuses (or
opens) it. So each top-level action button in a panel or the shell maps to a
command (the button invokes it), and each text input has a focus command —
the command input (=command-input:activate=), the metarecord-list query and
columns inputs (=metarecord-list:edit-query=, =metarecord-list:edit-columns=),
the workspace rename (=tab:rename=), the metarecord-detail add-field form
(=metarecord:add-field=). Shell controls map to the panel commands
(=panel:set-type=, =panel:swap=, =panel:split=, =panel:hide=). Transient
controls *inside* a flow already opened by a command — a form's =Cancel=, a
folder picker's navigation, a form's submit — and inherently positional
controls — an individual tab or a specific slot's =×= (their action is
covered by =tab:goto= / =tab:close= / =panel:hide=) — need not each carry
their own command.

Each command carries a =log= flag (default =true=): when set, every
invocation is echoed to the workspace [[*Message view][message panel]] as
=​> name args= before the command runs. Panels register commands with
=log= (=metafolder.commands.register=); the builtin editing primitives
(=editing:*=, =command-input:activate=) register with =log: false= so
keystroke-level actions do not flood the log.

** Keybinding

A mapping from a key combo (e.g. =ctrl+k=, =alt+x=, =g g= for a two-key
sequence) to a command invocation string (e.g. =metarecord-list:set-mode grid=).
Keybindings are loaded from a configuration file and can be edited at runtime.
No keybinding is hard-coded; the defaults ship in the default config file.

The user's =keybindings.toml= contains only their customisations and
overrides; it does not need to list every binding. The GUI engine supplies all
defaults separately and merges them at startup, with the user file taking
precedence.

Each keybinding has two independent scope dimensions:

- =when= (optional) — restricts the binding to a specific panel type. Absent
  means the binding is *global* (fires regardless of which panel type is
  focused). A panel type name (e.g. =metarecord-list=) makes it *local* (fires
  only when that panel type is focused).
- =text-input= (boolean, default =false=) — whether the binding also fires
  when a text input is focused (command input or field editor). =false= means
  the binding is suppressed while the user is typing; =true= means it fires
  regardless.

The two dimensions combine independently:

| =when=             | =text-input= | Fires when                                          |
|--------------------+--------------+-----------------------------------------------------|
| absent             | =false=      | Any panel focused; no text input active (default)   |
| absent             | =true=       | Any panel focused; even inside a text input         |
| =<panel-type>=     | =false=      | Named panel focused; no text input active           |
| =<panel-type>=     | =true=       | Named panel focused; even inside a text input       |

When the same key matches several bindings, a more specific binding wins: local
over global, =text-input = false= over =text-input = true= (the less permissive
one takes precedence when both would fire).

Multi-key sequences never expire: once a prefix is typed, the matcher
waits indefinitely for the next key. While a sequence is pending, the
shell shows the possible continuations (keys and invocations) in a strip
above the status bar — also for sequences typed inside a panel iframe
(the panel matcher forwards its pending state to the shell). =escape=
cancels the pending sequence (it never matches an =escape= binding in
that state); a key that fits no continuation aborts the sequence and is
*swallowed* — it does not fall back to its own single-key binding (a combo
in progress takes priority), and the shell flashes ='<sequence>' is
undefined=. The defaults use this for panel type switching: =s=
followed by one letter (=s l= metarecord-list, =s d= metarecord-detail, =s f=
file, =s m= file-manager, =s o= log, =s r= repos, =s g= message, =s w=
workspace-info).

Config file examples:

#+BEGIN_SRC toml
"escape"  = { command = "editing:unfocus",         text-input = true              }
"ctrl+a"  = { command = "editing:goto-line-start", text-input = true              }
"j"       = { command = "metarecord-list:next",         when = "metarecord-list"            }
"ctrl+e"  = { command = "my-panel:edit",           when = "my-panel", text-input = true }
"ctrl+t"  = { command = "tab:new"                                                  }
#+END_SRC

Panel types may suggest default keybindings via =metafolder.addKeybinding=;
these are applied only when the user has no conflicting binding configured.

** Status bar

A read-only line at the very bottom of the window, always visible and never
focused. Displays status messages, error notifications, the name of the last
executed command (dim text), and a spinner during long operations. Content is
*per-workspace*. When both slots display the same workspace, one status bar is
shown; when the two slots display different workspaces, two status bars are
shown (one per slot). Messages auto-clear after a configurable timeout
(default 5 s).

** Command input

A single-line text input permanently visible just above the status bar. A
configurable keybinding (default suggestion: =:=) gives it keyboard focus;
=Escape= removes the focus (the draft is kept). There is one command input;
it targets the focused slot's workspace. Content (text and cursor position)
is per-workspace: switching the focused slot to a different workspace
restores the command input state of that workspace, including any partial
command being typed.

Accepts:
- A command name (with inline completion) optionally followed by
  space-separated parameters. The suggestion list is filtered fuzzily and
  case-insensitively: the draft is split on whitespace and the terms must
  appear in the command name in order ("con def" matches like the regex
  =.*con.*def.*=). The first (best) suggestion is always selected;
  =Up=/=Down= move the selection. =Tab= writes the selected suggestion
  into the input (without executing); =Enter= always runs exactly what is
  typed — never the selection, since one is always highlighted. Because
  spaces are fuzzy term separators, the list simply empties when typed
  arguments stop matching any command name.
- A shell command prefixed with =!= — executed as a subprocess; stdout/stderr
  goes to the =message= panel type and to the terminal from which the GUI was
  launched. Running a =!= command switches the focused slot to the =message=
  panel type so the output is visible, unless a =message= panel is already
  visible in the workspace.

*** Placeholders in =!= commands

A =!= command line may reference the *selection* and the *active workspace*
through =%=-tokens, expanded *client-side* just before the command runs — so
the GUI can pass their identity to user scripts:

| Token            | Expands to                                                       |
|------------------+------------------------------------------------------------------|
| =%u=             | the selected metarecord's UUID (=workspace.selected_metarecord=) |
| =%v=             | its =version=                                                    |
| =%p=             | the selected file's OS-absolute path (=workspace.selected_paths=) |
| =%r=             | the active repository's UUID (=workspace.active_repo=; =%r:uuid=) |
| =%r:name=        | the active repository's name                                     |
| =%<field>=       | the value of a single-valued scalar field                        |
| =%<field>:path=  | a =TreeRef= field resolved to its path *within that forest*       |
| =%%=             | a literal =%=                                                    |

=<field>= is a field name (=[A-Za-z_][A-Za-z0-9_]*=); =u=, =v=, =p=, =r= are
reserved shorthands. A =%= not followed by =%= or a name is left literal
(e.g. =50% off=). =%p= and =%r= read their workspace var directly, so they do
not need a selected *metarecord* (=%p= works for an untracked file).

=%p= and =%<field>:path= are *not* the same path. =%p= is the selected file's
absolute path on disk (the panel's resolved =selected_paths=). =%<field>:path=
is the position of a =TreeRef= field in *its own* forest, resolved through
=…/resolve-tree= — for =mfr_path= that is the repository-relative tree path, not
the on-disk path. To hand a script the file on disk, use =%p=; use
=%<field>:path= to follow an arbitrary tree field.

Each substituted value is shell-quoted, so paths with spaces are safe.
Substitution is *strict*: if a token cannot be resolved — nothing is selected,
no active repository, no/several selected paths, the field is absent,
multi-valued, or empty, or a bare =%<field>= names a =TreeRef= (use =:path=) —
the command is *not* run and the reason is posted to the status bar. Only =!=
commands are expanded; named commands are not.

** Field editor

An inline editor for a single field value, adapting its widget to the type:

| Value type    | Widget                                         |
|---------------+------------------------------------------------|
| =String=      | Text input                                     |
| =Int=         | Integer input                                  |
| =Float=       | Float input                                    |
| =Bool=        | Checkbox / toggle                              |
| =DateTime=    | Date-time picker (ISO-8601 output)             |
| =Nothing=     | Read-only label "absent"                       |
| =Ref=         | UUID text input (lookup by path available)     |
| =TreeRef=     | Parent UUID + name, or path string             |
| =RefBase=     | UUID text input                                |
| =ExternalRef= | Repo UUID + metarecord UUID                         |

** Script session

An external process communicating with the GUI over the GUI API. A script
session can set panel types, display messages in the status bar, activate the
command input with a prompt, and block waiting for user key input. Only one
script session may hold the input-wait lock at a time.

** Repo indicator

A read-only label in the slot's header bar showing the name of the active
repository for that slot's workspace. It is purely informational; the user
cannot change =active_repo= through it.

To open a repository, the user uses =repos:open= (e.g. via a menu or
keybinding), which switches the focused slot to the =repos= panel type where
the desired repo can be selected or a new one initialised. Selecting a repo
in the =repos= panel type opens a new workspace (tab) with that repo as
=active_repo=, assigned to the focused slot. If the focused slot's workspace
already has =active_repo = null= (e.g. at startup), the selection sets it
in place instead.

* GUI

** Connection to the daemon

The GUI connects to the daemon on 127.0.0.1 at the port configured by the user:
=daemon-port= in =~/.config/metafolder/gui/config.toml= (default =7523=, the
daemon's own default port — so a fresh install connects with no flag),
overridable at launch with =--daemon-port=. At runtime the =daemon:set-url=
command can repoint the proxy at an arbitrary URL (a power-user override).

On startup (and whenever the URL changes) the GUI calls =GET /health=. If the
daemon is unreachable or returns a non-200 response, a prominent error banner
is shown and all daemon-dependent commands are disabled. The GUI itself
(panels, status bar, command input, keybindings, panel type system, and the
GUI API) remains functional.

No authentication: the daemon is local-only and requires none.

** Picker seeds

=ref= value pickers (see [[*Value picker]]) benefit from a sensible starting
query, but the daemon does not record what a =ref= field points to, so the seed
cannot be derived automatically. It is configured per field name in
=~/.config/metafolder/gui/config.toml=, in a =[picker-seeds]= table mapping a
field name to a query string — written in the same syntax as the
=metarecord-list= query box (simplified query text), since that is where the
seed is injected:

#+BEGIN_SRC toml
[picker-seeds]
tag    = 'type = "tag"'
author = 'type = "person"'
#+END_SRC

The map is exposed to panels through =metafolder.config.pickerSeed(field)= so a
calling form can fold the seed into its =pick.start= spec. A field with no
entry yields =null= (the picker opens unfiltered). =tree_ref= pickers do not
need a seed: the natural starting point is the field's own forest.

** Tab bar

A row of tabs at the top of the window. Each tab represents one workspace.
Tabs have a short user-editable name (default: ="Workspace 1"=,
="Workspace 2"=, etc.). Tab state is not persisted across sessions.

Color indicators show the current slot assignments:
- *Green*: workspace displayed in the focused slot.
- *Red*: workspace displayed in the non-focused slot.
- *Green + red*: workspace displayed in both slots simultaneously.
- No color: workspace not currently visible in either slot.

*Left-click* on a tab: assigns that workspace to the focused slot (the slot
keeps focus). *Right-click*: assigns that workspace to the non-focused slot.
Each tab carries a *close button* (=×=) that closes its workspace — the
mouse equivalent of =tab:close=, except it targets the clicked tab's
workspace rather than the focused slot's.

| Command           | Description                                               |
|-------------------+-----------------------------------------------------------|
| =tab:new=         | Create a workspace in BOTH slots (inherits the focused repo unless one is given) |
| =tab:close=       | Close the focused slot's workspace (removes its tab; its slots switch to the previous workspace — the next one for the first tab — or become unassigned when none remains) |
| =tab:rename=      | Rename the focused slot's workspace                       |
| =tab:next=        | Assign the next workspace to the focused slot (wraps)     |
| =tab:prev=        | Assign the previous workspace to the focused slot         |
| =tab:goto=        | Move BOTH panels to workspace N (1-based; N is a parameter) |
| =workspace:next=  | Move BOTH panels to the next workspace (wraps)            |
| =workspace:prev=  | Move BOTH panels to the previous workspace                |

Default keybinding suggestions: =t= for =tab:new=, =w= for =tab:close=,
=n= / =p= for =workspace:next= / =workspace:prev=, =1= … =9= for =tab:goto=.
The default navigation keys move both panels together: the two slots only
end up on different workspaces through an explicit tab click or the
focused-slot =tab:next= / =tab:prev= commands (no default binding). Command
names never bake in a parameter — =tab:goto= takes N as an argument.

Default bindings use plain keys: modifier combos (=ctrl+…=, =alt+…=) are
reserved for bindings with =text-input = true=, which must fire while a
text input is focused and therefore cannot use plain printable keys.

** Layout and panels

The window area (below the tab bar) is divided into:
- *Two panel slots* (left and right), each assigned one workspace (see
  [[*Panel slot]]). At startup only the left slot is visible.
- *Command input*: a single line just above the status bar(s), always
  visible.
- *Status bar*: one or two read-only lines at the very bottom (see
  [[*Status bar]]).

=panel:split= shows the right slot. If no workspace was previously assigned
to it, the focused workspace is shown in both slots; the same panel type
cannot be displayed twice for one workspace, so the new slot falls back to
=metarecord-detail= (pairing the list with the detail view is the expected
split). When the workspace has no active repo, or =metarecord-detail= is
already shown in the other slot, the new slot gets no panel type and the
type picker is presented. No workspace is silently created, except when
the layout has no focused workspace at all (the only meaningful split
then). =panel:unsplit= hides the non-focused slot; the
workspace remains in the tab bar. =panel:split-toggle= splits when a single
slot is visible and unsplits when both are — the everyday command; default
suggestions: =v= for =panel:split-toggle=, =c= for =panel:unsplit=, =o= for
=panel:focus-next=, =x= for =panel:swap=. The focused slot has a visible indicator (border
highlight — theme-controlled); focus switches via =panel:focus-next=.

Each slot has a thin header bar showing the current panel type name and the
repo indicator. The header contains a dropdown that lists all available panel
types (built-in and custom); selecting one switches the displayed panel type
for that slot. The same switch is available via =panel:set-type <name>= and
configurable keybindings.

The header also carries the mouse equivalents of the layout commands: when
the slot is the only visible one, a *split button* (=◫=, =panel:split=
semantics) shows the second slot; when both slots are visible, a *close
button* (=×=) hides that slot — its own slot, unlike =panel:unsplit= which
hides the non-focused one — and a *swap button* (=⇄=, =panel:swap=)
exchanges the panel types of the two slots. The hidden slot's workspace
assignment is remembered either way. A *fullscreen button* (=⛶=,
=panel:fullscreen=) focuses the slot and enters fullscreen.

*Fullscreen*: =panel:fullscreen= toggles an immersive mode showing only the
focused panel — the tab bar, command input, status bar, slot header and the
other slot are all hidden, and the OS window itself goes fullscreen (like a
video player). =Escape= leaves fullscreen (even from inside a panel text
input); the canonical flag lives in the Rust-owned GUI state.

** Status bar

Always-visible read-only area at the bottom. Shows:
- Status messages after commands (e.g. ="Metarecord deleted."=).
- Error messages from failed operations.
- The last executed command name (dim text).
- A spinner during long-running shell commands.

Live progress of daemon /tasks/ (reconcile, query, flush) is *not* shown here
but in a dedicated *task bar* just above the status bar, to avoid saturating it
(see [[*Task bar]] and [[file:spec-tasks.org][spec-tasks.org]] "GUI").

Content is per-workspace. When both slots display the same workspace, one
status bar is shown. When the two slots display different workspaces, two
status bars are shown (one per slot, each reflecting its own workspace).

** Task bar

A dedicated bar directly above the status bar, showing in-flight daemon tasks
(spec-tasks). It polls =GET /tasks= (all loaded repositories) and lists each
=running= / =pending= task with a determinate progress bar when the task
reports =done=/=total=, a spinner otherwise, labelled by =kind=/=phase=. It is
hidden when there are no tasks. Unlike the status bar it is not tied to a
workspace: it surfaces every blocking task on the loaded repositories,
including reconciles started from the CLI or another client.

** Command input

Always visible: a prompt character and a text field above the status
bar(s). A configurable keybinding (default: =:=) gives it keyboard focus;
=Escape= unfocuses it, keeping the draft. The autocomplete suggestions are
shown only while the input is focused. There is one command input; it
targets the focused slot's workspace. Content is per-workspace: switching
the focused slot to a different workspace restores that workspace's command
input state.

Behaviours:
- Typing a command name + =Enter= → invokes the command (=Tab= completes
  shell-style; =Up=/=Down= browse the suggestions, see [[*Command input]]).
- Parameters follow the command name, separated by spaces.
- =!shell-command= + =Enter= → runs in background; output goes to the
  =message= panel type and the launching terminal. The focused slot switches
  to the =message= panel (unless already visible), and =%=-placeholders are
  expanded from the selected metarecord first (see [[*Command input]]).
- When a script calls =POST /gui/prompt=, the command input grabs the focus
  and shows the script-supplied prompt text.

** Keyboard navigation

Every command is reachable from the keyboard. No command requires the mouse.

Within =metarecord-list=:
- =Up= / =Down= move the selection (updates =selected_metarecord=).
- =Enter= or =Right= switches the other slot to =metarecord-detail= (or =file= if
  =selected_paths= is non-empty), opening the slot if hidden. See
  [[*Cross-panel selection]].
- =Space= toggles selection for multi-select (updates =selected_metarecords=).

Within =metarecord-detail=:
- =Tab= / =Shift-Tab= move focus between field rows.
- =Enter= on a field starts inline editing; =Enter= again confirms, =Escape=
  cancels.
- =d= (configurable) deletes the focused field row (with confirmation).

Keybindings are stored in =~/.config/metafolder-gui/keybindings.toml=
(see [[*Keybinding]] for the =when= and =text-input= scope dimensions).
The =config:open= command opens a settings view where they can be edited
interactively; changes take effect immediately.

** Help

In-app help is a dedicated =help= panel type plus three shell builtins. The
builtins are *builtins* (not panel commands) so they work before the help panel
has ever been mounted.

- =help= / =help:help [text]= open the =help= panel in the focused slot. The
  optional =text= is handed to the panel through the =help.request= workspace
  variable (with a nonce so an identical repeated topic still re-triggers). If
  =text= is an *exact name* — a page id, an alias, or a =panel:command= (the
  command's verb is tried first, then its panel-type prefix, so e.g.
  =metarecord-list:edit-query= and =edit-query= both reach the queries page) —
  that page opens directly; otherwise the text greps across all pages. Prefixing
  =text= with =#= forces grep even on an exact name.
- =help:help-cursor= arms a "click-to-help" mode: a =?= pointer (a
  =mf-help-cursor= class on the document element) until the next click, which is
  swallowed and resolved to a help topic instead of reaching its target. Escape
  cancels. Resolution walks the click's =composedPath()= (piercing panel Shadow
  DOM) for the nearest =data-help-topic= attribute; failing that, the nearest
  =data-slot-body= gives the clicked slot's panel type. Shell chrome and panels
  tag their zones with =data-help-topic= (e.g. the metarecord-list query box →
  =edit-query=); everything else falls back to its panel type's page.

The =help= panel renders a set of static HTML pages (=panel-types/help/pages/=,
described by =pages/index.json= as ={id, title, file, aliases[]}=) with a grep
search box on top. There is at least one page per panel type. The queries page
embeds the *current* simplified-query grammar (read at startup via the
=grammar_source= command, =metafolder.query.grammarSource()=). The exact-name
resolution and the grep filter are the pure functions in =/__help.js=
(=resolvePage=, =filterPages=, =resolveClickTopic=), unit-tested in the frontend.

** Context menus

The WebView's native context menu (back/forward/reload/inspect…) is
suppressed everywhere — in the shell and inside every panel iframe — with
one exception: editable text fields keep the native menu (cut/copy/paste
has no cheap HTML replacement). Its Inspect Element metarecord is replaced by
the =devtools:open= command, which opens the WebKit inspector (no default
keybinding; the =devtools= Tauri feature keeps it available in release
builds).

Right-clicking anywhere else shows the *default menu*: =Copy= (the text
selection at open time; disabled when nothing is selected), the everyday
layout commands (=panel:split-toggle=, =panel:swap=) and =devtools:open=.
The shell installs it for its own areas (tab bar, slot headers, status
bar) and the shim installs one per panel document. It stands down where a
more specific right-click behaviour exists: editable text fields, the tab
bar's right-click (assign to the other slot), and any element handler
that opened its own menu. Panels prepend their own metarecords to it with
=metafolder.contextMenu.addDefaultItems(event => items)=.

Where a per-element context menu makes sense, panels show one via
=metafolder.contextMenu(event, items)= (see [[*The =metafolder= API]]); the
shared implementation lives in =/__menu.js=. The menu is rendered in the
document that opened it (a menu opened inside a panel iframe is clipped
to that iframe), positioned at the pointer (flipped at the viewport
edges), and dismissed by =Escape=, a click outside, or opening another
menu. =Up= / =Down= / =Enter= navigate it; while a menu is open the
keybinding matchers stand down so its navigation keys do not trigger
panel commands. Appearance: classes =mf-menu=, =mf-menu-item= (=active=,
=disabled=) and =mf-menu-separator=, with built-in defaults that the user
stylesheet overrides.

** Style and theming

The GUI loads =~/.config/metafolder/gui/style.css= at startup and applies it
globally to the WebView. The stylesheet is shipped and installed by
=metafolder-sync-config= (see [[file:spec-config.org][spec-config]]); a missing
or unreadable file is a startup error, not a fall back to an embedded default.
The =config:open= settings view shows the active CSS path and offers a reload
button; the GUI also auto-reloads on file change via inotify.

** Panel type system

*** Directory layout

Panel types live under =~/.config/metafolder/gui/panel-types/=:

#+BEGIN_SRC
~/.config/metafolder/gui/panel-types/
  metarecord-list/
    index.html        ← entry point
  metarecord-detail/
    index.html
  file/
    index.html
  message/
    index.html
  log/
    index.html
  repos/
    index.html
  my-custom-panel/    ← user-defined
    index.html
#+END_SRC

Built-in types ship as pre-populated examples installed by
=metafolder-sync-config= (see [[file:spec-config.org][spec-config]]). The user
edits them freely on the config repository's =main= branch; upgrades are a git
merge of the shipped =default= branch, not a copy-if-unmodified pass. There is
no =panel-types-defaults/= mirror.

Panel types are served by the GUI's own HTTP server (the GUI API port,
default 7524): =GET /panel/<name>/<file>=, **verbatim** (permissive CORS, so
the shell can fetch/=import()= them from its own origin). The shell mounts a
panel into a Shadow DOM root and applies the user stylesheet itself (no
serve-time injection). Raw local files for the =file= panel type are served by
=GET /fsraw?path=...= with HTTP Range support (media seeking). Video poster
thumbnails are served by =GET /thumbnail?path=...=: a small PNG of one frame
extracted with =ffmpeg= and cached on disk inside the file's repository, under
=<repo>/.metafolder/internal/thumbnails= (keyed by the source's
path/mtime/size), so the cache is scoped to the repo and removed with it. The
owning repository is resolved from the daemon's =GET /repos= (each entry's
=root= and =internal_dir=) by longest-root match — the innermost repo when
repos are nested — not by walking the filesystem, so the external-database
layout is handled and a stray =.metafolder/= directory cannot be mistaken for a
repo. A file inside no repository is not cached and gets no thumbnail — the
panel shows a glyph. A video must never be
handed to an =<img src=/fsraw>= — WebKit would decode the whole file and crash
the web process — so the panels point video tiles at =/thumbnail= instead; a
non-2xx (not a video, not in a repo, missing decoder…) falls back to a type
glyph.

*** The =metafolder= API

The shell passes a =metafolder= object to each panel type's =mount(root,
metafolder)=. It provides three namespaces:

**** =metafolder.daemon=

Async access to the daemon HTTP API (proxied by the Rust backend) plus a
few helpers:

#+BEGIN_SRC js
// Raw call: resolves to {status, body} whatever the status.
const response = await metafolder.daemon.request('GET', '/repos')
// Same, but throws Error(body.error) on status >= 400 and returns the
// body directly — the usual form inside try/catch.
const metarecord = await metafolder.daemon.call('GET', `/repos/${repo}/metarecords/${uuid}`)
// Lazy memoized mfr_path resolution (see "Path display").
await metafolder.daemon.resolvePath(repo, uuid)
await metafolder.daemon.resolveTreeRef(repo, { parent, name })
metafolder.daemon.invalidatePath(repo, uuid)
// Absolute root directory of a repository (cached GET /repos).
const root = await metafolder.daemon.repoRoot(repo)
// Absolute path of the repo's .metafolder/internal/ directory (cached
// GET /repos) — the only path always excluded from tracking.
const internal = await metafolder.daemon.repoInternalDir(repo)
// Absolute filesystem paths of a metarecord: one per mfr_path tree_ref
// (multi-map), unresolvable (stale) refs skipped.
const paths = await metafolder.daemon.metarecordPaths(repo, metarecord)
#+END_SRC

**** =metafolder.query=

Pure query transformations, run **locally** in the GUI backend (the shared
=metafolder-core= engine) — never a daemon round-trip. Both throw on a
parse/grammar error.

#+BEGIN_SRC js
// Compiles normal DSL text to the Query JSON IR.
const ir = await metafolder.query.parse('rating > 3')
// Expands simplified-language text to normal DSL text (relative-date macros
// use the local clock).
const dsl = await metafolder.query.expand('genre=jazz rating>=4')
#+END_SRC

(=metafolder.daemon.parseQuery= / =expandQuery= remain as deprecated aliases
for older panels.)

**** =metafolder.workspace=

Reactive per-workspace key-value store.

#+BEGIN_SRC js
metafolder.workspace.set('selected_metarecord', { uuid, repo })
const metarecord = metafolder.workspace.get('selected_metarecord')
metafolder.workspace.onChange('selected_metarecord', ({ uuid, repo }) => { ... })
#+END_SRC

Standard keys: =active_repo=, =selected_paths=, =selected_metarecord=,
=selected_metarecords=. Panel types may use namespaced custom keys (e.g.
=my-panel:state=).

**** =metafolder.pick=

Opens a value picker (see [[*Value picker]]). =start(spec)= creates a linked
picker workspace in the state =spec= describes and returns its id; the result
arrives later as the =pick_result= workspace variable, which the caller matches
by =token=.

#+BEGIN_SRC js
const token = String(++pickSeq)
metafolder.workspace.onChange('pick_result', (result) => {
  if (!result || result.token !== token) return
  if (!result.cancelled) parentInput.value = result.uuid  // a metarecord uuid
})
await metafolder.pick.start({
  token,
  repo,
  name: 'Pick: tag',
  prompt: 'Select a value for “tag” — Enter to confirm, Esc to cancel',
  panels: {
    left:  { type: 'metarecord-list', vars: { 'metarecord-list:query': seed } },
    right: { type: 'metarecord-detail' },          // optional second slot
  },
})
#+END_SRC

**** =metafolder.config=

Read-only access to GUI configuration the panel may need. =pickerSeed(field)=
returns the configured seed query for a =ref= field name (=metarecord-list=
query syntax), or =null= when none is set (see [[*Picker seeds]]). The caller
folds it into the =pick.start= spec, keeping the seed policy on the calling
side.

#+BEGIN_SRC js
const seed = (await metafolder.config.pickerSeed('tag')) ?? undefined
#+END_SRC

**** =metafolder.commands= and =metafolder.addKeybinding=

#+BEGIN_SRC js
// text-input controls whether the command's default bindings fire in text inputs;
// it is also used as the default for addKeybinding calls for this command.
// (a legacy `scope` option is accepted and ignored: every command is
// listed and invocable whatever panel is focused)
metafolder.commands.register('my-panel:change-mode', {
  label: 'My Panel: Change display mode',
  // log defaults to true: each invocation is echoed to the message panel.
  // Pass log: false for high-frequency commands that would flood the log.
  handler: ({ mode }) => { ... }
})

// Suggest default keybindings (applied only if no conflict in user config).
// Optional third argument: { when, textInput } overrides.
metafolder.addKeybinding('my-panel:change-mode list', 'ctrl+l')
metafolder.addKeybinding('my-panel:change-mode grid', 'ctrl+g')
metafolder.addKeybinding('my-panel:edit-text', 'ctrl+e', { textInput: true })
metafolder.addKeybinding('editing:unfocus',    'escape', { textInput: true })
#+END_SRC

Registered commands appear in the command input autocomplete alongside
built-in commands. The autocomplete shows global commands always, plus local
commands for the currently focused panel type.

**** =metafolder.fs=

Direct filesystem access via the GUI backend (Tauri/Rust). Available to all
panel types; not routed through the daemon.

#+BEGIN_SRC js
// List directory contents
const metarecords = await metafolder.fs.readDir('/path/to/dir')
// metarecords: [{name: "foo.mp3", path: "/path/to/dir/foo.mp3", is_dir: false}, ...]

// Stat a single path

// info: {path, is_dir, size, mtime}
#+END_SRC

**** =metafolder.contextMenu=

HTML context menu at the event's position (see [[*Context menus]]; the
native menu is suppressed). Items are ={label, action?, disabled?}=
objects and ='-'= separators; resolves with the chosen item (its

**** =metafolder.statusBar=

**** =metafolder.bench=

Profiling instrumentation (see [[*Bench harness]]). =measure(name, fn)= times an
async or sync phase and =record(name, durationMs)= logs a pre-computed
duration. Names are =mf:*= labels. Each measure is emitted as a
=performance.measure= (so it also appears in the WebKit inspector Timeline) and
reported to the shell's =GET /gui/bench= buffer. Daemon calls are instrumented
automatically (=mf:daemon <method> <path>=, ids collapsed to =:id=), so panels

#+BEGIN_SRC js

**** UI helpers: =/__ui.js=

A small module served by the GUI server next to the shim, importable by
any panel type (=import { el, formatValue, field } from '/__ui.js'=):

- =el(tag, props, ...children)= — DOM building without =innerHTML=
  (values come from user files; text stays text). =class= accepts a
  string or an array whose falsy metarecords are dropped; =on*= props attach
  event listeners; existing IDL properties are assigned and other keys
  become attributes. Child arrays are flattened and
  =null=/=undefined=/=false= children are skipped.
- =formatValue({type, value})= — the canonical display form of a data
  model =Value= (=∅= for =nothing=, =parent / name= for =tree_ref=,
  =repo :: metarecord= for =externalref=, =String(value)= otherwise).
- =valueEl(value, onOpen)= — same display as a DOM node where metarecord
  references are clickable links (class =ref-link=, styled by the user
  stylesheet): the uuid of =ref=/=refbase=, the =parent= of a
  =tree_ref=, the =metarecord= of an =externalref=. Clicking calls
  =onOpen(uuid, repo)= with =repo= null except for =externalref=.
- =field(metarecord, name)= — first field with that name (fields are a
  multi-map), or =undefined=.
- =thumbnail(guiServer, path, options)= — a thumbnail node for a filesystem
  entry, so the "never put a non-image in an =<img>=" rule lives in one place:
  an image becomes a lazy =<img>= at =/fsraw=, a video a lazy =<img>= at
  =/thumbnail= (a server-extracted poster frame), and directories, every other
  type and a failed load fall back to a type glyph =<span>= (=fileTypeGlyph=).
- =fileTypeGlyph(path, fallback)= — an emoji icon for a file's type (🎬 video,
  🎵 music, 📕 PDF, 🖼️ image, 🗜️ archive…), or =fallback= (default 📄) for an
  unrecognised extension. Used for the =file-manager= list icons and as the
  =thumbnail= fallback.

** Repository management

*** Repo list

Calls =GET /repos= at startup and on explicit refresh. Results populate the
Repo indicator and the =repos= panel type.

*** Init repo

=repos:init= is a command of the =repos= panel type. It transforms the
=repos= panel to show an inline form with an optional =name= input and a
=root= input (the daemon API also accepts an external =metafolder= location,
but the GUI does not expose it). The =root= input is paired with a *Browse…*
button opening an in-panel folder picker (a =metafolder.fs.readDir= navigator,
no native dialog) whose chosen folder fills the input — and prefills the name
with the folder name when blank. Submitting calls =POST /repos/init=. On
success the new repo is adopted in place when the current workspace has no
=active_repo= yet (the focused panel switches to =metarecord-list=); when the
workspace already has a repo, the new one is only added to the list (a status
message points the user to it). Init never spawns a workspace. On =409= an
error message appears in the status bar and the form remains open.

*** Load repo

=repos:load= is a command of the =repos= panel type. It transforms the
=repos= panel to show an inline form with a single =root= input, also paired
with the *Browse…* folder picker. Submitting calls =POST /repos/load=. On
success the loaded repo is adopted in place when the current workspace has no
=active_repo= yet, otherwise it is only added to the list; like init, load
never spawns a workspace.

The intended flow is: =repos:open= (global) → navigate to the =repos= panel
→ select a loaded repo or trigger =repos:init= / =repos:load= from there.

** Metarecord management

*** =metarecord-list= panel type

Displays metarecords for the active repo (=active_repo= workspace variable)
filtered by an embedded query written in the two-zone query editor at the top
of the panel (see [[*Query editor (two zones)]]). An empty query matches all
metarecords.

Default display mode: =table= with columns =mfr_path:path= (resolved path),
=mfr_type=, =&version=. The =metarecord-list:set-mode= command switches between
display modes.

The columns input is a separate row below the query editor, with its own
/Apply/ button (=metarecord-list:edit-columns= focuses it). The panel always
queries with =select: '*'=, so applying new columns is a pure re-render — no
request.

Each column is one token of the columns input (space- or comma-separated,
with spaces around =|= tolerated; an empty input restores the defaults;
invalid tokens are reported under the input and the previous columns are
kept). A token combines two orthogonal operators — a /projection/ (=:mode=)
and a /follow/ (=>=) — plus a fallback (=|=):

- =name= — the raw field value(s). A multi-map field shows every row in
  the cell, comma-joined. =version= is an ordinary field name here.
- =name:mode= — projects a =tree_ref= value: =:name= (leaf name), =:uuid=
  (parent uuid), =:path= (the path from the repository root — the root
  shows =/=; a stale/unresolved =tree_ref= falls back to its leaf name),
  =:raw= (the parent/name couple, the default). A mode that does not apply
  to the value's type falls back to the raw display.
- =name>sub= — follows a =ref=/=refbase= value to the target metarecord and
  shows its =sub= field rows (comma-joined; e.g. =tag>label=). =sub= may
  itself be projected (=tag>path:name=, or =tag>path:path= to resolve the
  target's tree path). One =>= only — no deep chains. A missing target
  metarecord falls back to the raw uuid; a present target whose =sub= field
  is absent yields no value (so a fallback can take over).
- =a | b= — fallback: the cell shows the first alternative that yields a
  value (e.g. =tag>label | tag>path:name= — the label if any, else the
  tag's leaf name).
- =&uuid= / =&version= — the metarecord's own uuid and version. These are
  metarecord metadata, not fields (=&= cannot appear in a field name), so a
  user field named =version= never collides with them.

Sorting always uses the underlying raw field (the daemon sorts raw
values): clicking a =tag>label= header sorts by the =tag= uuids (a
fallback sorts by its first alternative's field). Meta
(=&=) columns are not sortable. Column widths are adjustable by dragging
the right edge of a header. The column list and the widths are persisted
in the =metarecord-list:columns= and =metarecord-list:column-widths= workspace
variables, so they survive the panel and can be set by scripts.

Supports sorting by column, multi-selection (=selected_metarecords=), and full
keyboard navigation.

When the user moves the selection, =metarecord-list= writes the highlighted metarecord
to =workspace.selected_metarecord= and sets =workspace.selected_paths= to the list
of resolved absolute paths from all =mfr_path= field rows of that metarecord
(empty list if the metarecord has no valid =mfr_path=). Other panels update
reactively (see [[*Cross-panel selection]]).

**** Query editor (two zones)

The query is written in the [[file:spec-query.org][simplified query language]]
and transpiled to the normal DSL before execution. The editor exposes both
forms, so the user can write quickly *and* drop to the full DSL when a query is
too specific for the simplified surface — and learn the mapping by watching it.

- *Zone A (simplified)* is always visible and has its own /Apply/ button;
  =metarecord-list:edit-query= focuses it. An empty zone A matches all.
- A toggle reveals *zone B (normal DSL)* below A. While B is shown and **not
  frozen**, it is a read-only live mirror of =expand(A)= (debounced ~120–150 ms,
  via =metafolder.query.expand=). A *Freeze* checkbox next to B decouples
  it: B becomes editable and authoritative, and editing A no longer changes it.
  Unfreezing re-syncs B from =expand(A)= — manual edits to B are then discarded
  (accepted).

Execution always runs the content of B; when B is hidden, B is computed as
=expand(A)= on Apply. The resulting DSL text is compiled with
=metafolder.query.parse= and the query is issued (with =select: '*'=, as
above).

Errors appear under the zone that produced them: a grammar/expansion error under
A, a =parse_query= error under B (where the offending DSL text is visible when B
is shown).

The editor state persists in workspace variables, so it survives the panel and
can be set by scripts:

| Variable                        | Meaning                                                                    |
|---------------------------------+----------------------------------------------------------------------------|
| =metarecord-list:query=         | zone A text (simplified)                                                   |
| =metarecord-list:normal-shown=  | whether zone B is revealed                                                 |
| =metarecord-list:normal-frozen= | whether zone B is frozen (decoupled, authoritative)                        |
| =metarecord-list:normal-query=  | zone B text — authoritative while frozen, otherwise a cache of =expand(A)= |

*** =metarecord-detail= panel type

When =selected_metarecord= is =null=, displays a centred "No metarecord selected"
message and the /New metarecord/ button.

Otherwise calls =GET .../metarecords/:uuid= and displays all fields. Each field
row shows name, value, an edit button, and a delete button.

The panel reloads the displayed metarecord when another panel signals a data
change through the =metarecords:dirty= workspace variable (log navigation,
file-manager tracking, …) — unless an edit is in progress: reloading
would silently discard it, so the stale view persists until the edit is
saved or cancelled.

Reference values are clickable (=valueEl=, see [[*UI helpers: =/__ui.js=]]):
clicking the uuid of a =ref=/=refbase=, the parent of a =tree_ref= or the
metarecord of an =externalref= sets =selected_metarecord= to the referenced metarecord —
the panel follows it like any other selection change (the edit guard below
applies). For an =externalref= the target repository is the reference's;
loading fails with the usual error message when that repository is not
loaded.

Keyboard: =Tab= moves between rows, =Enter= edits, =d= deletes (with
confirmation).

At the bottom of the panel:
- /New metarecord/ button (or =metarecord:create= command) → clears the form and
  presents an empty field list for creating a new metarecord. Submitting calls
  =POST .../metarecords=; on success the new metarecord is set as =selected_metarecord=
  and the list in the linked =metarecord-list= panel (if any) refreshes.
- /Add field/ → =POST .../metarecords/:uuid/fields=
- /Set field/ → =PATCH .../metarecords/:uuid=
- /Delete metarecord/ → confirmation → =DELETE .../metarecords/:uuid=
- /Watch and reconcile/ → visible when the metarecord's =mf_watch= field is
  =false= or absent. Sets =mf_watch = true= (=PATCH=), then calls
  =POST .../metarecords/:uuid/reconcile=. A spinner appears in the status bar
  during the operation; the result is appended to the =message= panel type.
  Useful after adding a metarecord via =file-manager:add= to activate tracking and
  populate the subtree.

Protected =mfr_*= fields are read-only by default; an advanced toggle exposes
a =force= checkbox.

*** =file= panel type

Displays a preview of the file at the currently active path from
=selected_paths=. When =selected_paths= is empty, shows a "no file selected"
message. When =selected_paths= has more than one element (metarecord accessible at
multiple locations), a dropdown in the panel header lets the user choose which
path to preview. Uses the HTML5 =<img>=, =<audio>=, or =<video>= element
depending on the file type; falls back to a plain-text view for text files;
shows "no preview available" for unsupported formats. Works with both tracked
and untracked files.

When the active path is a directory, the panel instead lists its contents as
a thumbnail grid (images as thumbnails via =/fsraw=, videos as poster-frame
thumbnails via =/thumbnail=, other entries as a type glyph). The grid is
rendered in windows (page size from =[page-size].file= in =config.toml=, default
150 — larger than =metarecord-list='s since each tile is only a lazy =<img>=,
with more appended on scroll) so a folder with many thousands of files neither
freezes on open nor holds every thumbnail at once; a footer shows =shown/total=. Clicking an entry drills in — into a
subdirectory's listing, or a file's inline preview — and a back button in the
header steps back out, returning to the selection once the originally selected
path is reached.

Before creating an =<audio>= / =<video>= element, the panel queries
=GET /__media-support= on the GUI server and, when the required GStreamer
elements are missing, shows an explanatory message instead of attempting
playback (see [[*Media codec support]]).

When an image or video is previewed, a toolbar offers zoom controls, mirrored
by commands and default keybindings (active only while the =file= panel is
focused):

| Command           | Default key | Effect                                              |
|-------------------+-------------+-----------------------------------------------------|
| =file:zoom-in=    | =+=         | Enlarge by one step                                 |
| =file:zoom-out=   | =-=         | Shrink by one step                                  |
| =file:zoom-fit=   | ~=~         | Fit the available space (aspect preserved) — default |
| =file:zoom-reset= | =0=         | Original (1:1) size                                 |

"Fit" fills the viewer, scaling to whichever of full width or full height comes
first without distortion. A manual zoom (=+= / =-= / =0=) sizes the media in
pixels relative to its natural dimensions, with the viewer scrolling when the
result overflows; the first =+= / =-= continues smoothly from whatever "fit" is
currently showing. The chosen zoom state persists while browsing other files.
The "+" key is spelled =plus= in keybindings TOML (a literal =+= is the chord
separator).

Default keybindings map the arrow keys, when the =file= panel is focused,
to =metarecord-list:next= / =metarecord-list:prev=: they move the metarecord-list
selection of the same workspace (commands dispatch to their owning panel
type), and the preview follows through =selected_paths= — flipping through
files without leaving the preview. Pure keybindings, no coupling in
either panel; rebindable like everything else.

*** =file-manager= panel type

A filesystem tree navigator. The GUI backend (Tauri/Rust) reads the disk
directly; the daemon is not involved in directory listing.

*Display*: a tree/list of files and directories, each row prefixed with a
type icon (📁 for a directory, otherwise =fileTypeGlyph= — 🎬 video, 🎵 music,
📕 PDF…), with a visual distinction
between tracked metarecords (have =mfr_path= metadata in the active repo) and
untracked ones. When =active_repo= is =null=, all metarecords are shown as
untracked (no distinction is possible). The listing starts with two
synthetic rows, =.= (the displayed directory itself) and =..= (its parent),
as in a Unix directory listing; this makes the directory's own metarecord —
including the repo root — selectable. The tracked distinction is resolved
per displayed directory (the direct children plus the =.= / =..= rows),
never by scanning the whole repository.

The active repo's =.metafolder/internal/= directory and its contents are
shown in red (=--mf-error=) with an =internal= badge: they are hard-excluded
from tracking (see [[file:spec-file-tracking.org][spec-file-tracking.org]]),
so the add-to-DB actions are disabled on them. The path comes from the
=internal_dir= field of =GET /repos= (=metafolder.daemon.repoInternalDir=) —
it is matched absolutely, not by name, since the metafolder may live
anywhere.

*Root behaviour*: a checkbox (default: checked) constrains navigation to the
=active_repo= root. When unchecked, the user can navigate anywhere on the
disk. Adding files to the DB is only available for paths inside the
=active_repo= root; attempting it outside returns an error in the status bar.
When =active_repo= is =null=, the checkbox is disabled and =file-manager:add=
/ =file-manager:goto-root= return an error.

*Selection*: moving the cursor sets =workspace.selected_paths= to a
single-element list containing the highlighted path. If the path has an
existing metadata metarecord in the active repo, =workspace.selected_metarecord= is also
updated; otherwise it is set to =null=.

*Keyboard*: =Up= / =Down= navigate; =Enter= expands a directory or confirms
selection for a file; =Backspace= goes up one level; =r= refreshes.

*Listing*: the directory's entries are rendered in windows (page size from
=[page-size].file-manager= in =config.toml=, default 200 — the largest of the
list panels, since a plain text row is the cheapest to build — with more
revealed on scroll), so a directory with many thousands of files stays
responsive; a footer shows =shown/total entries= (mirroring the
=metarecord-list= count). The tracked status of the rows is queried *per
rendered window* (a =Follows(parent) AND Matches(^(name…)$)= query over just
that window's names), not for every child up front, so the per-entry daemon
lookups are bounded to what is on screen. =file-manager:refresh= re-reads the
current directory and its tracked status, preserving the cursor.

*Context menu*: right-clicking a row moves the cursor there and opens an
HTML menu ([[*Context menus]]): =Open= (directories) and =Track= (disabled
when there is no active repo, the path is already tracked, or it is
outside the repo root).

| Command                   | Description                                              |
|---------------------------+----------------------------------------------------------|
| =file-manager:add=        | Add the selected path to the DB via =POST .../track= (creates metarecord with =mf_watch = false=; parent metarecords created automatically). Error if already tracked (=409=) or outside =active_repo= root. |
| =file-manager:goto-root=  | Jump to the =active_repo= root directory                 |
| =file-manager:refresh=    | Reload the current directory (and tracked status), keeping the cursor |
| =file-manager:toggle-root= | Toggle the "constrain to root" checkbox                 |

*** =treeref= panel type

A navigator for a *TreeRef forest*, the way =file-manager= navigates the disk
— but over any TreeRef field (=mfr_path=, or a user tag tree), driven entirely
by the daemon. The primary use is observing a tag hierarchy.

*Field picker*: a dropdown of the repo's TreeRef field names
(=GET /repos/:repo/fields?type=tree_ref=, [[file:spec-data-model.org][spec-data-model.org]]); =mfr_path= is the
default. Changing the field resets navigation to the roots.

*Navigation*: by *parent UUID*, not by path string, so a name containing =/=
is handled correctly. The view *starts at the forest roots*, fetched from
=GET /repos/:repo/tree/roots?field=<field>= ([[file:spec-data-model.org][spec-data-model.org]]) — the roots' parent
is the root sentinel, which =Follows= cannot address (the empty path resolves
to the sentinel's *children*, and only when a root is literally named =""=, as
=mfr_path='s repo-root metarecord is). A node =U='s direct children are then
fetched with =Follows{field, target: \{uuid_in: [U]\}\}=, each row showing the
single name component it contributes (read directly from its =tree_ref= value —
no =resolve-tree= round-trip). The empty-named =mfr_path= root displays as =/=.
A breadcrumb shows the current path; clicking a crumb jumps there.
Branch-vs-leaf is *not* predicted up front: descending into a leaf simply shows
an empty list. Children are fetched page by page (=[page-size].treeref=,
default 200) with more on scroll; the (few) roots are fetched in one call.

*Selection*: highlighting a node sets =selected_metarecord= (=\{uuid, repo\}=,
so =metarecord-detail= follows) and =selected_treeref=
(=\{repo, field, uuid, path\}=, consumed by =ref-list=).

*Keyboard*: =Up=/=Down= (or =j=/=k=) move; =Enter=/=Right= descend;
=Backspace=/=Left= go up; =r= refreshes.

| Command            | Description                                  |
|--------------------+----------------------------------------------|
| =treeref:descend=  | Descend into the selected node               |
| =treeref:parent=   | Go up one level                              |
| =treeref:root=     | Jump to the forest roots                     |
| =treeref:refresh=  | Reload the field list and current children   |

*** =ref-list= panel type

Lists the metarecords whose chosen *Ref field* points to the TreeRef node
currently selected in a =treeref= panel — the companion view that answers "what
is tagged with this?". It reacts to =selected_treeref=.

*Field picker*: a dropdown of the repo's Ref field names
(=GET /repos/:repo/fields?type=ref=). A *scope* selector chooses how the
selected node is matched:

- =exact= — the Ref referent is the node itself:
  =Follows\{refField, target: \{uuid_in: [node]\}\}=.
- =+ descendants= — the referent is the node *or any descendant* of it in the
  TreeRef field's forest (classic tag inheritance — selecting =music= also
  surfaces things tagged =music/rock=). The target is
  =Or(uuid_in[node], FollowsTransitive\{treeField, uuid_in[node]\})= (the tree
  field comes from =selected_treeref.field=; =FollowsTransitive= excludes its
  own roots, so the node itself is OR-ed back in).

*Display*: one row per referencing metarecord, labelled by its =mfr_path=
basename (resolved from the cache) or a short UUID when it has no path.
Selecting a row sets =selected_metarecord= / =selected_paths=; =Enter= reveals
it in the other slot (=file= when it has a path, else =metarecord-detail=).
Paged on scroll (=[page-size].ref-list=, default 100).

| Command               | Description                                   |
|-----------------------+-----------------------------------------------|
| =ref-list:open=       | Reveal the selection in the other slot        |
| =ref-list:toggle-scope= | Toggle between =exact= and =+ descendants=  |
| =ref-list:refresh=    | Reload the field list and results             |

*** =workspace-info= panel type

Displays all workspace variables for the current workspace: =active_repo=,
=selected_paths=, =selected_metarecord=, =selected_metarecords=, and any custom
namespaced variables. Values are shown in JSON format and update reactively
as they change. Read-only; useful for debugging panel type communication and
for monitoring GUI state from scripts.

*** Cross-panel selection

Selecting a metarecord in =metarecord-list= or =file-manager= updates the workspace
variables =selected_paths= and/or =selected_metarecord= (see the respective panel
type sections for details). Any slot in the same workspace displaying
=metarecord-detail=, =file=, or =workspace-info= subscribes reactively and updates
automatically. Slots showing different workspaces are unaffected.

*Double-click* (or =Enter= / =Right= keybinding) in =metarecord-list=: if the
other slot shows the same workspace and is displaying a different panel type,
it is switched to =metarecord-detail= (or =file= if the metarecord has a valid
=mfr_path=). If the other slot is hidden, it is opened (showing the same
workspace).

*Edit guard*: before =metarecord-detail= replaces its content with a new metarecord, it
checks whether any field is being edited. If so, an alert is shown:
="Unsaved changes — discard and switch metarecord? [Confirm / Cancel]"=. The switch
proceeds only if the user confirms.

*** Batch set

When =selected_metarecords= is non-empty, =metarecord:batch-set= opens a dialog for
field name and value and sets it on every selected metarecord. (v1
implementation note: the query IR has no UUID predicate, so this is one
=PATCH .../metarecords/:uuid= per selected metarecord rather than a single
=POST .../set=; to be revisited if a UUID predicate is added.)

** Message view

The =message= panel type is a scrollable, append-only log, *per-workspace*.
It receives:
- All messages posted to the active workspace's status bar (with timestamp).
- Every command invocation dispatched in this workspace, echoed as
  =​> name args= before the command runs — unless the command opted out of
  logging at registration (see "Command", the =log= flag). Basic editing
  primitives (=editing:*=, =command-input:activate=) opt out to avoid noise.
- stdout/stderr of shell commands run via =!= in this workspace (echoed as
  =$ command=).
- Reconcile progress and results triggered from this workspace.
- Any message posted by a script via =POST /gui/message= targeting this
  workspace (also shown in the status bar as a transient notification).

The log is cleared by the =message:clear= command. It is not persisted across
sessions.

** Reconcile

=reconcile:run= calls =POST .../reconcile= (asynchronous — 202 + task id, see
[[file:spec-tasks.org][spec-tasks.org]]) and polls the task only to know when it
finishes; live progress is shown by the [[*Task bar]], not the status bar. On
completion:
- Status bar: ="Reconcile: N created, M updated, P unchanged."=
- Full output (per-file actions and candidate stale matches) is appended to the
  =message= panel type.

** Event log

*** Log view (=log= panel type)

Displays revisions in reverse chronological order. Columns: =rev_id=,
timestamp, operation count, =status= (HEAD marker). A =Graph= checkbox
(command =log:toggle-graph=) switches the presentation, refetching the log:

- Unchecked (default) — calls =GET .../log?mode=active=: the active line
  through HEAD only. After a rollback the most-recent forward continuation
  stays listed (so the user can navigate forward again), but revisions on
  divergent branches are hidden.
- Checked — calls =GET .../log?mode=tree= and draws every branch as a graph:
  a leading monospace gutter cell with connector rows, the active line in the
  leftmost column and divergent branches opening a column to the right.

A click — or =up= / =down= (also =k= / =j=), via =log:prev= / =log:next= —
moves the selection (over every listed node, in either presentation). A double
click or =Enter= (=log:toggle-ops=) expands the revision to show its individual
operations (and collapses it again).

*** Revision commands

- =log:rollback= → confirmation → =POST .../rollback= targeting the last
  operation of the selected revision. The target may be an ancestor
  (rollback) or a descendant / another branch (redo): the same command
  navigates in both directions, which is also how a rollback is undone.
  Refreshes =metarecord-list= and =metarecord-detail= afterwards (=metarecords:dirty=).
- =log:undo= / =log:redo= — shell builtins (they work without the log
  panel, defaults =ctrl+z= / =ctrl+shift+z=; the log panel also has Undo /
  Redo toolbar buttons): navigate the active repo's log one revision at a
  time. Undo targets ={"prev_revision": true}=; redo re-applies the
  revision of HEAD's child in the operation tree — the daemon has no
  direct target for it, so the GUI reads =GET .../log?mode=tree= and
  targets the last operation of that revision (the most recent child when
  a rollback created several branches). At a tip there is nothing to
  redo. No confirmation prompt: the operation is itself undoable, that is
  the point. Refreshes the panels through =metarecords:dirty=.
- =log:mark-checkpoint= → =PATCH .../log/revisions/:rev_id=.

*** Prune log

=log:prune= opens a dialog (target revision or "keep last N revisions") and
calls =POST .../log/prune=.

** Scripting / GUI API

The GUI exposes a local HTTP API on =127.0.0.1:7524= (=gui-port= in
=gui/config.toml=, overridable with =--gui-port=). External scripts can drive
the GUI: set panel types, display messages, wait for user input.

All endpoints are available as long as the GUI is running, regardless of
daemon connectivity.

*** =GET /gui/workspaces=

Returns the list of all open workspaces.

#+BEGIN_SRC json
[
  {"id": "ws-1", "name": "Workspace 1", "active_repo": "<repo_uuid>"},
  {"id": "ws-2", "name": "Workspace 2", "active_repo": null}
]
#+END_SRC

*** =POST /gui/workspaces=

Creates a new workspace. Returns its id.

Request body (all fields optional):
#+BEGIN_SRC json
{"active_repo": "<repo_uuid>"}
#+END_SRC

If =active_repo= is absent, the workspace uses the daemon's first loaded
repository (from =GET /repos=), or =null= if no repositories are loaded.

Response: ={"id": "ws-3"}=

*** =DELETE /gui/workspaces/:id=

Closes a workspace. Any slot that had this workspace assigned switches to
the previous workspace in tab order (the next one when the first tab was
closed; a slot hidden by =panel:unsplit= stays hidden), or becomes
unassigned (shows "No workspace selected") when no workspace remains.
Returns =204 No Content=.

*** =GET /gui/layout=

Returns the current slot assignments.

#+BEGIN_SRC json
{"left": "ws-1", "right": "ws-2"}
#+END_SRC

A slot value of =null= means the slot is hidden (no workspace assigned).

*** =PUT /gui/layout=

Assigns workspaces to slots. Only the keys present in the request body are
updated; omitted keys are unchanged.

#+BEGIN_SRC json
{"left": "ws-1", "right": "ws-2"}
#+END_SRC

Setting a slot to =null= hides it. Returns =200 OK=, body ={}=.

*** =PUT /gui/panels/:slot/view=

Sets the panel type displayed in slot =:slot= (=left= or =right=). If the slot
is hidden, it is shown first.

Request body examples:

#+BEGIN_SRC json
{"type": "metarecord-list"}
{"type": "metarecord-detail"}
{"type": "file", "path": "/absolute/path/to/image.jpg"}
{"type": "message"}
{"type": "log"}
{"type": "my-custom-panel", "state": { ... }}
#+END_SRC

When =type= is =file= and =path= is provided, the workspace variable
=selected_paths= is set to =[path]= and =selected_metarecord= is set to the
matching metadata metarecord (or =null= if the file is untracked). Omitting =path=
leaves =selected_paths= unchanged and the =file= panel type displays whatever
path was already active.

Response: =200 OK=, body ={}=.

*** =GET /gui/panels/:slot/view=

Returns the current panel type descriptor for slot =:slot=, including a
=status= field (="loading"= / ="ready"=).

*** =POST /gui/command=

Runs an arbitrary command invocation through the *same dispatcher the command
input and keybindings use* — there is a single command path, so internal and
external invocation stay in lockstep. The =invocation= string is parsed exactly
like command-input text (whitespace-separated name and arguments, quotes
honoured); a leading =!= runs a shell command.

The request blocks until the command resolves: the GUI shell runs =dispatch()=
and reports the outcome. Unlike =POST /gui/input= / =POST /gui/prompt=,
concurrent command calls are allowed (each gets its own id), since this is
external control rather than a modal wait.

Request body:
#+BEGIN_SRC json
{"invocation": "panel:set-type file", "timeout_ms": null}
#+END_SRC

Responses: ={"event": "ok"}= / ={"event": "error", "message": "unknown command: …"}=
/ ={"event": "timeout"}= / ={"event": "closed"}=. A missing or empty
=invocation= is a =400=.

Note: the call resolves when =dispatch()= and its awaited work complete, which
for commands that open or refresh a panel does *not* include the panel's own
asynchronous render inside its iframe.

*** =GET /gui/bench= / =POST /gui/bench/clear=

The profiling harness (see [[*Bench harness]]). Panels time their phases with
=metafolder.bench.measure= and the shim auto-instruments every daemon call;
each measure is reported to the shell, which appends it to an in-memory buffer.
=GET /gui/bench= snapshots that buffer; =POST /gui/bench/clear= empties it.

#+BEGIN_SRC json
{"records": [
  {"name": "mf:daemon POST /repos/:id/query", "duration_ms": 8.4},
  {"name": "mf:list:enrich", "duration_ms": 3.1},
  {"name": "mf:list:render", "duration_ms": 2.0}
]}
#+END_SRC

A driver script clears the buffer, runs a reproducible scenario through
=POST /gui/command=, then reads the per-phase timings back.

*** =POST /gui/message=

Posts a message to a workspace's status bar (transient) and appends it to
that workspace's =message= panel type log. An optional =?workspace_id=<id>=
query parameter targets a specific workspace; defaults to the focused slot's
workspace.

Request body:
#+BEGIN_SRC json
{"text": "← left  → right  Esc to stop", "timeout_ms": null}
#+END_SRC

=timeout_ms=: null → status bar message persists until the next message;
integer → auto-clears after that many milliseconds.

*** =POST /gui/input=

Registers an input wait and blocks until the =answer:send= command is
executed (by any means: a temporary binding installed by this request, a
permanent user binding, or a manual invocation via the command input).
No keystrokes are suppressed; all normal shortcuts continue to work.

The optional =keys= array installs temporary global keybindings for the
duration of the wait: each listed key is bound to =answer:send <key>=
(scope: global, =text-input = false=). These bindings are removed when the
request resolves or times out. If =keys= is absent or empty, no temporary
bindings are installed; the user can still resolve the wait by typing
=answer:send <value>= manually in the command input.

Only one =POST /gui/input= at a time; concurrent calls return =409=.

Request body:
#+BEGIN_SRC json
{"keys": ["left", "right", "escape"], "timeout_ms": null}
#+END_SRC

Responses: ={"event": "answer", "value": "left"}= / ={"event": "timeout"}= /
={"event": "closed"}=.

*** =POST /gui/prompt=

Activates the command input with a custom prompt. Blocks until =Enter= or
=Escape=. Mutually exclusive with =POST /gui/input= (same lock).

Request body:
={"prompt": "Enter a rating (1-5): ", "completions": [], "timeout_ms": null}=

The optional =completions= array is offered by the command input's
autocomplete while the prompt is active (same prefix-then-substring
ranking and Tab/arrow navigation as command completion; accepting a value
writes it into the input without submitting). The user may still type any
free-form answer.

Responses: ={"event": "confirm", "text": "4"}= / ={"event": "cancel"}=.

*** =GET /gui/status=

*** CLI: =mf gui=

The =mf= CLI exposes the scripting API as plain-text subcommands, so shell
scripts need neither =curl= nor =jq=. The GUI base URL comes from
=--gui-url= / =METAFOLDER_GUI_URL=, else =gui-port= in the GUI
=config.toml= (see [[*GUI API port persistence]]), else the default
=http://127.0.0.1:7524=.

| Command                                              | Effect                                                             |
|------------------------------------------------------+--------------------------------------------------------------------|
| =mf gui status=                                      | =GET /gui/status=, pretty-printed JSON                             |
| =mf gui repo=                                        | Prints the active repo of the focused workspace (exit 1 if none)   |
| =mf gui workspace new [--repo <uuid>]=               | Creates a workspace, prints its id                                 |
| =mf gui workspace rm <id>=                           | Closes a workspace                                                 |
| =mf gui layout [<slot> [<ws or ->]]=                 | Prints both slots / one slot / assigns (=-= hides the slot)        |
| =mf gui view <slot> [<type>] [--path] [--state]=     | Prints or sets the panel type of a slot                            |
| =mf gui message <text> [--workspace] [--timeout-ms]= | Posts a status bar message                                         |
| =mf gui command <invocation>... [--timeout-ms]=      | Runs a command invocation through the GUI (exit 1 on error/timeout) |
| =mf gui bench [--clear]=                             | Prints the recorded bench measures (JSON), or clears the buffer    |
| =mf gui input <key>... [--timeout-ms]=               | Waits for one of the keys, prints it (exit 1 on timeout/close)     |
| =mf gui prompt <text> [--completion <v>]... [--completions-stdin] [--timeout-ms]= | Prompts in the command input, prints the answer (exit 1 on cancel) |

*** Example: image comparison script

#+BEGIN_SRC bash
#!/usr/bin/env bash
GUI="http://127.0.0.1:7524"

# Create two dedicated workspaces for the comparison
WS_LEFT=$(curl -s -X POST $GUI/gui/workspaces \
               -H "Content-Type: application/json" -d '{}' | jq -r .id)
WS_RIGHT=$(curl -s -X POST $GUI/gui/workspaces \
                -H "Content-Type: application/json" -d '{}' | jq -r .id)

# Assign them to the two panel slots
curl -s -X PUT $GUI/gui/layout \
     -H "Content-Type: application/json" \
     -d "{\"left\": \"$WS_LEFT\", \"right\": \"$WS_RIGHT\"}"

while true; do
  A=$(find /photos -name "*.jpg" | shuf -n1)
  B=$(find /photos -name "*.jpg" | shuf -n1)

  curl -s -X PUT $GUI/gui/panels/left/view \
       -H "Content-Type: application/json" \
       -d "{\"type\":\"file\",\"path\":\"$A\"}"
  curl -s -X PUT $GUI/gui/panels/right/view \
       -H "Content-Type: application/json" \
       -d "{\"type\":\"file\",\"path\":\"$B\"}"
  curl -s -X POST "$GUI/gui/message?workspace_id=$WS_LEFT" \
       -H "Content-Type: application/json" \
       -d '{"text":"← left  → right  Esc to stop"}'

  RESP=$(curl -s -X POST $GUI/gui/input \
              -H "Content-Type: application/json" \
              -d '{"keys":["left","right","escape"]}')
  KEY=$(echo "$RESP" | jq -r .value)

  [ "$KEY" = "escape" ] && break

  WINNER=$( [ "$KEY" = "left" ] && echo "$A" || echo "$B" )
  # metarecord winner as metadata via daemon …
done

# Clean up the temporary workspaces
curl -s -X DELETE "$GUI/gui/workspaces/$WS_LEFT"
curl -s -X DELETE "$GUI/gui/workspaces/$WS_RIGHT"
#+END_SRC

* Resolved design decisions (v1)

** Frontend framework

Svelte 5 + Vite for the shell. Built-in panel types are plain HTML/JS (no
build step) so users can read and copy them as examples.

** =metafolder= API delivery mechanism

Panel types run **in the shell's JS realm**, not in iframes. Each instance
(one per (workspace, panel type), kept alive while hidden for state retention,
never reparented) is a host element with its own **Shadow DOM** root for CSS
isolation. The shell fetches the panel's =index.html= into the shadow root
(scripts stripped), =import()=s its =main.js= and calls the contract
=export function mount(root, metafolder)= (returning an optional cleanup
function). =metafolder= is a plain object whose methods call the Rust backend
directly — there is no =postMessage= bridge. Because everything shares one
realm, panel key events bubble to the shell's single key matcher (text-input
detection pierces shadow roots via =composedPath()=).

This abandons iframe isolation deliberately: it lets a later shared data cache
live in the realm and be read by every panel without per-access copies. Panels
are **trusted code** (they already have filesystem and shell access through the
API — see [[*Custom panel type trust model]]); a throw in =mount= is contained
by an error boundary, but a panel can otherwise touch shared state.

A corollary: the same panel type cannot be displayed in both slots for the
same workspace (one instance cannot render twice); =panel:set-type= rejects
this with a status bar error.

** Daemon access

Panels and the shell reach the daemon through the Rust backend (a reqwest
proxy exposed as =metafolder.daemon.request=). The WebView cannot call the
daemon directly (CORS), and the daemon stays GUI-agnostic.

** Data cache

Because panels share one JS realm, a single in-realm cache holds daemon data —
read directly by every panel, no per-panel copy. It has three stores per repo:
metarecords (=uuid → metarecord=), TreeRef paths (=(uuid, field) → [path]=) and
query results (=query → [uuid]=).

Panels use =metafolder.cache=: =query(repo, body)= (returns the page's uuids +
=nextCursor=/=total=, populating the entity store from a =select '*'= result),
=fetchMetarecords= / =fetchTreeRefs= (fetch only the uuids not already cached),
and the synchronous =readMetarecord= / =readTreeRef= used during render —
returning =cache.REFRESH= when the datum is absent/invalidated, on which the
panel renders a placeholder and schedules a refresh. So a metarecord-list query
populates the entities the metarecord-detail panel then reads for free (the
list↔detail dedup). =daemon.call= stays for writes and non-cacheable endpoints;
its query / single-metarecord / batch / tree-resolve reads are also served from
the cache transparently. Cached data is read-only — panels never mutate it.

Freshness comes from the daemon change feed (=GET /log/since=, see
[[file:spec-event-log.org][spec-event-log]]): a background poll (~7 s) plus
deliberate points (a query, a refresh, a panel becoming visible) compare the
log =head=; on a change the touched metarecords are invalidated by delta (fine)
and query results cleared (coarse). A panel's own write invalidates the edited
metarecord immediately. Memory is bounded by LRU eviction (reads keep an entry
recent; an evicted datum is simply re-fetched on the next read).

** Commands targeting an absent panel type

Commands registered by a panel type whose instance is not mounted yet in
the focused workspace are dispatched after lazily creating the hidden
instance. Commands registered with =reveal: true= (e.g. =metarecord:create=,
=log:rollback=) additionally switch a slot to the owning panel type first:
the other slot when it shows the same workspace, otherwise the focused
slot. Limitation: a panel type's commands only exist in the registry once
the panel has been instantiated at least once in the session.

** Path display

Path resolution runs **daemon-side**, in one round-trip whatever the depth:
=POST /repos/:repo/query/fields/resolve-tree= ={ query, field? }= walks each
matching metarecord's TreeRef positions through the in-memory tree cache and
returns repo-root-relative paths (the explicit-UUID case uses a =uuid_in= query).
It is general over any TreeRef field (=mfr_path= by default) and multi-map
(an array per metarecord; stale positions skipped). A single metarecord can also
use the direct =GET …/metarecords/:uuid/fields/:name/resolve-tree=. The CLI's =mf path= and the
GUI shim's =metafolder.daemon.resolvePath= / =resolveTreeRef= /
=metarecordPaths= are thin wrappers over it (the shim keeps a per-repo memo
cache; =invalidatePath= drops one metarecord after a move); =repoRoot= gives the
absolute root. Panels that render many rows resolve a whole page in batch when
it is fetched (see the =metarecord-list= data/view split) rather than per cell —
rendering itself makes no daemon calls.

** GUI API port persistence

The GUI binds the =gui-port= configured in =~/.config/metafolder/gui/config.toml=
(default 7524; overridable with =--gui-port=). The CLI reads that same
=config.toml=, so the port is a fixed, shared setting — there is no runtime
=gui.port= discovery file to write or clean up.

** Concurrent script sessions

Only one =POST /gui/input= / =POST /gui/prompt= wait at a time; concurrent
calls fail immediately with =409= (no queueing).

** Panel type update strategy

Resolved by the git-backed config repository: =metafolder-sync-config= merges
the shipped =default= branch into the user's =main= branch (see
[[file:spec-config.org][spec-config]]).

** Bench harness

A JS profiling harness for finding GUI slowness, built on the existing single
command path rather than a parallel control API.

Mechanism:
- Panels wrap their phases with =metafolder.bench.measure(name, fn)= and the
  shim auto-instruments every daemon round-trip. Built-in labels:
  =mf:list:enrich= / =mf:list:render=, =mf:detail:load= / =mf:detail:render=,
  =mf:fm:load= / =mf:fm:render=, and =mf:daemon <method> <path>=.
- Each measure is both a =performance.measure= (visible in the WebKit inspector
  Timeline) and a report to the shell, appended to an in-memory buffer.
- =GET /gui/bench= reads the buffer, =POST /gui/bench/clear= empties it.
- A scenario is driven externally with =POST /gui/command= (the same
  =dispatch()= the command input uses), so internal and external invocation
  stay identical. =scripts/bench-gui.sh= runs the built-in scenarios (panel
  opens, list+detail navigation, file-manager navigation, paging), clearing the
  buffer before each and printing the per-phase timings after.

Because a panel's render happens asynchronously inside its iframe, the
=POST /gui/command= call alone cannot time it — the harness's per-phase reports,
not the command's resolution, are the source of truth for render cost. The
=metarecord-list:page-next= command loads the next page on demand (the
scriptable equivalent of scrolling to the bottom) so paging can be driven
reproducibly.

* Open questions

** Custom panel type trust model (resolved: trusted)

The =metafolder.fs= API gives panel types unrestricted filesystem access and
=commands.invoke('!…')= runs arbitrary shell commands, so the iframe was never
a security boundary against panel code — the boundary is Tauri's capability
layer (GUI ↔ OS). Panel types are therefore treated as **trusted code**, and
the GUI runs them in the shell's own JS realm (Shadow DOM for CSS isolation,
no realm/process isolation; see [[*=metafolder= API delivery mechanism]]),
which is what makes a single shared in-realm data cache possible. A throw in a
panel's =mount= is contained by an error boundary, but a buggy panel can still
corrupt shared state. To be revisited only if a panel-type *distribution*
mechanism (untrusted third-party panels) is ever added — that would need real
sandboxing, not the iframe.

** Panel type selector UX

The current design uses a dropdown in the panel header. This may not be
practical when many panel types are installed. Alternatives: a dedicated
keybinding that opens an autocomplete picker (command-input style), or a
floating palette. To be revisited once the panel type ecosystem grows.

** Media codec support

The =file= panel type uses HTML5 =<audio>= / =<video>=; supported formats
depend on the system WebView, which differs per platform (WebKit2GTK on
Linux, WKWebView on macOS, WebView2 on Windows — see
[[file:spec-platform.org][spec-platform.org]]). On Linux, H.264 may require
GStreamer plugin packages (e.g. =gst-plugins-bad=, =gst-libav=).

On Linux, a missing audio sink does not fail gracefully: WebKitGTK
crashes its web process while building the playback pipeline, which
freezes the entire GUI (the shell and all panel iframes share that one
process). Two defenses are implemented:

- =GET /__media-support= on the GUI server reports
  ={"audio": bool, "video": bool, "missing": [elements]}=, probed once
  per process with =gst-inspect-1.0 --exists= on =autoaudiosink= /
  =autovideosink= (falling back to a plugin-file scan for
  =libgstautodetect.so= when the tool is unavailable; undeterminable
  counts as missing). The =file= panel checks it before creating a media
  element and otherwise shows a message naming the missing elements
  (provided by =gst-plugins-good=).
- The GUI connects WebKitGTK's =web-process-terminated= signal and
  reloads the shell when the web process dies for any reason — Rust owns
  all canonical state, so the session survives. A second termination
  within 10 seconds of the previous one is assumed to be a reload-crash
  loop and is not followed by another reload.
