#+TITLE: Metafolder — GUI Specification
#+OPTIONS: toc:2 num:t

* Overview

The GUI is a graphical interface to the Metafolder daemon. It exposes the full
CLI surface in visual form: repository management (init/load/list), entry
CRUD, query execution, reconcile, and event log inspection. It communicates
with an already-running daemon over HTTP and does not manage the daemon's
lifecycle.

Four cross-cutting design principles govern the GUI:
- *Keyboard-first*: every command is reachable from the keyboard; all
  keybindings are configurable.
- *Command-driven*: a persistent status bar shows notifications; a command
  input accepts named commands and shell invocations.
- *Scriptable*: the GUI exposes its own local HTTP API so that external scripts
  can drive panels and wait for user input.
- *Themeable*: appearance is controlled by a CSS file loaded at startup.

The GUI is built with *Tauri* (v2): a Rust backend communicating with a
system WebView frontend. The shell frontend is *Svelte 5* (Vite build);
panel types are plain HTML/JS with no build step. The Rust backend owns all
canonical state (workspaces, layout, variables, keybindings, command
registry); the shell is a thin reflection updated through Tauri events, so
the scripting HTTP API and the window always agree.

* Concepts

** Workspace

A self-contained state container. Each workspace holds: the state of every
panel type that has been loaded in it during the session (scroll position,
query draft, display mode, etc.), the workspace variables, and the status bar
history. Panel types retain their state for the lifetime of the session —
switching to a different panel type and back restores the previous state.
Workspace state is not persisted across sessions; the GUI starts fresh each
time.

The =active_repo= of a workspace is set when the workspace is created and
cannot be changed. To work with a different repository, a new workspace is
created (see [[*Tab bar]]). This prevents =selected_entry= and other workspace
variables from becoming invalid due to a repo switch.

** Panel slot

One of two fixed content areas in the window (left and right). Each slot is
assigned at most one workspace. An unassigned slot remains visible but shows a
centred "No workspace selected" message; the user can assign a workspace by
clicking a tab or using =tab:new=. A workspace can be assigned to zero, one,
or both slots simultaneously.

When both slots display the same workspace, each slot independently selects
which panel type to show. This enables, for example, =entry-list= on the left
and =file= on the right within one workspace, sharing its variables
(=selected_entry=, etc.).

The two slots are separated by a draggable divider; the width ratio is part of
the window state (not per-workspace). The focused slot has a visible indicator
(border highlight — theme-controlled) and receives keyboard input.

** Panel type

The content/functionality kind displayed inside a panel. Each panel type is
implemented as a directory containing an =index.html= entry point; built-in
types ship as default examples in the user's config directory so they can be
inspected and modified. The panel type name is the directory name.

Built-in panel types:

| Name             | Description                                                  |
|------------------+--------------------------------------------------------------|
| =entry-list=     | Metadata entries filtered by an embedded query; primary selection source |
| =entry-detail=   | All fields of the selected entry; follows =selected_entry=   |
| =file=           | Preview of the file at the active path from =selected_paths=        |
| =file-manager=   | Filesystem tree navigator; browse and add files/dirs to the DB |
| =workspace-info= | Current workspace variables — useful for debugging and scripting |
| =message=        | Persistent log of status bar messages, shell output, and reconcile results |
| =log=            | Event log (revisions and operations)                         |
| =repos=          | Repository list, init, load                                  |

Users can add custom panel types by creating a new directory under
=~/.config/metafolder-gui/panel-types/<name>/=. Custom types appear alongside
built-in types in the panel type selector.

Each panel type retains its full state for the lifetime of the session, even
when not displayed. Switching to another panel type and back restores the
previous state unchanged.

** Display mode

The visual representation used within a panel type that supports several
layouts. Stored as part of the panel type's state in the workspace.

The =entry-list= panel type supports at minimum:
- =table= — tabular list with configurable columns.
- =grid= — thumbnail grid, useful for image collections.

For large result sets, =entry-list= uses the daemon's cursor pagination
(=limit= / =cursor= / =sort= on =POST /query=, see the Pagination section of
[[file:spec-data-model.org][spec-data-model.org]]), fetching further pages as
the user scrolls.

** Workspace variables

A reactive, per-workspace key-value store shared across all panel types
loaded in the workspace. Panel types read and write these variables to communicate without
being directly coupled to each other.

Standard variables:

| Key                | Type                          | Description                        |
|--------------------+-------------------------------+------------------------------------|
| =active_repo=      | =string \vert null=           | UUID of the active repository      |
| =selected_paths=   | =string[]=                    | Absolute paths of the highlighted filesystem entry (usually one; multiple if the entry has several =mfr_path= values) |
| =selected_entry=   | =\{uuid, repo\} \vert null=   | Metadata entry for the highlighted path (null if untracked) |
| =selected_entries= | =string[]=                    | UUIDs of all selected entries      |

Panel types may define their own variables using namespaced keys (e.g.
=my-panel:current-node=).

** Command

A named, invocable operation with an optional parameter string. Commands are
registered by panel types (built-in or custom) and are available in the
command input with autocomplete. Every user-visible behaviour is a command.
Keybindings map key combos to command invocation strings (command name plus
optional parameters).

Each command has a *scope* that determines when it is active (see [[*Keybinding]]).
Built-in commands default to =global=; commands registered by a panel type
default to =local=.

** Keybinding

A mapping from a key combo (e.g. =ctrl+k=, =alt+x=, =g g= for a two-key
sequence) to a command invocation string (e.g. =entry-list:set-mode grid=).
Keybindings are loaded from a configuration file and can be edited at runtime.
No keybinding is hard-coded; the defaults ship in the default config file.

The user's =keybindings.toml= contains only their customisations and
overrides; it does not need to list every binding. The GUI engine supplies all
defaults separately and merges them at startup, with the user file taking
precedence.

Each keybinding has two independent scope dimensions:

- =when= (optional) — restricts the binding to a specific panel type. Absent
  means the binding is *global* (fires regardless of which panel type is
  focused). A panel type name (e.g. =entry-list=) makes it *local* (fires
  only when that panel type is focused).
- =text-input= (boolean, default =false=) — whether the binding also fires
  when a text input is focused (command input or field editor). =false= means
  the binding is suppressed while the user is typing; =true= means it fires
  regardless.

The two dimensions combine independently:

| =when=             | =text-input= | Fires when                                          |
|--------------------+--------------+-----------------------------------------------------|
| absent             | =false=      | Any panel focused; no text input active (default)   |
| absent             | =true=       | Any panel focused; even inside a text input         |
| =<panel-type>=     | =false=      | Named panel focused; no text input active           |
| =<panel-type>=     | =true=       | Named panel focused; even inside a text input       |

When the same key matches several bindings, a more specific binding wins: local
over global, =text-input = false= over =text-input = true= (the less permissive
one takes precedence when both would fire).

Config file examples:

#+BEGIN_SRC toml
"escape"  = { command = "editing:unfocus",         text-input = true              }
"ctrl+a"  = { command = "editing:goto-line-start", text-input = true              }
"j"       = { command = "entry-list:next",         when = "entry-list"            }
"ctrl+e"  = { command = "my-panel:edit",           when = "my-panel", text-input = true }
"ctrl+t"  = { command = "tab:new"                                                  }
#+END_SRC

Panel types may suggest default keybindings via =metafolder.addKeybinding=;
these are applied only when the user has no conflicting binding configured.

** Status bar

A read-only line at the very bottom of the window, always visible and never
focused. Displays status messages, error notifications, the name of the last
executed command (dim text), and a spinner during long operations. Content is
*per-workspace*. When both slots display the same workspace, one status bar is
shown; when the two slots display different workspaces, two status bars are
shown (one per slot). Messages auto-clear after a configurable timeout
(default 5 s).

** Command input

A single-line text input permanently visible just above the status bar. A
configurable keybinding (default suggestion: =:=) gives it keyboard focus;
=Escape= removes the focus (the draft is kept). There is one command input;
it targets the focused slot's workspace. Content (text and cursor position)
is per-workspace: switching the focused slot to a different workspace
restores the command input state of that workspace, including any partial
command being typed.

Accepts:
- A command name (with inline completion) optionally followed by
  space-separated parameters. =Tab= completes shell-style: a single match
  is completed entirely, several matches complete to their longest common
  prefix. =Up=/=Down= navigate the suggestion list; =Enter= on a
  highlighted suggestion writes it into the input (without executing);
  =Escape= leaves the list navigation (typing also does), then a second
  =Escape= unfocuses the input.
- A shell command prefixed with =!= — executed as a subprocess; stdout/stderr
  goes to the =message= panel type and to the terminal from which the GUI was
  launched.

** Field editor

An inline editor for a single field value, adapting its widget to the type:

| Value type    | Widget                                         |
|---------------+------------------------------------------------|
| =String=      | Text input                                     |
| =Int=         | Integer input                                  |
| =Float=       | Float input                                    |
| =Bool=        | Checkbox / toggle                              |
| =DateTime=    | Date-time picker (ISO-8601 output)             |
| =Nothing=     | Read-only label "absent"                       |
| =Ref=         | UUID text input (lookup by path available)     |
| =TreeRef=     | Parent UUID + name, or path string             |
| =RefBase=     | UUID text input                                |
| =ExternalRef= | Repo UUID + entry UUID                         |

** Script session

An external process communicating with the GUI over the GUI API. A script
session can set panel types, display messages in the status bar, activate the
command input with a prompt, and block waiting for user key input. Only one
script session may hold the input-wait lock at a time.

** Repo indicator

A read-only label in the slot's header bar showing the name of the active
repository for that slot's workspace. It is purely informational; the user
cannot change =active_repo= through it.

To open a repository, the user uses =repos:open= (e.g. via a menu or
keybinding), which switches the focused slot to the =repos= panel type where
the desired repo can be selected or a new one initialised. Selecting a repo
in the =repos= panel type opens a new workspace (tab) with that repo as
=active_repo=, assigned to the focused slot. If the focused slot's workspace
already has =active_repo = null= (e.g. at startup), the selection sets it
in place instead.

* GUI

** Connection to the daemon

The GUI connects to a daemon URL configured by the user (default
=http://127.0.0.1:7523=), editable via the =daemon:set-url= command.

On startup (and whenever the URL changes) the GUI calls =GET /health=. If the
daemon is unreachable or returns a non-200 response, a prominent error banner
is shown and all daemon-dependent commands are disabled. The GUI itself
(panels, status bar, command input, keybindings, panel type system, and the
GUI API) remains functional.

No authentication: the daemon is local-only and requires none.

** Tab bar

A row of tabs at the top of the window. Each tab represents one workspace.
Tabs have a short user-editable name (default: ="Workspace 1"=,
="Workspace 2"=, etc.). Tab state is not persisted across sessions.

Color indicators show the current slot assignments:
- *Green*: workspace displayed in the focused slot.
- *Red*: workspace displayed in the non-focused slot.
- *Green + red*: workspace displayed in both slots simultaneously.
- No color: workspace not currently visible in either slot.

*Left-click* on a tab: assigns that workspace to the focused slot (the slot
keeps focus). *Right-click*: assigns that workspace to the non-focused slot.

| Command           | Description                                               |
|-------------------+-----------------------------------------------------------|
| =tab:new=         | Create a new workspace and assign it to the focused slot  |
| =tab:close=       | Close the focused slot's workspace (removes its tab; slot becomes unassigned) |
| =tab:rename=      | Rename the focused slot's workspace                       |
| =tab:next=        | Assign the next workspace to the focused slot (wraps)     |
| =tab:prev=        | Assign the previous workspace to the focused slot         |
| =tab:goto-N=      | Assign workspace N to the focused slot                    |

Default keybinding suggestions: =alt+t= for =tab:new=, =alt+w= for
=tab:close=, =alt+1= … =alt+9= for =tab:goto-N=.

** Layout and panels

The window area (below the tab bar) is divided into:
- *Two panel slots* (left and right), each assigned one workspace (see
  [[*Panel slot]]). At startup only the left slot is visible.
- *Command input*: a single line just above the status bar(s), always
  visible.
- *Status bar*: one or two read-only lines at the very bottom (see
  [[*Status bar]]).

=panel:split= shows the right slot. If no workspace was previously assigned to
it, a new workspace is created (inheriting =active_repo= from the focused
workspace). =panel:close= hides the non-focused slot; the workspace remains in
the tab bar. The focused slot has a visible indicator (border highlight —
theme-controlled); focus switches via =panel:focus-next=.

Each slot has a thin header bar showing the current panel type name and the
repo indicator. The header contains a dropdown that lists all available panel
types (built-in and custom); selecting one switches the displayed panel type
for that slot. The same switch is available via =panel:set-type <name>= and
configurable keybindings.

** Status bar

Always-visible read-only area at the bottom. Shows:
- Status messages after commands (e.g. ="Entry deleted."=).
- Error messages from failed operations.
- The last executed command name (dim text).
- A spinner during long-running operations (reconcile, shell commands).

Content is per-workspace. When both slots display the same workspace, one
status bar is shown. When the two slots display different workspaces, two
status bars are shown (one per slot, each reflecting its own workspace).

** Command input

Always visible: a prompt character and a text field above the status
bar(s). A configurable keybinding (default: =:=) gives it keyboard focus;
=Escape= unfocuses it, keeping the draft. The autocomplete suggestions are
shown only while the input is focused. There is one command input; it
targets the focused slot's workspace. Content is per-workspace: switching
the focused slot to a different workspace restores that workspace's command
input state.

Behaviours:
- Typing a command name + =Enter= → invokes the command (=Tab= completes
  shell-style; =Up=/=Down= browse the suggestions, see [[*Command input]]).
- Parameters follow the command name, separated by spaces.
- =!shell-command= + =Enter= → runs in background; output goes to the
  =message= panel type and the launching terminal.
- When a script calls =POST /gui/prompt=, the command input grabs the focus
  and shows the script-supplied prompt text.

** Keyboard navigation

Every command is reachable from the keyboard. No command requires the mouse.

Within =entry-list=:
- =Up= / =Down= move the selection (updates =selected_entry=).
- =Enter= or =Right= switches the other slot to =entry-detail= (or =file= if
  =selected_paths= is non-empty), opening the slot if hidden. See
  [[*Cross-panel selection]].
- =Space= toggles selection for multi-select (updates =selected_entries=).

Within =entry-detail=:
- =Tab= / =Shift-Tab= move focus between field rows.
- =Enter= on a field starts inline editing; =Enter= again confirms, =Escape=
  cancels.
- =d= (configurable) deletes the focused field row (with confirmation).

Keybindings are stored in =~/.config/metafolder-gui/keybindings.toml=
(see [[*Keybinding]] for the =when= and =text-input= scope dimensions).
The =config:open= command opens a settings view where they can be edited
interactively; changes take effect immediately.

** Style and theming

The GUI loads =~/.config/metafolder-gui/style.css= at startup and applies it
globally to the WebView. The GUI ships a default stylesheet the user copies
and modifies. The =config:open= settings view shows the active CSS path and
offers a reload button; the GUI also auto-reloads on file change via inotify.

** Panel type system

*** Directory layout

Panel types live under =~/.config/metafolder-gui/panel-types/=:

#+BEGIN_SRC
~/.config/metafolder-gui/panel-types/
  entry-list/
    index.html        ← entry point
  entry-detail/
    index.html
  file/
    index.html
  message/
    index.html
  log/
    index.html
  repos/
    index.html
  my-custom-panel/    ← user-defined
    index.html
#+END_SRC

Built-in types ship as pre-populated examples. The user can edit them freely;
the GUI does not overwrite them on update: installation only writes missing
files under =panel-types/=, while =panel-types-defaults/= (a sibling
directory) is rewritten on every startup with the current shipped versions,
so the user can diff their edits against the defaults after an upgrade.

Panel types are served to the WebView by the GUI's own HTTP server (the
GUI API port, default 7524): =GET /panel/<name>/<file>=. HTML documents get
the =metafolder= shim and a =<link>= to the user stylesheet injected at
serve time. Raw local files for the =file= panel type are served by
=GET /fsraw?path=...= with HTTP Range support (media seeking).

*** The =metafolder= API

The GUI injects a =window.metafolder= object into every panel type's JS
context. It provides three namespaces:

**** =metafolder.daemon=

Typed async wrappers around the daemon HTTP API. Examples:

#+BEGIN_SRC js
await metafolder.daemon.query(repo, query, select)
await metafolder.daemon.getEntry(repo, uuid)
await metafolder.daemon.setField(repo, uuid, name, value)
await metafolder.daemon.reconcile(repo)
// ... mirrors the full daemon REST API
#+END_SRC

**** =metafolder.workspace=

Reactive per-workspace key-value store.

#+BEGIN_SRC js
metafolder.workspace.set('selected_entry', { uuid, repo })
const entry = metafolder.workspace.get('selected_entry')
metafolder.workspace.onChange('selected_entry', ({ uuid, repo }) => { ... })
#+END_SRC

Standard keys: =active_repo=, =selected_paths=, =selected_entry=,
=selected_entries=. Panel types may use namespaced custom keys (e.g.
=my-panel:state=).

**** =metafolder.commands= and =metafolder.addKeybinding=

#+BEGIN_SRC js
// scope: 'local' (default) | 'global'
// text-input controls whether the command's default bindings fire in text inputs;
// it is also used as the default for addKeybinding calls for this command.
metafolder.commands.register('my-panel:change-mode', {
  label: 'My Panel: Change display mode',
  scope: 'local',
  handler: ({ mode }) => { ... }
})

metafolder.commands.register('my-panel:global-action', {
  label: 'My Panel: Global action',
  scope: 'global',
  handler: () => { ... }
})

// Suggest default keybindings (applied only if no conflict in user config).
// Optional third argument: { when, textInput } overrides.
metafolder.addKeybinding('my-panel:change-mode list', 'ctrl+l')
metafolder.addKeybinding('my-panel:change-mode grid', 'ctrl+g')
metafolder.addKeybinding('my-panel:edit-text', 'ctrl+e', { textInput: true })
metafolder.addKeybinding('editing:unfocus',    'escape', { textInput: true })
#+END_SRC

Registered commands appear in the command input autocomplete alongside
built-in commands. The autocomplete shows global commands always, plus local
commands for the currently focused panel type.

**** =metafolder.fs=

Direct filesystem access via the GUI backend (Tauri/Rust). Available to all
panel types; not routed through the daemon.

#+BEGIN_SRC js
// List directory contents
const entries = await metafolder.fs.readDir('/path/to/dir')
// entries: [{name: "foo.mp3", path: "/path/to/dir/foo.mp3", is_dir: false}, ...]

// Stat a single path
const info = await metafolder.fs.stat('/path/to/file')
// info: {path, is_dir, size, mtime}
#+END_SRC

**** =metafolder.statusBar=

#+BEGIN_SRC js
metafolder.statusBar.message('Sync complete.', 5000)  // timeout_ms or null
#+END_SRC

** Repository management

*** Repo list

Calls =GET /repos= at startup and on explicit refresh. Results populate the
Repo indicator and the =repos= panel type.

*** Init repo

=repos:init= is a local command of the =repos= panel type (scope:
=when = "repos"=). It transforms the =repos= panel to show an inline form
with a single =root= input (the daemon API also accepts an external
=metafolder= location, but the GUI does not expose it). Submitting calls
=POST /repos/init=. On success a new workspace is created with the new repo
as =active_repo= and assigned to the focused slot. On =409= an error
message appears in the status bar and the form remains open.

*** Load repo

=repos:load= is a local command of the =repos= panel type (scope:
=when = "repos"=). It transforms the =repos= panel to show an inline form
with a single =root= input. Submitting calls =POST /repos/load=. On success
a new workspace is created with the loaded repo as =active_repo= and
assigned to the focused slot.

The intended flow is: =repos:open= (global) → navigate to the =repos= panel
→ select a loaded repo or trigger =repos:init= / =repos:load= from there.

** Entry management

*** =entry-list= panel type

Displays entries for the active repo (=active_repo= workspace variable)
filtered by an embedded query. The query editor is shown/hidden by the
=entry-list:edit-query= command; it uses the same DSL as the CLI. An empty
query matches all entries.

Default display mode: =table= with columns =mfr_path= (resolved path),
=mfr_type=, =version=. Columns are configurable. The =entry-list:set-mode=
command switches between display modes.

Supports sorting by column, multi-selection (=selected_entries=), and full
keyboard navigation.

When the user moves the selection, =entry-list= writes the highlighted entry
to =workspace.selected_entry= and sets =workspace.selected_paths= to the list
of resolved absolute paths from all =mfr_path= field rows of that entry
(empty list if the entry has no valid =mfr_path=). Other panels update
reactively (see [[*Cross-panel selection]]).

*** =entry-detail= panel type

When =selected_entry= is =null=, displays a centred "No entry selected"
message and the /New entry/ button.

Otherwise calls =GET .../metadata/:uuid= and displays all fields. Each field
row shows name, value, an edit button, and a delete button.

Keyboard: =Tab= moves between rows, =Enter= edits, =d= deletes (with
confirmation).

At the bottom of the panel:
- /New entry/ button (or =entry:create= command) → clears the form and
  presents an empty field list for creating a new entry. Submitting calls
  =POST .../metadata=; on success the new entry is set as =selected_entry=
  and the list in the linked =entry-list= panel (if any) refreshes.
- /Add field/ → =POST .../metadata/:uuid/fields=
- /Set field/ → =PATCH .../metadata/:uuid=
- /Delete entry/ → confirmation → =DELETE .../metadata/:uuid=
- /Watch and reconcile/ → visible when the entry's =mf_watch= field is
  =false= or absent. Sets =mf_watch = true= (=PATCH=), then calls
  =POST .../metadata/:uuid/reconcile=. A spinner appears in the status bar
  during the operation; the result is appended to the =message= panel type.
  Useful after adding an entry via =file-manager:add= to activate tracking and
  populate the subtree.

Protected =mfr_*= fields are read-only by default; an advanced toggle exposes
a =force= checkbox.

*** =file= panel type

Displays a preview of the file at the currently active path from
=selected_paths=. When =selected_paths= is empty, shows a "no file selected"
message. When =selected_paths= has more than one element (entry accessible at
multiple locations), a dropdown in the panel header lets the user choose which
path to preview. Uses the HTML5 =<img>=, =<audio>=, or =<video>= element
depending on the file type; falls back to a plain-text view for text files;
shows "no preview available" for unsupported formats. Works with both tracked
and untracked files.

Before creating an =<audio>= / =<video>= element, the panel queries
=GET /__media-support= on the GUI server and, when the required GStreamer
elements are missing, shows an explanatory message instead of attempting
playback (see [[*Media codec support]]).

*** =file-manager= panel type

A filesystem tree navigator. The GUI backend (Tauri/Rust) reads the disk
directly; the daemon is not involved in directory listing.

*Display*: a tree/list of files and directories with a visual distinction
between tracked entries (have =mfr_path= metadata in the active repo) and
untracked ones. When =active_repo= is =null=, all entries are shown as
untracked (no distinction is possible).

*Root behaviour*: a checkbox (default: checked) constrains navigation to the
=active_repo= root. When unchecked, the user can navigate anywhere on the
disk. Adding files to the DB is only available for paths inside the
=active_repo= root; attempting it outside returns an error in the status bar.
When =active_repo= is =null=, the checkbox is disabled and =file-manager:add=
/ =file-manager:goto-root= return an error.

*Selection*: moving the cursor sets =workspace.selected_paths= to a
single-element list containing the highlighted path. If the path has an
existing metadata entry in the active repo, =workspace.selected_entry= is also
updated; otherwise it is set to =null=.

*Keyboard*: =Up= / =Down= navigate; =Enter= expands a directory or confirms
selection for a file; =Backspace= goes up one level.

| Command                   | Description                                              |
|---------------------------+----------------------------------------------------------|
| =file-manager:add=        | Add the selected path to the DB via =POST .../track= (creates entry with =mf_watch = false=; parent entries created automatically). Error if already tracked (=409=) or outside =active_repo= root. |
| =file-manager:goto-root=  | Jump to the =active_repo= root directory                 |
| =file-manager:toggle-root= | Toggle the "constrain to root" checkbox                 |

*** =workspace-info= panel type

Displays all workspace variables for the current workspace: =active_repo=,
=selected_paths=, =selected_entry=, =selected_entries=, and any custom
namespaced variables. Values are shown in JSON format and update reactively
as they change. Read-only; useful for debugging panel type communication and
for monitoring GUI state from scripts.

*** Cross-panel selection

Selecting an entry in =entry-list= or =file-manager= updates the workspace
variables =selected_paths= and/or =selected_entry= (see the respective panel
type sections for details). Any slot in the same workspace displaying
=entry-detail=, =file=, or =workspace-info= subscribes reactively and updates
automatically. Slots showing different workspaces are unaffected.

*Double-click* (or =Enter= / =Right= keybinding) in =entry-list=: if the
other slot shows the same workspace and is displaying a different panel type,
it is switched to =entry-detail= (or =file= if the entry has a valid
=mfr_path=). If the other slot is hidden, it is opened (showing the same
workspace).

*Edit guard*: before =entry-detail= replaces its content with a new entry, it
checks whether any field is being edited. If so, an alert is shown:
="Unsaved changes — discard and switch entry? [Confirm / Cancel]"=. The switch
proceeds only if the user confirms.

*** Batch set

When =selected_entries= is non-empty, =entry:batch-set= opens a dialog for
field name and value and sets it on every selected entry. (v1
implementation note: the query IR has no UUID predicate, so this is one
=PATCH .../metadata/:uuid= per selected entry rather than a single
=POST .../set=; to be revisited if a UUID predicate is added.)

** Message view

The =message= panel type is a scrollable, append-only log, *per-workspace*.
It receives:
- All messages posted to the active workspace's status bar (with timestamp).
- stdout/stderr of shell commands run via =!= in this workspace.
- Reconcile progress and results triggered from this workspace.
- Any message posted by a script via =POST /gui/message= targeting this
  workspace (also shown in the status bar as a transient notification).

The log is cleared by the =message:clear= command. It is not persisted across
sessions.

** Reconcile

=reconcile:run= calls =POST .../reconcile=. A spinner appears in the status
bar. On completion:
- Status bar: ="Reconcile: N created, M updated, P unchanged."=
- Full output (per-file actions and candidate stale matches) is appended to the
  =message= panel type.

** Event log

*** Log view (=log= panel type)

Calls =GET .../log= and displays revisions in reverse chronological order.
Columns: =rev_id=, timestamp, operation count, =status=.

Selecting a revision (=Enter=) expands it to show individual operations.

*** Revision commands

- =log:rollback= → confirmation → =POST .../rollback=. Refreshes =entry-list=
  afterwards.
- =log:mark-checkpoint= [v2] → =PATCH .../log/revisions/:rev_id=.

*** Prune log

=log:prune= opens a dialog (target revision or "keep last N revisions") and
calls =POST .../log/prune=.

** Scripting / GUI API

The GUI exposes a local HTTP API on =127.0.0.1:7524= (configurable via
=--gui-port=). External scripts can drive the GUI: set panel types, display
messages, wait for user input.

All endpoints are available as long as the GUI is running, regardless of
daemon connectivity.

*** =GET /gui/workspaces=

Returns the list of all open workspaces.

#+BEGIN_SRC json
[
  {"id": "ws-1", "name": "Workspace 1", "active_repo": "<repo_uuid>"},
  {"id": "ws-2", "name": "Workspace 2", "active_repo": null}
]
#+END_SRC

*** =POST /gui/workspaces=

Creates a new workspace. Returns its id.

Request body (all fields optional):
#+BEGIN_SRC json
{"active_repo": "<repo_uuid>"}
#+END_SRC

If =active_repo= is absent, the workspace uses the daemon's first loaded
repository (from =GET /repos=), or =null= if no repositories are loaded.

Response: ={"id": "ws-3"}=

*** =DELETE /gui/workspaces/:id=

Closes a workspace. Any slot that had this workspace assigned becomes
unassigned (shows "No workspace selected"). Returns =204 No Content=.

*** =GET /gui/layout=

Returns the current slot assignments.

#+BEGIN_SRC json
{"left": "ws-1", "right": "ws-2"}
#+END_SRC

A slot value of =null= means the slot is hidden (no workspace assigned).

*** =PUT /gui/layout=

Assigns workspaces to slots. Only the keys present in the request body are
updated; omitted keys are unchanged.

#+BEGIN_SRC json
{"left": "ws-1", "right": "ws-2"}
#+END_SRC

Setting a slot to =null= hides it. Returns =200 OK=, body ={}=.

*** =PUT /gui/panels/:slot/view=

Sets the panel type displayed in slot =:slot= (=left= or =right=). If the slot
is hidden, it is shown first.

Request body examples:

#+BEGIN_SRC json
{"type": "entry-list"}
{"type": "entry-detail"}
{"type": "file", "path": "/absolute/path/to/image.jpg"}
{"type": "message"}
{"type": "log"}
{"type": "my-custom-panel", "state": { ... }}
#+END_SRC

When =type= is =file= and =path= is provided, the workspace variable
=selected_paths= is set to =[path]= and =selected_entry= is set to the
matching metadata entry (or =null= if the file is untracked). Omitting =path=
leaves =selected_paths= unchanged and the =file= panel type displays whatever
path was already active.

Response: =200 OK=, body ={}=.

*** =GET /gui/panels/:slot/view=

Returns the current panel type descriptor for slot =:slot=, including a
=status= field (="loading"= / ="ready"=).

*** =POST /gui/message=

Posts a message to a workspace's status bar (transient) and appends it to
that workspace's =message= panel type log. An optional =?workspace_id=<id>=
query parameter targets a specific workspace; defaults to the focused slot's
workspace.

Request body:
#+BEGIN_SRC json
{"text": "← left  → right  Esc to stop", "timeout_ms": null}
#+END_SRC

=timeout_ms=: null → status bar message persists until the next message;
integer → auto-clears after that many milliseconds.

*** =POST /gui/input=

Registers an input wait and blocks until the =answer:send= command is
executed (by any means: a temporary binding installed by this request, a
permanent user binding, or a manual invocation via the command input).
No keystrokes are suppressed; all normal shortcuts continue to work.

The optional =keys= array installs temporary global keybindings for the
duration of the wait: each listed key is bound to =answer:send <key>=
(scope: global, =text-input = false=). These bindings are removed when the
request resolves or times out. If =keys= is absent or empty, no temporary
bindings are installed; the user can still resolve the wait by typing
=answer:send <value>= manually in the command input.

Only one =POST /gui/input= at a time; concurrent calls return =409=.

Request body:
#+BEGIN_SRC json
{"keys": ["left", "right", "escape"], "timeout_ms": null}
#+END_SRC

Responses: ={"event": "answer", "value": "left"}= / ={"event": "timeout"}= /
={"event": "closed"}=.

*** =POST /gui/prompt=

Activates the command input with a custom prompt. Blocks until =Enter= or
=Escape=. Mutually exclusive with =POST /gui/input= (same lock).

Request body: ={"prompt": "Enter a rating (1-5): ", "timeout_ms": null}=

Responses: ={"event": "confirm", "text": "4"}= / ={"event": "cancel"}=.

*** =GET /gui/status=

Returns the current GUI state.

#+BEGIN_SRC json
{
  "workspaces": [
    {"id": "ws-1", "name": "Workspace 1", "active_repo": "<repo_uuid>"},
    {"id": "ws-2", "name": "Workspace 2", "active_repo": null}
  ],
  "layout": {
    "left":  {"workspace_id": "ws-1", "panel_type": "entry-list", "focused": true},
    "right": {"workspace_id": "ws-2", "panel_type": "file",        "focused": false}
  },
  "daemon_connected": true,
  "input_wait_active": false
}
#+END_SRC

*** Example: image comparison script

#+BEGIN_SRC bash
#!/usr/bin/env bash
GUI="http://127.0.0.1:7524"

# Create two dedicated workspaces for the comparison
WS_LEFT=$(curl -s -X POST $GUI/gui/workspaces \
               -H "Content-Type: application/json" -d '{}' | jq -r .id)
WS_RIGHT=$(curl -s -X POST $GUI/gui/workspaces \
                -H "Content-Type: application/json" -d '{}' | jq -r .id)

# Assign them to the two panel slots
curl -s -X PUT $GUI/gui/layout \
     -H "Content-Type: application/json" \
     -d "{\"left\": \"$WS_LEFT\", \"right\": \"$WS_RIGHT\"}"

while true; do
  A=$(find /photos -name "*.jpg" | shuf -n1)
  B=$(find /photos -name "*.jpg" | shuf -n1)

  curl -s -X PUT $GUI/gui/panels/left/view \
       -H "Content-Type: application/json" \
       -d "{\"type\":\"file\",\"path\":\"$A\"}"
  curl -s -X PUT $GUI/gui/panels/right/view \
       -H "Content-Type: application/json" \
       -d "{\"type\":\"file\",\"path\":\"$B\"}"
  curl -s -X POST "$GUI/gui/message?workspace_id=$WS_LEFT" \
       -H "Content-Type: application/json" \
       -d '{"text":"← left  → right  Esc to stop"}'

  RESP=$(curl -s -X POST $GUI/gui/input \
              -H "Content-Type: application/json" \
              -d '{"keys":["left","right","escape"]}')
  KEY=$(echo "$RESP" | jq -r .value)

  [ "$KEY" = "escape" ] && break

  WINNER=$( [ "$KEY" = "left" ] && echo "$A" || echo "$B" )
  # record winner as metadata via daemon …
done

# Clean up the temporary workspaces
curl -s -X DELETE "$GUI/gui/workspaces/$WS_LEFT"
curl -s -X DELETE "$GUI/gui/workspaces/$WS_RIGHT"
#+END_SRC

* Resolved design decisions (v1)

** Frontend framework

Svelte 5 + Vite for the shell. Built-in panel types are plain HTML/JS (no
build step) so users can read and copy them as examples.

** =metafolder= API injection mechanism

Each panel type instance is an =<iframe>= (one per (workspace, panel type),
kept alive while hidden for state retention, never reparented). A shim
script injected at serve time provides =window.metafolder= and talks to the
shell over =postMessage=; the shell forwards to the Rust backend. Key events
never cross the iframe boundary, so the shim runs the same key matcher as
the shell against the compiled binding table and forwards resolved
invocations.

A corollary: the same panel type cannot be displayed in both slots for the
same workspace (one iframe cannot render twice); =panel:set-type= rejects
this with a status bar error.

** Daemon access

Panels and the shell reach the daemon through the Rust backend (a reqwest
proxy exposed as =metafolder.daemon.request=). The WebView cannot call the
daemon directly (CORS), and the daemon stays GUI-agnostic.

** Commands targeting an absent panel type

Commands registered by a panel type whose iframe is not instantiated yet in
the focused workspace are dispatched after lazily creating the hidden
iframe. Commands registered with =reveal: true= (e.g. =entry:create=,
=log:rollback=) additionally switch a slot to the owning panel type first:
the other slot when it shows the same workspace, otherwise the focused
slot. Limitation: a panel type's commands only exist in the registry once
the panel has been instantiated at least once in the session.

** Path display

Lazy resolution: =metafolder.daemon.resolvePath(repo, uuid)= /
=resolveTreeRef(repo, value)= walk the =mfr_path= parent chain through the
daemon API with a per-repo memo cache (=invalidatePath= drops one entry
after a move). Paths are repo-root-relative; =metafolder.daemon.repoRoot=
gives the absolute root.

** GUI API port persistence

The actually-bound port is written to =$XDG_RUNTIME_DIR/metafolder/gui.port=
(fallback: =~/.config/metafolder-gui/gui.port=) and removed on clean exit.

** Concurrent script sessions

Only one =POST /gui/input= / =POST /gui/prompt= wait at a time; concurrent
calls fail immediately with =409= (no queueing).

** Panel type update strategy

Resolved by the =panel-types-defaults/= mirror (see [[*Directory layout]]).

* Open questions

** Custom panel type trust model

The =metafolder.fs= API gives panel types unrestricted read access to the
filesystem (any directory, not just the repo root). Combined with
=metafolder.daemon= (full DB access) and =metafolder.workspace= (shared
variables), a malicious panel type installed from an untrusted source would
have broad access. No sandboxing is currently planned; users are expected to
only install panel types they trust. This is consistent with the trust
requirement for the daemon access itself. To be revisited if a panel type
distribution mechanism is ever added.

** Panel type selector UX

The current design uses a dropdown in the panel header. This may not be
practical when many panel types are installed. Alternatives: a dedicated
keybinding that opens an autocomplete picker (command-input style), or a
floating palette. To be revisited once the panel type ecosystem grows.

** Media codec support

The =file= panel type uses HTML5 =<audio>= / =<video>=; supported formats
depend on the system WebView, which differs per platform (WebKit2GTK on
Linux, WKWebView on macOS, WebView2 on Windows — see
[[file:spec-platform.org][spec-platform.org]]). On Linux, H.264 may require
GStreamer plugin packages (e.g. =gst-plugins-bad=, =gst-libav=).

On Linux, a missing audio sink does not fail gracefully: WebKitGTK
crashes its web process while building the playback pipeline, which
freezes the entire GUI (the shell and all panel iframes share that one
process). Two defenses are implemented:

- =GET /__media-support= on the GUI server reports
  ={"audio": bool, "video": bool, "missing": [elements]}=, probed once
  per process with =gst-inspect-1.0 --exists= on =autoaudiosink= /
  =autovideosink= (falling back to a plugin-file scan for
  =libgstautodetect.so= when the tool is unavailable; undeterminable
  counts as missing). The =file= panel checks it before creating a media
  element and otherwise shows a message naming the missing elements
  (provided by =gst-plugins-good=).
- The GUI connects WebKitGTK's =web-process-terminated= signal and
  reloads the shell when the web process dies for any reason — Rust owns
  all canonical state, so the session survives. A second termination
  within 10 seconds of the previous one is assumed to be a reload-crash
  loop and is not followed by another reload.
