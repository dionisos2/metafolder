#+TITLE: Metafolder — User Schema Specification
#+OPTIONS: toc:2 num:t

* Overview

The user schema system lets users declare /entry types/ and attach
/constraints/ to them: which fields an entry of a given type may carry, with
which value types and cardinalities. Constraints can also apply globally (to
every entry in the repository), typically to fix the value type of a field
name database-wide.

The schema is described in a JSON file loaded by the daemon; its location is
recorded in the repository's =config.json=. The database itself stores no
constraint: the only schema-related data in the database is the reserved
=mf_schema= field, which names the type(s) of an entry. The daemon validates
user writes against the loaded schema and rejects violating writes with
=400=.

* Concepts

** Entry types

An entry declares its type(s) with the reserved field =mf_schema=
(=String=). Because fields are multi-maps, an entry may carry several
=mf_schema= rows and thus several types; the constraints of all its types
apply cumulatively.

#+BEGIN_SRC json
{"name": "mf_schema", "value": {"type": "string", "value": "film"}}
#+END_SRC

=mf_schema= values are free-form strings. A value that names a type declared
in the schema file activates that type's constraints; a value with no
declaration in the schema file carries no constraints (it can still be used
as a plain label and queried normally). Entries without any =mf_schema= field
are /untyped/; only global constraints apply to them.

** Schema file

The schema is a single JSON document, external to the database. It is a list
of /constraint groups/; each group declares its targets (one or more type
names, or ="*"= for every entry) and the constraints shared by those targets:

#+BEGIN_SRC json
{
  "version": 1,
  "groups": [
    {"targets": "*",
     "constraints": [
       {"field": "rating", "type": "int"}
     ]},
    {"targets": ["film"],
     "constraints": [
       {"field": "rating", "type": "int", "min": 0, "max": 1},
       {"field": "director", "type": "ref"}
     ]},
    {"targets": ["person", "film"],
     "constraints": [
       {"field": "name", "type": "string", "min": 1, "max": 1}
     ]}
  ]
}
#+END_SRC

Top-level keys:

| Key       | Type | Description                                |
|-----------+------+--------------------------------------------|
| =version= | int  | Schema file format version (currently =1=) |
| =groups=  | list | Constraint groups                          |

Each group object:

| Key           | Type                  | Description                                                  |
|---------------+-----------------------+--------------------------------------------------------------|
| =targets=     | ="*"= or list<string> | Type names the group applies to; ="*"= = every entry (global) |
| =constraints= | list                  | Constraints applying to each target                          |

Each constraint object:

| Key           | Type   | Default   | Description                                     |
|---------------+--------+-----------+-------------------------------------------------|
| =field=       | string | required  | Name of the constrained field                   |
| =type=        | string | any       | Value type of the field                         |
| =min=         | int    | 0         | Minimum number of rows for the field            |
| =max=         | int    | unlimited | Maximum number of rows for the field            |
| =description= | string | —         | Free text (informational, e.g. for GUI display) |

=type= is a single value type name of the data model: =nothing=, =string=,
=int=, =float=, =bool=, =datetime=, =ref=, =tree_ref=, =refbase=,
=externalref=.

The example above expresses: globally, wherever the field =rating= exists on
any entry, it must be an Int; the type ="film"= additionally bounds =rating=
to cardinality 0 or 1; and both ="person"= and ="film"= require exactly one
=name= of type String.

A type may appear in the targets of several groups, and several constraints
may target the same =(type, field)= pair (global constraints stack with
per-type ones); all of them apply.

** Schema file location and loading

The =config.json= of the repository gains an optional =schema= key holding
the path of the schema file, relative to the =.metafolder/= directory (or
absolute). When the key is absent, the daemon looks for
=.metafolder/schema.json=; if that file does not exist either, the
repository simply has no schema and no validation is performed.

The daemon parses the schema file when the repository is loaded. A schema
file that is unreadable or invalid (malformed JSON, unknown value type name,
=min > max=, =min < 0=, =targets= neither ="*"= nor a non-empty list of
strings, constrained field with a reserved =mfr_*= / =mf_*= name) causes
=POST /repos/load= to fail with =400= and a message identifying the
offending constraint. Rationale: silently loading the repository without
validation could let violating writes through; a broken schema file is a
configuration error to fix, like a broken =config.json=.

The file is read once at load time. After editing it, use the reload
endpoint (see [[*=POST /repos/:repo_uuid/schema/reload=]]) or reload the
repository.

** Constraint semantics

For a given entry =E= and field name =F=, the /applicable constraints/ are
the constraints with =field = F= of:

1. Every group with =targets = "*"=.
2. Every group whose =targets= list contains at least one of =E='s
   =mf_schema= values.

=E= satisfies a constraint on =F= when:

- *Type*: every row of =F= whose value is not =Nothing= has the value type
  =type= (or the constraint has no =type= key). =Nothing= is always
  permitted: explicit absence ("this field does not apply") must remain
  expressible regardless of the declared type.
- *Cardinality*: the number of rows of =F= on =E= (including rows whose value
  is =Nothing=) is within =[min, max]=. Rationale: =Nothing= rows count
  because they are explicit statements — =min = 1= therefore means "the field
  must be explicitly stated", even if the statement is "does not apply".

Because all applicable constraints apply cumulatively, two applicable
constraints declaring different =type= values for the same field make every
non-=Nothing= value invalid for that field. This situation is not detected at
schema load time (it depends on which types an entry combines); it surfaces
as validation errors on write.

** Restrictions

- A constraint's =field= must not name a reserved field (=mfr_*= or =mf_*=
  prefix); the schema file is rejected at load otherwise. Constraints can
  therefore never block the daemon's own writes (watcher, reconcile) nor the
  =mf_schema= mechanism itself.
- The schema file constrains user data only; it has no effect on link or
  snapshot entries managed by cross-repo sync.

** Validation model: delta validation

Validation is triggered by /user writes/ only — the API endpoints that modify
fields (entry creation, =PATCH=, field append/replace/delete, batch =set=).
Daemon-internal writes (watcher, reconcile, sync crash-recovery) are never
validated against the schema.

Validation is /delta-based/: a write only validates the field names it
touches, against the entry state resulting from the write. Fields not touched
by the request are not checked, even if they violate constraints (e.g.
violations that pre-date the constraint). Consequences, by design:

- Pre-existing violations on other fields of the same entry do not block
  unrelated writes.
- Editing the schema file (or reloading it) never scans existing entries;
  existing violations are only surfaced by the check endpoint (see
  [[*=POST /repos/:repo_uuid/schema/check=]]) and only block future writes
  that touch the violating field.
- Writing =mf_schema= (adding or removing a type on an entry) only validates
  the =mf_schema= field itself — which no constraint can target — so changing
  an entry's type never fails, even if the entry violates constraints of the
  new type. Run a schema check afterwards to list the violations.
- Deleting a field row is a write touching that field: removing the last row
  of a field with an applicable =min ≥ 1= constraint is rejected. To remove
  it anyway, first remove the type from the entry (or relax the constraint in
  the schema file and reload).
- Deleting an /entire entry/ (=DELETE /metadata/:uuid=) is always allowed;
  cardinality constraints do not apply to entry deletion.

* Daemon

** In-memory schema state

The parsed schema is kept in the repository's in-memory state (=RepoState=),
indexed by field name for direct lookup during validation: validation adds no
database query. It is replaced atomically on reload.

** Validation algorithm

On every user write inside the write transaction, after applying the
modifications but before committing:

1. Collect the set of touched field names (for batch =set=, per matched
   entry; the single =name= of the request).
2. Skip touched names with a reserved prefix (=mfr_*=, =mf_*=) — these are
   covered by the structural reserved-field checks, not by the schema.
3. Look up the applicable constraints for each =(entry, touched field)= pair
   (see [[*Constraint semantics]]); this requires the entry's resulting
   =mf_schema= values.
4. Evaluate type and cardinality checks on the resulting rows of the touched
   field.
5. On any violation: roll back the transaction and return =400= with a
   =violations= array (same item format as the check endpoint below). On
   batch =set=, the whole batch is rolled back.

** HTTP API

*** =GET /repos/:repo_uuid/schema=

Returns the currently loaded schema as parsed by the daemon (same JSON
structure as the schema file), or ={"version": 1, "groups": []}= when the
repository has no schema file. Intended for clients that need to
discover constraints — e.g. the GUI generating entry forms or field
auto-completion.

*** =POST /repos/:repo_uuid/schema/reload=

Re-reads and re-parses the schema file from disk and atomically replaces the
in-memory schema. On parse or validation error, returns =400= with a message
identifying the offending constraint; the previously loaded schema stays in
effect.

Response body on success: the newly loaded schema (as in =GET /schema=).

*** =POST /repos/:repo_uuid/schema/check=

Scans entries and reports every constraint violation. Used to audit the
repository after adding or tightening constraints (editing the schema file
never validates retroactively).

Request body (optional):
#+BEGIN_SRC json
{"query": <Query>}
#+END_SRC

When =query= is given, only matching entries are checked; otherwise the whole
repository (link entries excluded, as with normal queries).

Response body:
#+BEGIN_SRC json
{
  "checked": 1542,
  "violations": [
    {
      "entry_uuid": "abc...",
      "type": "film",
      "field": "rating",
      "kind": "type",
      "message": "value of type string not allowed (expected: int)"
    },
    {
      "entry_uuid": "abc...",
      "type": null,
      "field": "rating",
      "kind": "max_cardinality",
      "message": "3 rows, maximum is 1"
    }
  ]
}
#+END_SRC

=type= identifies the constraint's origin: the type name for a per-type
constraint, =null= for a global constraint. =kind= is one of =type=,
=min_cardinality=, =max_cardinality=.

The check is read-only and runs on a consistent snapshot (single read
transaction). It does not block writers.

* CLI

** =mf schema check [<predicate>]=

Calls =POST /schema/check= (with the optional DSL predicate compiled to a
query) and prints one line per violation:

#+BEGIN_SRC
abc...  film  rating  type             value of type string not allowed (expected: int)
abc...  -     rating  max_cardinality  3 rows, maximum is 1
Checked 1542 entries, 2 violations.
#+END_SRC

Exit code: =0= when no violation was found, =1= when at least one violation
was found, =2= on error.

Options:

| Flag     | Description                      |
|----------+----------------------------------|
| =--json= | Print the raw JSON response body |

** =mf schema reload=

Calls =POST /schema/reload= after the schema file has been edited. Prints a
confirmation, or the daemon's error message (exit code 1) when the file is
invalid — the previous schema then remains in effect.

** =mf schema show=

Calls =GET /schema= and prints the loaded schema as pretty-printed JSON.

** Declaring types on entries

Types are assigned with the standard entry commands:

#+BEGIN_SRC
# Declare an entry as a film
mf set <uuid> mf_schema:string=film

# An entry with two types (multi-map append)
mf add <uuid> mf_schema:string=documentary

# List all films
mf query 'mf_schema = "film"'
#+END_SRC

* Open questions

** Schema file distribution across linked repositories

The schema file is per-repository and outside the database: it is not
recorded in the event log and not propagated by cross-repo sync. Two linked
repositories that share entry types must keep their schema files consistent
manually. Whether sync should compare (or copy) schema files is unresolved.

** Full validation at entry creation

Delta validation means =POST /metadata= only validates the fields supplied in
the request: an entry created with =mf_schema = "film"= but without a field
required by a =min ≥ 1= constraint is accepted. Whether entry creation should
instead validate the full constraint set of the declared types (stricter, but
inconsistent with the delta model) is to be revisited with usage experience.

** Value-range constraints

Constraints currently cover value type and cardinality only. Range
constraints (e.g. =0 ≤ rating ≤ 10=), enumerations, or regex constraints on
string values could be added later with additional constraint keys in the
schema file.
