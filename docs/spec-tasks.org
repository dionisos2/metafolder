#+TITLE: Metafolder — Tasks (background operations & progress)
#+OPTIONS: toc:2 num:t

* Overview

Some daemon operations can take a long time on large repositories: a full
=reconcile= (filesystem walk + fingerprinting + record creation) and some
=query= evaluations (parts of which are O(n) in the repository size, even with
=limit=). Clients (CLI, GUI) need to show progress — a determinate bar when an
amount of work is known, an indeterminate spinner otherwise — and to keep
observing an operation even across their own restarts.

This spec introduces *tasks*: observable, daemon-owned units of work. A task
carries a progress record that any client can poll. The work itself is *not*
tied to the lifetime of the HTTP request that started it, so closing the GUI or
CLI mid-operation does not interrupt it (this already holds today — reconcile
runs on a blocking thread that runs to completion regardless of the request;
tasks make that work *observable*).

The word /operation/ is already used by the event log
([[file:spec-event-log.org][spec-event-log.org]]) for the rows of a revision;
this feature deliberately uses /task/ to avoid the collision.

* Concepts

** Task

A task is one observable unit of work on a single repository. It has:

- a *UUID* =id= (generated by the daemon),
- a *kind*: =reconcile=, =query=, =flush= (the watcher's executor flush;
  see [[file:spec-file-tracking.org][spec-file-tracking.org]] "Event
  batching"), or =load= (warming a freshly loaded repository — populating the
  tree cache and building the query index; reported so the GUI shows a load
  progress bar, phases =tree cache= then =index=),
- a *status*: =pending=, =running=, =done=, =failed=, or =cancelled=,
- a *progress* record: a =phase= string and optional =done= / =total= counts
  (see [[*Progress]]),
- timestamps =started_at= (ISO-8601, set when the task is created) and
  =finished_at= (ISO-8601, set when it reaches =done= / =failed=),
- on completion, exactly one of a =result= payload (=done=) or an =error=
  message (=failed=). The =result= is only populated for kinds whose initiating
  request does *not* already carry the result (currently =reconcile=; see
  [[*Reconcile as a task]]).

The UUID is what makes consecutive runs unambiguous: a client that starts a
task keeps its id and always polls that exact task, so it can never confuse its
own run's result with a later run that happens to reuse a slot.

** Task lifecycle and statuses

#+BEGIN_SRC
pending  → running → done
                   → failed
                   → cancelled
#+END_SRC

- =pending=: the task is registered but is waiting for the per-repository
  connection lock (e.g. a query queued behind a running reconcile). It is
  already observable.
- =running=: the task holds the connection and is doing work; its progress
  record is updated as it advances.
- =done= / =failed= / =cancelled=: terminal. The task is retained in the
  registry for a short time (see [[*Result retention and eviction]]) so a client
  polling slightly after completion can still read the outcome. =cancelled= is a
  task stopped at the user's request (see [[*Cancellation]]); it is distinct from
  =failed= so a deliberate stop is not read as an error.

** Progress

The progress record is uniform across kinds:

#+BEGIN_SRC json
{"phase": "fingerprint", "done": 1240, "total": 5000}
#+END_SRC

- =phase=: a short, kind-specific label (e.g. =walk=, =fingerprint=, =create=,
  =refresh=, =mime= for reconcile; =querying= for query).
- =done= / =total=: optional unsigned integers. Either may be =null= when the
  amount of work is unknown. In particular, a single =query= is mostly opaque
  SQL the daemon cannot peek into, so a query task usually reports
  =done = null=, =total = null= for its whole duration.

*Determinate vs indeterminate display.* Clients render:

- a *progress bar* when both =done= and =total= are known (and =total > 0=),
- an *indeterminate spinner / loading animation* otherwise (either count
  =null=), labelled with =phase=.

** The task registry

Each loaded repository owns an in-memory task registry
(=Mutex<HashMap<Uuid, Task>>= on its =RepoState=). The registry is *separate
from the SQLite connection*: this is required, not incidental. A reconcile
holds the connection lock for its entire duration, so any progress reader that
touched the database would itself block until the reconcile finished. Reading
the registry never touches the database, so =GET /tasks= responds instantly
even while a reconcile is running.

The registry is never persisted: a daemon restart starts with an empty
registry (any in-flight work was on threads that died with the process).

** Concurrency and serialization

A repository has a single =Mutex<Connection>=; all database work (reconcile,
query, watcher flush) takes it. Therefore *at most one task per repository is
=running= at any instant*; others are =pending=. Tasks on *different*
repositories run in parallel (each has its own connection and registry).

Per kind:

- *reconcile* — at most one =reconcile= task per repository may exist in a
  non-terminal state (=pending= or =running=). A second =reconcile= request is
  *rejected* with =409= rather than queued: a duplicate run would only redo
  work already in progress. (See [[*Reconcile as a task]].)
- *query* — not rejected. Multiple query tasks may be =pending= / =running=;
  they serialize naturally on the connection lock. A query that arrives during
  a reconcile waits (status =pending=) until the reconcile releases the lock.
- *flush* — the watcher's executor flush is recorded as a =flush= task. Its
  serialization against reconcile/query is *already* guaranteed by the
  connection lock; recording it as a task adds *observability and uniformity*
  (a client can see "a flush is running, that is why my query is pending"), not
  a new correctness guarantee. The executor still coalesces pending filesystem
  events as before, so there is at most one =flush= task per repository.

** Result retention and eviction

A terminal task (=done= / =failed=) stays in the registry until it is older
than *10 seconds* (measured from =finished_at= against a monotonic internal
clock). Eviction is *lazy*: it happens on any registry access (insert / list /
get). Because the only producers of tasks are requests, which touch the
registry, an expired task is reclaimed promptly in practice. A lingering
terminal task with no further activity is only memory (a finished result is
small) and is reclaimed on the next access.

*Reads are idempotent.* Polling a task — via =GET /tasks= or
=GET /tasks/:id= — never deletes it. This is required for the motivating
scenarios: several observers (a polling GUI plus a CLI) must each be able to
read the same task's result, and a single client polling in a loop must not
delete the result out from under its own next read. Memory is bounded by the
10-second TTL, not by consumption.

(A future explicit =DELETE /tasks/:id= /ack/ could free a result early, and is
the natural place to add cancellation — see [[*Future extensions]] — but
neither is part of this spec.)

* Daemon

** Task JSON shape

#+BEGIN_SRC json
{
  "id": "0f1e2d3c4b5a69788796a5b4c3d2e1f0",
  "repo_uuid": "aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa",
  "kind": "reconcile",
  "status": "running",
  "phase": "fingerprint",
  "done": 1240,
  "total": 5000,
  "started_at": "2026-06-21T11:30:00.000Z",
  "finished_at": null,
  "result": null,
  "error": null
}
#+END_SRC

A terminal =reconcile= task carries its =ReconcileResult= in =result=:

#+BEGIN_SRC json
{
  "id": "...", "repo_uuid": "...", "kind": "reconcile",
  "status": "done", "phase": "mime", "done": 5000, "total": 5000,
  "started_at": "...", "finished_at": "2026-06-21T11:31:12.000Z",
  "result": {"created": 2, "moved": 1, "candidates": []},
  "error": null
}
#+END_SRC

UUIDs are 32-char lowercase hex, like everywhere else in the API.

** =GET /repos/:repo_uuid/tasks=

Lists the repository's tasks currently in the registry: all =pending= /
=running= tasks plus any terminal task still within its retention window.

Response: =200 OK=, a JSON array of task objects (possibly empty).

** =GET /repos/:repo_uuid/tasks/:task_uuid=

Returns one task by id.

Response: =200 OK= with the task object, or =404= if the id is unknown (never
created, or already evicted).

** =GET /tasks=

Global overview across all loaded repositories: the concatenation of every
repository's listed tasks (each object carries its =repo_uuid=). This is the
"what is running right now" view; it is what a status display polls.

Response: =200 OK=, a JSON array of task objects.

(=GET /health= is unchanged — a minimal liveness check returning
={"status": "ok"}=. The list of in-flight work lives at =GET /tasks=.)

** Reconcile as a task

=POST /repos/:repo_uuid/reconcile= is the *single* reconcile endpoint and is
*asynchronous*. The request body carries =threshold=, =mime=, =refresh= (see
[[file:spec-file-tracking.org][spec-file-tracking.org]] "Reconcile") plus an
optional =metarecord= (32-char hex): when present the reconcile is *scoped* to
that metarecord's subtree; absent, it covers the whole repository. (=threshold=
applies to the whole-repository reconcile only.) Instead of blocking until the
work finishes and returning the =ReconcileResult=, the daemon:

1. rejects the request with =409 Conflict= if a =reconcile= task for this
   repository is already =pending= or =running= (the in-flight task is
   discoverable via =GET /repos/:repo_uuid/tasks=);
2. otherwise creates a =reconcile= task, starts the work on a background
   thread, and returns =202 Accepted= immediately:

#+BEGIN_SRC json
{"task_id": "0f1e2d3c4b5a69788796a5b4c3d2e1f0"}
#+END_SRC

The background work runs exactly as specified in
[[file:spec-file-tracking.org][spec-file-tracking.org]] (single transaction,
same steps), updating the task's =phase= / =done= / =total= as it advances
through the walk, fingerprint, create, refresh, and mime phases. On success the
task becomes =done= with the =ReconcileResult= in =result=; on error it becomes
=failed= with the message in =error=.

The client retrieves the outcome by polling =GET /repos/:repo_uuid/tasks/:id=
until the task is terminal, then reads =result= (or =error=). The result is
available for the retention window after completion.

A scoped reconcile (with =metarecord=) behaves the same way: same task, same
=202= / poll / =result= flow, same =409= dedup (at most one reconcile of any
scope at a time per repository). Scope errors — an unknown metarecord, or one
with no valid =mfr_path= — surface as the task becoming =failed= with the
message in =error= (there is no separate synchronous endpoint to return a 404
on). The former =POST /repos/:repo_uuid/metarecords/:metarecord_uuid/reconcile=
endpoint is *removed*.

*** Reconcile progress phases

The whole-repository reconcile reports these phases (in order) on its task:

| Phase            | Counts                                                       |
|------------------+-------------------------------------------------------------|
| =walk (depth N)= | =done=/=total= = directories processed at the current depth  |
| =stat=           | =done=/=total= over the discovered paths (stat each)         |
| =index=          | =done=/=total= over the discovered paths (resolving each)    |
| =scan=           | =done=/=total= over the tracked metarecords (orphan detection) |
| =fingerprint=    | =done=/=total= over orphaned metarecords                     |
| =similarity=     | indeterminate (only when =threshold= is set)                |
| =create=         | =done=/=total= over new files                                |
| =refresh=        | =done=/=total= over discovered paths (when enabled)          |
| =mime=           | =done=/=total= over disk files (when enabled)                |

The traversal is split so progress is determinate for the heavy work. The
=walk= is a *pure* breadth-first traversal (just =read_dir= + the directory
entry's file type — *no* stat) that collects every eligible path; it is fast
and reports progress by depth level (the current depth, and how far through that
level's directories it is). Its true total is unknowable up front, but it is
cheap. Once it finishes, the full path list is known, so every later phase —
starting with =stat= (the expensive syscall pass that was previously folded
into the walk) — has an exact total and a determinate bar.

To keep the walk cheap on large repositories, eligibility evaluation memoises
ancestor =mf_watch= / =mf_ignore= reads and compiles each distinct =mf_ignore=
pattern once per reconcile (rather than per entry).

** Query as a task

=POST /repos/:repo_uuid/query= remains *synchronous*: it still returns its
paginated results (or =select: "*"= records) directly in the response, exactly
as before. The only addition is observability: for the duration of the call the
daemon registers a =query= task so that another client can see a query is
running.

The query task carries *no* =result= payload (the results are delivered on the
query response itself), and typically reports =done = null= / =total = null=
for its whole life (the heavy part is opaque SQL). When the response is sent the
task becomes terminal (=done=, or =failed= on error) and is subject to the same
retention/eviction.

This asymmetry is deliberate: =reconcile= is asynchronous so its result *must*
be retrievable via the task; =query= is synchronous so its result travels with
the response and the task is purely an observation handle.

** Watcher flush as a task

When the executor flushes the =pending_operation= buffer (see
[[file:spec-file-tracking.org][spec-file-tracking.org]] "Event batching"), it
registers a =flush= task for the duration of the flush. It is internal
(not client-initiated) but observable, and is shown by default alongside
user-initiated tasks. (Clients MAY offer a filter to hide =flush= tasks, since
flushes are frequent — one per 500 ms quiet period of filesystem activity — but
the default is to show them.)

** Cancellation

A task that is still active (=pending= or =running=) can be stopped:

*** =POST /repos/:repo_uuid/tasks/:task_uuid/cancel=

Requests cancellation of the task. The mechanism depends on the kind:

- *reconcile* — *cooperative*. The reconcile polls a cancellation flag at its
  progress checkpoints (the same points it reports =done= / =total=); when the
  flag is set it stops at the next checkpoint and *drops its transaction without
  committing*, so the database is left exactly as it was before the reconcile
  started (the single-transaction guarantee makes the rollback automatic — there
  is no partial reconcile). Cancellation therefore takes effect within one
  progress step of the request, not instantly mid-syscall.
- *query* — *interrupted*. The running SQL is aborted via SQLite's interrupt
  handle, registered on the task while the query holds the connection. The
  waiting =POST …/query= client receives a =409= (=query cancelled=) instead of
  results.
- *flush* — *not cancellable* (it is short and internal). → =400=.

On success the task transitions to =cancelled= (terminal) once its worker
unwinds, and is retained/evicted like any other terminal task. The response is
=200 OK= with the task's current view (it may still read =running= for the brief
window before the worker observes the request).

| Situation                                  | HTTP status |
|--------------------------------------------+-------------|
| Cancellation requested, task still active  | 200         |
| Task already terminal (nothing to stop)    | 409         |
| Task kind not cancellable (=flush=)        | 400         |
| Task id unknown / evicted                  | 404         |

The flag is set and the SQLite interrupt fired *synchronously* in the cancel
request; the =cancelled= status appears once the worker reaches its next
checkpoint (reconcile) or the interrupted statement returns (query).

** Error handling

| Situation                                   | HTTP status | Notes                                   |
|---------------------------------------------+-------------+-----------------------------------------|
| Reconcile requested while one is in flight  | 409         | In-flight task at =GET …/tasks=         |
| Task id unknown / evicted                   | 404         |                                         |
| Cancel of a terminal task                   | 409         | Nothing to stop                         |
| Cancel of a =flush= task                    | 400         | Not cancellable                         |

(Other statuses are as in [[file:spec-main.org][spec-main.org]] "Error
handling".)

* CLI

The CLI commands for tasks. Repo scoping follows the shared conventions
([[file:spec-main.org][spec-main.org]] "Global options"): task commands take
=--repo= / =METAFOLDER_REPO=, except =mf tasks --all= which lists across all
loaded repositories.

** =mf tasks [--all] [--json]=

Lists tasks. With =--repo=, calls =GET /repos/:repo/tasks=; with =--all=, calls
the global =GET /tasks=. Human-oriented table by default (id, kind, status,
phase, progress); =--json= prints the raw array. All kinds, including =flush=,
are shown by default.

** =mf task <id> [--stop] [--json]=

Without =--stop=, calls =GET /repos/:repo/tasks/:id= and prints the single task.
Exit code 1 if the id is unknown (404).

With =--stop= (alias =--cancel=), calls =POST /repos/:repo/tasks/:id/cancel= to
stop the task instead of showing it (see [[*Cancellation]]), then prints the
returned task line. A task that is already finished (409), of a non-cancellable
kind (400), or unknown (404) is reported as an error on stderr with exit code 1.

** =mf reconcile= (updated)

=mf reconcile= keeps its current blocking *user experience* but is now built on
the async endpoint: it calls =POST …/reconcile=, takes the returned =task_id=,
*polls* =GET …/tasks/:id= until the task is terminal (rendering progress to
stderr — a bar when =done= / =total= are known, a spinner otherwise; drawn only
when stderr is a terminal, so pipes and scripts see nothing), then
prints the final summary from =result= to stdout (unchanged format). A =409=
(reconcile already running) is reported as such. =mf repo load= waits on the
=load= warmup task with the same rendering (spec-main "mf repo load").

- =--no-wait= prints the =task_id= and returns immediately (exit 0) instead of
  polling.
- =--poll-interval <ms>= overrides the poll cadence (the cadence is purely a
  client concern; the daemon does not dictate it).

* GUI

See [[file:spec-gui.org][spec-gui.org]] for the GUI architecture. In-flight
tasks are shown in a *dedicated task bar*, separate from the status bar so
progress never saturates it:

- The task bar polls =GET /tasks= (all loaded repositories) at its own cadence
  and lists every =running= / =pending= task with a *progress bar* when the
  task reports =done= / =total=, otherwise a *spinner* labelled with the
  =kind= / =phase=. It is hidden when there are no tasks. Because it polls the
  daemon (not only GUI-launched work), it surfaces *any* in-flight task on the
  loaded repos — reconcile, query, flush — which matters since these tasks are
  blocking. Closing and reopening the GUI mid-reconcile resumes observing it.
- *Reconcile flow* (=reconcile:run=): the GUI posts the async reconcile and
  polls =GET …/tasks/:id= only to know when to show the final summary from
  =result= (or the error) in the status bar; live progress is shown by the task
  bar, not the status bar.
- *Query / flush*: shown by the task bar like any other task (a query that
  outlives a poll interval appears with a spinner; very short ones may complete
  between polls).
- The poll cadence is chosen entirely client-side, independently of the CLI.

The *repos panel* additionally lists each repository's active tasks beneath it,
each with a *Stop* button for the cancellable kinds (reconcile, query) that
calls =POST …/tasks/:id/cancel= (see [[*Cancellation]]). It polls =GET /tasks=
on its own cadence while mounted. This is the in-panel counterpart to the global
task bar: it ties a running task to the repository it belongs to and gives a
direct affordance to stop it.

* Future extensions

- *Explicit result ack*: a =DELETE= that frees a terminal task's result before
  its TTL, for clients that want to reclaim memory eagerly. The retention TTL
  makes this unnecessary for correctness.
- *Richer query progress*: if future query optimisations expose checkpoints
  (e.g. per-CTE or per-page counts), query tasks could report determinate
  progress instead of always-unknown counts.
