#+TITLE: Metafolder — General Specification
#+OPTIONS: toc:2 num:t

* Project Overview

Metafolder is a local metadata management system. It tracks files and
directories in /repositories/ — watched directory trees — and associates
arbitrary structured metadata with each metarecord. The system is split into
four crates:

- =core=: shared data model (metarecords, fields, values, queries)
- =daemon=: background HTTP server managing repositories and exposing a REST API
- =cli=: command-line tool that communicates with the daemon
- =gui=: graphical interface (Tauri v2 + Svelte 5, see spec-gui.org)

* Daemon

** General description

The daemon is a long-running background HTTP server that manages one or more
/repositories/. A repository is a directory tree on the filesystem whose
metarecords are tracked in a local SQLite database. The daemon exposes a REST API
consumed by the CLI and GUI.

The daemon is local-only (binds to =127.0.0.1= by default). It performs no
*user* access control (single-user model), but it does gate every request with
a per-session token that only a process able to read a user-only token file can
supply (see [[file:spec-auth.org][spec-auth.org]]). This keeps web content
running in the user's browser — which can reach =127.0.0.1= — out.

** Non-goals

- No built-in remote protocol: the daemon is local-only and makes no network
  calls. Cross-machine synchronisation is handled at the CLI/script level (e.g.
  loading a remote repo over an SSH-mounted filesystem into the local daemon).
- No multi-user access control. A single local user is assumed; the only
  authentication is the loopback session token of [[file:spec-auth.org][spec-auth.org]]
  (which keeps non-token-bearing local processes, notably browser content, out
  — it is not a user-to-user mechanism).
- No conflict resolution between concurrent daemon instances on the same repo.

** Version scope

Features tagged =:v2:= on headings or marked =[v2]= inline are deferred to
version 2. Version 1 covers: CRUD, =TreeRef= / tree cache, file watcher (basic
events), reconcile (fingerprint phase only), query language, reserved fields
(=mfr_*= / =mf_*=), the watch/ignore system, the user schema
([[file:spec-schema.org][spec-schema.org]]), cursor pagination, and the
daemon-side event log ([[file:spec-event-log.org][spec-event-log.org]]):
logged write flow, =pending_operation= buffer, atomic metadata-only rollback,
log reading and pruning. The coordinated navigation protocol and the
log/rollback/prune CLI (originally v2) are now implemented as well. Background
tasks and progress reporting ([[file:spec-tasks.org][spec-tasks.org]]) — async
reconcile, observable queries, the task registry — are part of version 1.

** Key invariants

- Every filesystem metarecord in a watched repository has exactly one corresponding
  metadata metarecord in the database.
- When a file or directory is deleted, its metadata metarecord is preserved; the
  =mfr_path= field is set to =Nothing= rather than deleting the row.
- =set_field= replaces *all* rows for =(uuid, field_name)=, collapsing multi-map
  fields to a single value.
- A metadata metarecord whose =db_ids= contains more than one repository UUID (a
  /link metarecord/) can only be created or modified when every listed repository is
  simultaneously loaded in the same daemon instance. [v2]
- The =version= counter on a metadata metarecord is managed exclusively by the
  daemon; it is incremented on every write to =fields= and cannot be set
  directly by users.
- A repository database must be held under an exclusive SQLite lock for the
  entire lifetime of the daemon connection (=PRAGMA locking_mode = EXCLUSIVE=).
  This prevents a second daemon instance from loading the same repo
  simultaneously, which would cause version drift and watcher conflicts (e.g.
  one daemon's watcher updating =mfr_path= and bumping =version= while the other
  is committing a sync).
- =TreeRef= references must form a forest (no cycles per =field_name=) with a
  maximum depth of 1000. The daemon rejects any write that would create a cycle
  or exceed this depth. The check walks the parent chain iteratively using the
  tree cache (fetching missing nodes from the DB step by step as needed).

The daemon listens on =127.0.0.1:7523= by default (configurable via =--port=).
All request and response bodies are JSON. Errors return a JSON object
={"error": "<message>"}= with an appropriate HTTP status code.

** Daemon configuration

The daemon reads an optional configuration file at startup:
=$XDG_CONFIG_HOME/metafolder/daemon/config.toml= (i.e.
=~/.config/metafolder/daemon/config.toml= on Linux). The =--config= flag
overrides this path. This file is distinct from the per-repository
=.metafolder/config.json= (machine-managed repo data, kept as JSON).

#+BEGIN_SRC toml
[[load]]
root = "/data/music"

[[load]]
metafolder = "/ssd/meta/music/.metafolder"
#+END_SRC

- =load= (optional): repositories to load at startup. Each entry has the
  same shape as a =POST /repos/load= body: exactly one of =root= or
  =metafolder=.
- A missing configuration file is equivalent to an empty one.
- A malformed configuration file (invalid TOML, unknown keys, a metarecord with
  both or neither of =root=/=metafolder=) aborts daemon startup with an
  error message.
- A repository that fails to load (missing path, locked database, ...) is
  reported as a warning on stderr; the daemon starts anyway and the other
  metarecords are still loaded.

** Repository management

#+BEGIN_SRC
GET  /repos
POST /repos/init
POST /repos/load
#+END_SRC

*** =GET /repos=

Returns the list of currently loaded repositories.

Response body:
#+BEGIN_SRC json
  [
  {
  "repo_uuid": "...",
  "name": "...",
  "root": "/path/to/dir",
  "internal_dir": "/path/to/dir/.metafolder/internal",
  [...]
  }
  ]
#+END_SRC

*** =POST /repos/init=

Initialises a new repository. Creates =config.json= and =internal/db.sqlite=
inside the =.metafolder/= directory. Fails if a repository already exists
there.

Request body:
#+BEGIN_SRC json
{"root": "/absolute/path"}
#+END_SRC

Optional: =metafolder= overrides where =.metafolder/= is created (external
database). Defaults to =<root>/.metafolder/=.
#+BEGIN_SRC json
{"root": "/data/music", "metafolder": "/ssd/meta/music/.metafolder"}
#+END_SRC

Optional: =name= sets the repository name. When absent (or empty), the name is
derived from the root directory's file name.
#+BEGIN_SRC json
{"root": "/data/music", "name": "My Music"}
#+END_SRC

Response body: ={"repo_uuid": "..."}=

*** =POST /repos/load=

Loads an existing repository into the daemon's in-memory state, starts the
file watcher.

Standard form (=.metafolder/= is inside =root=):
#+BEGIN_SRC json
{"root": "/absolute/path"}
#+END_SRC

External database form (specify the =.metafolder/= path directly; =root= is
read from =config.json= inside it):
#+BEGIN_SRC json
{"metafolder": "/ssd/meta/music/.metafolder"}
#+END_SRC

Response body: ={"repo_uuid": "..."}=

** Error handling

| Situation                              | HTTP status | Notes                          |
|----------------------------------------+-------------+--------------------------------|
| Repository not found                   | 404         |                                |
| Metarecord not found              | 404         |                                |
| Task id unknown / evicted              | 404         | see [[file:spec-tasks.org][spec-tasks.org]]            |
| Repository already initialised         | 409         |                                |
| Reconcile requested while one in flight | 409        | see [[file:spec-tasks.org][spec-tasks.org]]            |
| Malformed JSON body                    | 400         |                                |
| Invalid query                          | 400         |                                |
| TreeRef parent metarecord not found         | 400         |                                |
| TreeRef write would create a cycle     | 400         |                                |
| TreeRef depth exceeds 1000             | 400         |                                |
| Schema constraint violation            | 400         | =violations= array in body (see [[file:spec-schema.org][spec-schema.org]]) |
| Invalid schema file (load / reload)    | 400         | Message identifies the offending constraint |
| Invalid pagination cursor              | 400         | Cursor/query-sort mismatch     |
| Internal error (DB, I/O)               | 500         | Error message in body          |

** Open questions

*** Daemon lifecycle
No mechanism yet for cleanly unloading a repository (stopping its watcher,
releasing its DB connection) without restarting the whole daemon.

* CLI

** General description

The CLI is a thin client over the daemon's HTTP API: it performs no direct
database or filesystem-watching work itself. The binary is named =mf=. Each
spec file documents the CLI commands for its own feature area in its =* CLI=
section; this section defines the conventions shared by all commands.

** Global options

| Option         | Env variable             | Description                                                |
|----------------+--------------------------+------------------------------------------------------------|
| =--daemon-url= | =METAFOLDER_DAEMON_URL=  | Daemon base URL; default =http://127.0.0.1:7523=           |
| =--repo <UUID>= | =METAFOLDER_REPO=       | Target repository; required by all repo-scoped commands    |

A repo-scoped command invoked without =--repo= / =METAFOLDER_REPO= fails
immediately with a usage error (exit code 2), without contacting the daemon.

** Output conventions

- Normal results go to stdout; errors and warnings to stderr.
- Machine-friendly by default: UUID lists are printed one per line; structured
  results are JSON. Commands with a human-oriented default format provide a
  =--json= flag that prints the raw response body instead.
- Daemon error responses (={"error": "..."}=) are printed to stderr as
  =error: <message>=.

** Exit codes

| Code | Meaning                                                          |
|------+------------------------------------------------------------------|
| 0    | Success                                                          |
| 1    | Operation failed (daemon error, violations found, no match, ...) |
| 2    | Usage error (bad arguments, unparsable DSL/field spec, missing =--repo=) |

** Repository commands

*** =mf init <root> [--metafolder <path>]=

Calls =POST /repos/init= and prints the new repository UUID. =--metafolder=
selects an external database location (see [[*=POST /repos/init=]]).

*** =mf load <root | --metafolder <path>>=

Calls =POST /repos/load= (standard form with a root path, or external
database form with =--metafolder=) and prints the repository UUID.

*** =mf repos=

Calls =GET /repos= and prints the loaded repositories as pretty-printed JSON.
