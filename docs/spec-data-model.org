#+TITLE: Metafolder — Data Model Specification
#+OPTIONS: toc:2 num:t

* Overview

The data model defines the core structures shared across all components:
repositories, metadata metarecords, fields, and values.

* Concepts

** Repository

Persisted as =config.json= inside the =.metafolder/= directory.

By default, =.metafolder/= is a subdirectory of the watched root. It can also
live at an arbitrary path (external database), useful when the data disk is
read-only or when metadata should be kept on a separate, faster device.

The database and other daemon-managed volatile files live in the
=.metafolder/internal/= subdirectory, which is always excluded from tracking
(see [[file:spec-file-tracking.org][spec-file-tracking.org]]). The rest of
=.metafolder/= (=config.json=, =schema.json=) is ordinary content that the
user may track like any other file. Repositories created before the
=internal/= layout are migrated automatically at load time (the =db.sqlite*=
family is moved into =internal/=).

| Field       | Type   | Description                            |
|-------------+--------+----------------------------------------|
| =repo_uuid= | UUID   | Stable identifier for this repository  |
| =name=      | string | Name of the repository                 |
| =version=   | string | Config schema version                  |
| =root=      | path   | Absolute path to the watched directory |
| =schema=    | path   | Optional path to the user schema file, relative to =.metafolder/= (or absolute); default =.metafolder/schema.json= (see [[file:spec-schema.org][spec-schema.org]]) |

** Metarecord

A metarecord is a coherent set of metadata about one subject — typically a file —
or a value shared by such metadata (a tag, a collection, a cross-repo link).
The fields of a metarecord are its metadata; "metarecord" names the unit, "metadata"
names what the fields are.

| Field     | Type       | Description                                              |
|-----------+------------+----------------------------------------------------------|
| =uuid=    | UUID       | Stable identifier for this metarecord                         |
| =db_ids=  | list<UUID> | Repositories that own this metarecord (usually one)           |
| =version= | u64        | Monotonic write counter, managed by the daemon           |
| =fields=  | list       | Ordered list of =Field= values                           |

Fields form a *multi-map*: multiple fields with the same name are allowed.

=db_ids= normally contains a single repository UUID (the owning repo). When it
contains two UUIDs the metarecord is a /link metarecord/ shared between two repositories
(see [[*Cross-repo Synchronisation]]).

By default, queries return only metarecords whose =db_ids= equals exactly
={current_repo}=. Metarecords with multiple =db_ids= are excluded unless explicitly
requested.

** Field and Value

A field is a =(name, value)= pair. Value is one of:

| Variant    | Rust type       | Semantics                                    |
|------------+-----------------+----------------------------------------------|
| =Nothing=  | —               | Explicit absence ("this field does not apply") |
| =String=   | =String=        | UTF-8 text                                   |
| =Int=      | =i64=           | 64-bit signed integer                        |
| =Float=    | =f64=           | 64-bit float                                 |
| =Bool=     | =bool=          |                                              |
| =DateTime= | =i64=           | Unix milliseconds (UTC); JSON wire form is an ISO-8601 string (=YYYY-MM-DDTHH:MM:SSZ=) |
| =Ref=          | =UUID=         | Reference to another metarecord's UUID (same repo)              |
| =TreeRef=      | =(UUID?, String)= | Position in a named tree: =(parent_uuid, name)=             |
| =RefBase=      | =UUID=         | Reference to a repository UUID                             |
| =ExternalRef=  | =(UUID, UUID)= | Cross-repo reference: (repo UUID, metarecord UUID)              |

A =TreeRef= stores the position of a metarecord in a named tree as =(parent_uuid, name)=,
kept in the =value_uuid= and =value_name= columns of the =field= table
(see [[*SQLite Schema]]):

- =value_uuid=: UUID of the parent metarecord (zero UUID =x'00...00'= for root nodes).
- =value_name=: the name component contributed by this metarecord (e.g. filename, tag name).

Multiple independent trees can coexist within the same repository by using different
field names (e.g. =mfr_path= for the filesystem tree, =parent= for a tag hierarchy).

Rename (same parent, new name): update =value_name= of one row — O(1).
Move (different parent): update =value_uuid= of one row — O(1).

Path-to-UUID resolution is accelerated by the *Tree Cache* (see [[*Tree Cache]]).

=Nothing= is distinct from the field simply not existing:
- field present with =Nothing= value → "this field does not apply" (IsAbsent)
- field absent entirely → "unknown" (IsUnknown)

All values serialize to JSON as ={"type": "<variant>", "value": <json_value>}=.
=DateTime= is held internally as Unix milliseconds but encoded on the wire as an
ISO-8601 UTC string; a non-parsable datetime string is rejected on deserialization.

** One value type per field name

A field name has a *single non-=Nothing= value type* across the whole
repository. =Nothing= always coexists with it (it is absence, not a type): a
field may be its type where it applies and =Nothing= where it does not. The
multi-map is unaffected — several rows of the same name are still allowed, but
they all share that one type (e.g. many =tag= rows, all =String=; many =author=
rows, all =Ref=). Mixing value types under one field name is rejected.

The reference variants are treated as *distinct* types, not a family: a field is
=Ref=, or =RefBase=, or =ExternalRef=, or =TreeRef= — never a mix. A link that
is sometimes local and sometimes cross-repo is modelled uniformly as
=ExternalRef=.

The type is established by the first non-=Nothing= write to that name; the user
schema may also declare it explicitly (see
[[file:spec-schema.org][spec-schema.org]]). A later write of a conflicting type
is rejected with =400=. The current implementation realises the =field_type=
registry as a *derived probe* with no separate registry table: the established
type is read as one representative non-=Nothing= row of the name
(=… WHERE field_name = ? AND value_type != 'nothing' LIMIT 1=, seeking the
=field_name= range via the existing =idx_field_name=, stopping at the first such
row), amortised by a per-revision cache in the =Writer= (one probe per field
name per revision). The type therefore unlocks automatically when a name's last
non-=Nothing= value is removed. A materialised registry (and its role as the
per-field index-encoding selector) comes with the indexing work (see
[[file:spec-indexing.org][spec-indexing.org]]).

This uniformity simplifies the rest of the model: comparisons
([[file:spec-query.org][spec-query.org]]) are same-type, and sorting needs no
cross-type precedence (see [[*Sort specification (=POST /query=)]]).

*** Changing a field's type (retype)

Because the first write locks the type, an explicit *retype* operation changes a
field name's type repository-wide, converting every existing non-=Nothing= row:

- HTTP: =POST /repos/:repo_uuid/retype= with body ={"name": "<field>", "to":
  "<type>"}=; CLI: =mf retype <name> <type>=.
- It is a logged write (one revision through =log::Writer=, so it is rollbackable
  like any other change) and updates the =field_type= registry.
- Conversions follow a defined table: lossless where possible (=Int ↔ Float=,
  =Bool ↔ Int= as =0/1=, =DateTime ↔ Int= as Unix-ms, any type → =String= via
  its display form), parsed where textual (=String → Int/Float/Bool/DateTime=).
  =Ref=/=RefBase=/=ExternalRef=/=TreeRef= have no scalar conversion.
- A value that cannot be converted is set to the target type's *default
  sentinel*, chosen to be easily found afterwards: =Int= =0=, =Float= =0.0=,
  =Bool= =false=, =String= ="="" (empty), =DateTime= the Unix epoch
  (=1970-01-01T00:00:00Z=), the reference types =Nothing= (findable via
  =IS ABSENT=). The operation returns a summary (counts, and the uuids whose
  values fell back to the sentinel) so they can be located and fixed.

** Reserved fields

Two categories of reserved field names exist, distinguished by prefix:

- =mfr_*= (*metafolder restricted*) — written by the daemon; require =force= to
  override via the API (see [[*Reserved fields]]).
- =mf_*= (*metafolder*) — read by the daemon to control its behaviour; written
  freely by the user without =force=.

The stored fields for a file/directory metarecord are:

| Field              | Prefix | Value type | Description                                                                                                |
|--------------------+--------+------------+------------------------------------------------------------------------------------------------------------|
| =mfr_path=         | mfr    | TreeRef    | Position in filesystem tree; =Nothing= if explicitly deleted (IsAbsent); stale TreeRef if location unconfirmed; absent if never assigned (IsUnknown) |
| =mfr_type=         | mfr    | String     | ="file"=, ="dir"= or ="symlink"= (symlinks are never dereferenced — see [[file:spec-platform.org][spec-platform.org]]) |
| =mfr_size=         | mfr    | Int        | Size in bytes at last watcher/reconcile update; for a symlink, the length of its own target path (=lstat=) |
| =mfr_mtime=        | mfr    | DateTime   | Last content modification time (from =lstat=; the link's own, for a symlink)                               |
| =mfr_symlink_target= | mfr  | String     | Target **path** of a symlink (from =readlink=); present only when =mfr_type = "symlink"=                   |
| =mfr_btime=        | mfr    | DateTime   | File creation time, when the platform/filesystem provides one; absent otherwise (see [[file:spec-platform.org][spec-platform.org]]) |
| =mfr_permissions=  | mfr    | String     | Unix permission bits as octal string, e.g. ="0644"=; Unix only, never written on Windows (see [[file:spec-platform.org][spec-platform.org]]) |
| =mfr_uid=          | mfr    | Int        | Owner user ID; Unix only, never written on Windows                                                         |
| =mfr_gid=          | mfr    | Int        | Owner group ID; Unix only, never written on Windows                                                        |
| =mfr_mime=         | mfr    | String     | MIME type from content analysis (see [[file:spec-platform.org][spec-platform.org]]); computed by reconcile (opt-out) for files that lack it; absent when undetectable or not yet computed |
| =mfr_partial_hash= | mfr    | String     | Hex xxHash3 of first+last 4 KB; absent if not yet computed or invalidated by =Modify=                      |
| =mfr_full_hash=    | mfr    | String     | Hex xxHash3 of full content; absent if not yet computed                                                    |
| =mf_watch=         | mf     | Bool       | Opt-in watch flag; controls whether this metarecord and its descendants are tracked (see [[*Watch and Ignore]])  |
| =mf_ignore=        | mf     | String     | Regex pattern (multi-map: one row per pattern); controls which paths are excluded (see [[*Watch and Ignore]]) |

The user schema feature defines one additional reserved =mf_*= field:
=mf_schema= (String, multi-map — metarecord type declaration). The constraints
themselves live in an external schema file, not in the database; see
[[file:spec-schema.org][spec-schema.org]].

The =mfr_path= TreeRef stores =(value_uuid, value_name)= for each metarecord.

The three states of the =mfr_path= field have distinct semantics:
- =mfr_path= present (valid TreeRef) → the metarecord is at that location.
- =mfr_path= stale (TreeRef whose path no longer exists on disk) → location
  uncertain; reconcile will attempt to find a match via fingerprint or
  similarity and return it as a candidate.
- =mfr_path= =Nothing= (IsAbsent) → file was explicitly removed (watcher
  =Remove= / =Rename(From)= event); no search needed. *Only the watcher sets
  =mfr_path= to =Nothing=; reconcile never does.*
- =mfr_path= absent (IsUnknown) → no path was ever assigned.

*=mfr_*= fields* are written by the daemon (watcher and reconcile). Attempting
to modify them without =force= returns =400 Bad Request=:

#+BEGIN_SRC json
{"name": "mfr_permissions", "value": {"type": "string", "value": "0755"}, "force": true}
#+END_SRC

*Warning*: daemon operations always overwrite =mfr_*= fields unconditionally,
regardless of whether the value was previously set with =force=. =force= is an
explicit acknowledgement that the value may be overwritten at any time.

Typical legitimate use-cases: restoring permissions after an erroneous
recursive permission change; correcting a misdetected MIME type; resolving an
=mfr_path= conflict the watcher cannot handle automatically.

*=mf_*= fields* are read by the daemon to control its behaviour and written
freely by the user (no =force= required). They are still reserved: any unknown
=mf_*= field name is rejected with =400= to prevent typos from silently having
no effect.

Direct lookup by path uses the tree cache (see [[*Tree Cache]]); on cache
miss, the daemon resolves each component iteratively via the
=(field_name, value_uuid, value_name)= index.

Inherited property lookup (e.g. nearest ancestor with =sync_method=) is performed
in two steps: first, walk the parent chain via the tree cache (fetching missing
nodes from the DB step by step) to collect the ordered list of ancestor UUIDs;
then issue a single query =WHERE metarecord_uuid IN (...)= to retrieve the desired
field across all ancestors at once, keeping the closest result.

* Daemon

** SQLite Schema

Each repository uses its own =.metafolder/internal/db.sqlite= file (WAL mode, foreign
keys enabled). Exception: when a repository is loaded from a network
filesystem (e.g. SSHFS, NFS, SMB), the daemon switches to =DELETE= journal
mode (=PRAGMA journal_mode=DELETE=) at connection time, because WAL requires
shared-memory files (=.wal-shm=) which network filesystems do not support
(see [[file:spec-platform.org][spec-platform.org]]). This switch is applied
per-connection and does not modify the database permanently — the next local
connection resumes WAL mode normally. A benchmark should be run once sync is
implemented to verify that DELETE journal mode is acceptable for large syncs.

*** UUID encoding

All UUIDs (metarecord UUIDs and repository UUIDs) are stored as =BLOB= (16 bytes,
raw binary). This applies uniformly to every UUID column in the schema.

The HTTP API encodes UUIDs as 32-char lowercase hex strings (no hyphens) for
readability and interoperability.

Rationale: 16-byte BLOBs enable fast binary comparison in SQLite indexes and
are the most compact representation. Hex encoding in the API is standard and
unambiguous.

The zero UUID (=x'00000000000000000000000000000000'=) is reserved as a sentinel
for =TreeRef= root nodes and must never be assigned to a real metarecord. UUID v4
has 122 random bits (2¹²² ≈ 5×10³⁶ possible values); the probability of
collision among 1 billion metarecords is ≈ 10⁻¹⁹ — negligible in practice.

#+BEGIN_SRC sql
CREATE TABLE metarecord (
    uuid     BLOB    PRIMARY KEY NOT NULL,  -- 16-byte UUID
    version  INTEGER NOT NULL DEFAULT 0     -- incremented by daemon on every field write
);

-- One row per owning repository (usually one; two for link metarecords).
CREATE TABLE metarecord_db (
    metarecord_uuid  BLOB NOT NULL REFERENCES metarecord(uuid) ON DELETE CASCADE,
    db_id          BLOB NOT NULL,           -- 16-byte repo UUID
    PRIMARY KEY (metarecord_uuid, db_id)
);

CREATE INDEX idx_metarecord_db ON metarecord_db(db_id);

CREATE TABLE field (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    metarecord_uuid   BLOB    NOT NULL REFERENCES metarecord(uuid) ON DELETE CASCADE,
    field_name      TEXT    NOT NULL,
    value_type      TEXT    NOT NULL,  -- nothing | string | int | float | bool |
                                       -- datetime | ref |
                                       -- tree_ref | refbase | externalref
    value_text      TEXT,              -- string
    value_int       INTEGER,           -- int, bool (0/1), datetime (Unix ms)
    value_real      REAL,              -- float
    value_uuid      BLOB,              -- ref/externalref: metarecord UUID; tree_ref: parent UUID (zero UUID for root)
    value_ref_repo  BLOB,              -- externalref only: repo UUID
    value_name      TEXT               -- tree_ref: name component contributed by this metarecord
);

CREATE INDEX idx_field_metarecord        ON field(metarecord_uuid, field_name);
CREATE INDEX idx_field_reverse      ON field(field_name, value_uuid, value_ref_repo)
    WHERE value_type IN ('ref', 'externalref');
CREATE UNIQUE INDEX idx_field_tree  ON field(field_name, value_uuid, value_name)
    WHERE value_type = 'tree_ref';
#+END_SRC

** Pagination

=GET /metarecords= and =POST /query= (see
[[file:spec-query.org][spec-query.org]]) support keyset (cursor-based)
pagination. Pagination is opt-in: without a =limit= parameter, the full
result set is returned in one response with the unwrapped response shape, and
nothing changes for existing clients.

*** Request parameters

| Parameter | Type   | Description                                                          |
|-----------+--------+----------------------------------------------------------------------|
| =limit=   | int    | Maximum number of results in this page; enables pagination           |
| =cursor=  | string | Opaque token from the previous page's =next_cursor=; absent for page 1 |
| =sort=    | list   | Sort specification (=POST /query= only; see below)                   |
| =count=   | bool   | Adds =total= (the full result count) to the envelope (=POST /query= only; requires =limit= — the unwrapped response has nowhere to carry it). Computed daemon-side with a =COUNT(*)= over the compiled query, without fetching rows. |

For =GET /metarecords= the parameters are query-string parameters
(=?limit=500&cursor=...=); results are sorted by metarecord UUID (ascending byte
order). For =POST /query= they are top-level JSON body keys.

*** Sort specification (=POST /query=)

#+BEGIN_SRC json
"sort": [{"field": "rating", "order": "desc"}, {"field": "title", "order": "asc"}]
#+END_SRC

- =order= is ="asc"= (default) or ="desc"=.
- The metarecord UUID (ascending) is always implicitly appended as the final sort
  key, so the total order is unambiguous and the keyset cursor can resume
  exactly after the last returned metarecord even when sort values are equal.
- =sort= without =limit= is allowed (sorted full response).
- Sorting by reserved or user fields is allowed alike.

Sort semantics per metarecord for a sort field =F=:

- Metarecords where =F= is unknown (no row) or where every row is =Nothing= sort
  *after* all metarecords with a value, regardless of =order=.
- Multi-map fields: the metarecord's sort key is its smallest value of =F= for
  ="asc"=, its largest for ="desc"=.
- All non-=Nothing= rows of =F= share one value type (see [[*One value type per
  field name]]), so values compare within that single type by its natural order:
  numeric for =Int=/=Float=/=DateTime= (the Unix-ms =value_int=), lexicographic
  for =String=, =value_name= for =TreeRef=, UUID bytes for the reference types.
  No cross-type precedence is needed (it cannot arise within one field); only the
  =Nothing=/unknown-last rule above applies.

*** Response envelope

When =limit= is present, the response is wrapped:

#+BEGIN_SRC json
{
  "results": [...],
  "next_cursor": "b64token-or-null"
}
#+END_SRC

=results= has exactly the same item format as the unpaginated response
(UUID strings, or metadata objects when =select= is used). =next_cursor= is
=null= when there are no further results. With =count: true= the envelope
also carries ="total": N= — the full result count of the query at the time
of this page (live data may shift it between pages; clients typically
request it on the first page only, e.g. the metarecord-list status line
"200/5000").

*** Cursor semantics

The cursor is an opaque token (base64-encoded JSON) encoding the sort-key
values and UUID of the last returned metarecord, plus a hash of the =(query,
sort)= pair. Clients must not parse it.

- A cursor is only valid for the same query and sort specification that
  produced it; the daemon verifies the embedded hash and returns =400=
  (=invalid cursor=) on mismatch. Cursors do not expire otherwise.
- Keyset resumption is a =WHERE (sort_key, uuid) > (cursor values)= filter:
  pages are stable against metarecords inserted or deleted in portions of the
  order already consumed. If a metarecord's sort-field value changes between two
  page requests, that metarecord may be returned twice or skipped — this is
  inherent to keyset pagination over live data and is accepted.

** HTTP API

Two layers: *resources* (a single, directly-addressed thing — cheap, idempotent)
and the *=/query/*= set layer* ([[file:spec-query.org][spec-query.org]]). They overlap only on the n=1 case
(a resource path vs a =uuid_in= query), by design. Listing/reading *several*
metarecords is a query (=POST /query=, empty query = all, =uuid_in= = a named
set) — there is no =GET /metarecords= list nor a batch-read endpoint.

#+BEGIN_SRC
POST   /repos/:repo_uuid/metarecords                                   create
GET    /repos/:repo_uuid/metarecords/:uuid                             read one
PUT    /repos/:repo_uuid/metarecords/:uuid                             set whole record
DELETE /repos/:repo_uuid/metarecords/:uuid                             delete one
POST   /repos/:repo_uuid/metarecords/:uuid/fields                      append one row
GET    /repos/:repo_uuid/metarecords/:uuid/fields/:name               read a field's value(s)
PUT    /repos/:repo_uuid/metarecords/:uuid/fields/:name               set a field (replace rows)
DELETE /repos/:repo_uuid/metarecords/:uuid/fields/:name               unset a field
GET    /repos/:repo_uuid/metarecords/:uuid/fields/:name/resolve-tree  TreeRef → path(s)
GET    /repos/:repo_uuid/fields/:id                                    read a field row by id
PATCH  /repos/:repo_uuid/fields/:id                                    rename and/or revalue a row
DELETE /repos/:repo_uuid/fields/:id                                    delete a row by id
POST   /repos/:repo_uuid/retype                                        retype a field repo-wide
GET    /repos/:repo_uuid/fields[?type=<value_type>]                    distinct field names + types
#+END_SRC

*** =POST /repos/:repo_uuid/metarecords=

Creates a metadata metarecord. Body ={"fields": [{"name", "value"}, ...], "force"?}=.
For =TreeRef= fields the client supplies =(parent_uuid, name)= directly; the
daemon validates the =parent_uuid= (null = a root, else an existing metarecord
carrying that =TreeRef= name). The filesystem root (=value_uuid = x'00...00'=,
=value_name = ""=) is created at =repo init= and is unique per field name. A
metarecord may carry several =mfr_path= rows (multi-map = several locations).
Response: the full metadata object (=id= on each field).

*** =GET /repos/:repo_uuid/metarecords/:uuid=

Returns one metarecord with all its fields. Each field carries its DB-row =id=
(used by the by-id endpoints).
#+BEGIN_SRC json
{ "uuid": "...", "db_ids": ["<repo>"], "version": 3,
  "fields": [ {"id": 42, "name": "rating", "value": {"type": "int", "value": 5}} ] }
#+END_SRC

*** =PUT /repos/:repo_uuid/metarecords/:uuid=

*Whole-record set*: replaces the entire field set, keeping the UUID, as one
=SetRecord= operation (=before= = all old fields, =after= = the given set).
Literal overwrite — fields not listed are dropped, reserved =mfr_*=/=mf_*=
included (so a careless set can untrack a file; round-trip =GET= to preserve).
Body ={"fields": [...], "force"?}= → updated full object.

*** =DELETE /repos/:repo_uuid/metarecords/:uuid=

Deletes the metarecord and all its fields (one =DeleteRecord= op). =204=.

*** =POST /repos/:repo_uuid/metarecords/:uuid/fields=

Appends one row without touching existing rows of that name (multi-map append).
Body ={"name", "value", "force"?}= → updated full object. Protected name without
=force= → =400=.

*** =GET|PUT|DELETE /repos/:repo_uuid/metarecords/:uuid/fields/:name=

A field of one metarecord, addressed by name (the direct analog of the
=/query/fields/{...}= set operations):
- =GET= → the field's value(s): ={"name": "tag", "values": [<Value>, ...]}=
  (empty list when the field is unknown).
- =PUT= → *set*: replaces all rows of =name= (one =SetField= op). Body =value=
  (one row) or =values= (a list, multi-map) — exactly one.
- =DELETE= → *unset*: removes every row of =name= (the field becomes unknown).
Protected names need =force= (in the body for PUT/DELETE). Updated full object
(PUT) / =204= (DELETE).

*** =GET /repos/:repo_uuid/metarecords/:uuid/fields/:name/resolve-tree=

Resolves a =TreeRef= field of this metarecord to its root-relative path(s).
Response ={"paths": ["<path>", ...]}= (empty when the field is absent or not a
=TreeRef=). The set form is =POST /query/fields/resolve-tree= ([[file:spec-query.org][spec-query.org]]).

*** =GET|PATCH|DELETE /repos/:repo_uuid/fields/:id=

A single field *row* by its repository-unique DB id (no metarecord UUID in the
path — the id alone addresses it). =404= if the id is unknown.
- =GET= → the row: ={"id", "name", "value"}=.
- =PATCH= → change its *name and/or value* in place, keeping the id; the value
  type is validated against the (target) field name. Body ={"name"?, "value"?, "force"?}=.
- =DELETE= → remove the row. Body ={"force"?}=, =204=.

*** =POST /repos/:repo_uuid/retype=

Converts a field's value type repository-wide (=string|int|float|bool|datetime=)
— see [[*Changing a field's type (retype)]]. Body ={"name": "<field>", "to": "<type>"}= →
={"converted": <n>, "fallback_count": <n>, "fallback_uuids": [...]}=. Reserved
=mfr_*=/=mf_*= names are rejected (the system owns their types).

*** =GET /repos/:repo_uuid/fields=

Lists the distinct field names present on the repository's metarecords, each
with its value type, ordered by name:

#+BEGIN_SRC json
[{"name": "category", "type": "tree_ref"}, {"name": "tag", "type": "ref"}, ...]
#+END_SRC

Since a field name has a single non-=Nothing= value type repository-wide ([[*One value type per field name]]),
each name appears at most once. =Nothing= rows are excluded (an explicit
absence is not a usable value). An optional =?type=<value_type>= query
parameter restricts the result to one value type (=tree_ref=, =ref=, =string=,
…). The isolation matches a query's universe — only metarecords owned
exclusively by this repository. The GUI uses it to populate field pickers (e.g.
the TreeRef-field selector of the tree-explorer panel and the Ref-field
selector of the ref-list panel, see [[file:spec-gui.org][spec-gui.org]]).

* CLI

Metarecord manipulation commands. Global conventions (binary name, exit codes)
are defined in [[file:spec-main.org][spec-main.org]].

*** Global selection flags

The repository and daemon are chosen *once*, by flags that precede the command —
never per-command:

| Flag           | Selects                                                     |
|----------------+------------------------------------------------------------|
| =-n <name>=    | the loaded repository by its (unique) name                 |
| =-u <uuid>=    | the loaded repository by UUID                              |
| =-p <port>=    | the daemon TCP port on =127.0.0.1= (default =7523=)        |

=-n= and =-u= are mutually exclusive. The daemon only listens on loopback, so a
port is all that is needed to address it (no full URL). Environment fallbacks:
=METAFOLDER_REPO= (UUID), =METAFOLDER_REPO_NAME= (name), =METAFOLDER_DAEMON_PORT=.
Daemon-wide commands (=mf repo=, =mf task=) ignore =-n=/=-u=.

Repository *names are unique among loaded repositories*: the daemon rejects
loading a repo whose name collides with one already loaded, so =-n= always
resolves to a single UUID.

** Field specification syntax

Commands that write fields take a /field spec/ argument of the form
=name:type[=value]=:

| Type          | Syntax                                  | Example                                           |
|---------------+-----------------------------------------+---------------------------------------------------|
| =nothing=     | =name:nothing= (no value)               | =rating:nothing=                                  |
| =string=      | =name:string=text= (quotes optional)    | =genre:string=jazz=, =genre:string="hard bop"=    |
| =int=         | =name:int=N=                            | =rating:int=5=                                    |
| =float=       | =name:float=X=                          | =score:float=3.14=                                |
| =bool=        | =name:bool=true|false=                  | =seen:bool=true=                                  |
| =datetime=    | =name:datetime=ISO-8601=                | =added:datetime=2024-01-01T12:00:00Z=             |
| =ref=         | =name:ref=<metarecord_uuid>=                 | =author:ref=8f3a...=                              |
| =tree_ref=    | =name:tree_ref=<parent_uuid>/<name>=; empty parent for a root: =name:tree_ref=/<name>= | =parent:tree_ref=8f3a.../félins= |
| =refbase=     | =name:refbase=<repo_uuid>=              | =origin:refbase=47ab...=                          |
| =externalref= | =name:externalref=<repo_uuid>:<metarecord_uuid>= | =link_a:externalref=47ab...:8f3a...=          |

** Selector options

=mf metarecord= picks its target metarecord(s) with a *selector option that
precedes the verb* (so the selector and verb don't compete for the same
positional, and one selector serves every verb including =field=):

| Option        | Selects                                                       |
|---------------+--------------------------------------------------------------|
| =-i <uuid>=   | one metarecord by UUID (no daemon query round-trip)          |
| =-q "<DSL>"=  | the metarecords matching a DSL predicate ([[file:spec-query.org][spec-query.org]])         |
| =-s=          | with =-q=: treat it as simplified-language text, expand first |
| (none)        | all metarecords                                              |

=-q= and =-i= are mutually exclusive; =-s= requires =-q=. =-s= expansion is a
pure client-side transformation (the shared grammar in =core=), never a daemon
round-trip. Example: =mf -n music metarecord -q 'mfr_path ->* "/docs"' field set reviewed:bool=true=.

** Verb pattern

The repo-scoped data commands form a tree (=metarecord=, =field=) whose verbs
follow one pattern at every level:

- =get= reads; =set= replaces *everything at that level*; =add= creates a new
  element (with a fresh id/UUID); =delete= removes one element.

So =get= is the inverse of =set=, and =add= the inverse of =delete=. =set= at the
metarecord level replaces all of its fields; at the field-name level, all rows of
that name; at the row level, the one row. =add= lives only where new ids are
born (a metarecord, or a row under a field name), so the by-id =field= group has
no =add=.

** Daemon- and repo-management groups

*** =mf repo {list|init|load|unload}=

Daemon-wide; take no =-n=/=-u= selector. =list= (the default for =mf repo=)
pretty-prints the loaded repositories; =init <path>= / =load <path>= print the
repo UUID; =unload= releases the repo selected by =-n=/=-u=.

*** =mf task {list|show}=

Background tasks (spec-tasks). =list= (default) lists tasks (=--all= across
repos); =show <id>= shows one (=--stop= cancels it).

*** =mf log {list|rollback|prune}=

Repository history (spec-event-log). =list= (default) is the former flat =mf
log=; =rollback= and =prune= are unchanged.

*** =mf retype <name> <type>=

Converts a field's value type repository-wide (=string|int|float|bool|datetime=).
Kept at the top level deliberately: it is a *name-scoped, whole-repository*
migration, not a selection- or id-scoped edit, so it belongs to none of the
get/set/add/delete groups.

** =mf metarecord [selector] <verb>=

The verb follows the selector options above. =add= takes no selector; =set=
needs =-i=; =delete= and =field= need a selector; =get= defaults to all.

*** =mf metarecord [-i|-q] get=

Reads metarecords; subsumes the former =list=, =query= and =get=:
- with =-i <uuid>=, prints that metarecord's full JSON object (fields incl. ids);
- with =-q= or no selector (match-all), prints the matching UUIDs, one per line
  (internal pagination over =next_cursor=).

=--select=, =--sort=, =--limit=, =--values= override the default output shape.

*** =mf metarecord add <spec> [<spec> ...]=

Creates a metarecord with the given fields and prints its UUID (the former =mf
create=). Takes *no selector* (it makes a new metarecord). The inverse of
=mf metarecord -i <uuid> delete=.

| Flag      | Description                                                |
|-----------+------------------------------------------------------------|
| =--force= | Required to write =mfr_*= fields (see [[*Reserved fields]]) |

*** =mf metarecord -i <uuid> set <spec> [<spec> ...]=

Replaces the *entire* field set of the selected metarecord, keeping its UUID —
the inverse of =mf metarecord -i <uuid> get=, equivalent to a delete + create at
fixed UUID. *Literal overwrite*: any field not listed is removed, including
reserved =mfr_*=/=mf_*= ones, so a careless set can untrack a file (round-trip
the output of =get= to preserve everything). Recorded as *one* =SetRecord=
operation. *Needs =-i=* (a single metarecord) and *requires =-f=/=--force= to run
at all* — a deliberate guard so this rare, scripting-oriented overwrite is not
confused with =field set= (one field).

| Flag          | Description                                                |
|---------------+------------------------------------------------------------|
| =-f=/=--force= | Mandatory: confirms the full-record overwrite             |

*** =mf metarecord [-i|-q] delete=

Deletes the selected metarecords (metadata and all fields); prints the count.
Requires a selector. Confirms for a =-q= selector unless =--force= is passed. A
query goes through the atomic =POST /repos/:repo/query/delete= (one revision); the
confirmation count comes from a =count=+=limit= query.

*** =mf metarecord [-i|-q] field <verb> ...=

Field operations scoped to the selected metarecord(s) (the selector is required).
=get=/=unset= take a bare field =name=; =set=/=add=/=delete= take field specs:

- =get <name>= — prints the field's value(s);
- =set <spec> [<spec> ...]= — replaces *all* rows of the field with the given
  value set (one or several — multi-map; the specs must share one name), in *one*
  log operation per metarecord;
- =add <spec>= — appends one row (inverse of =delete=);
- =delete <spec>= — removes the row(s) equal to the spec's =(name, value)=
  (inverse of =add=); other rows of the same name, and non-matching metarecords,
  are kept;
- =unset <name>= — removes the field *entirely* (all rows → /unknown/), distinct
  from =set name:nothing= (an explicit =Nothing= row).

A UUID selector targets that one metarecord; a predicate applies the write across
every match in one server-side transaction (one revision, one operation per
affected metarecord). =-f=/=--force= is required for reserved =mfr_*= fields.

** =mf field <verb> <id>= (direct row access)

Operates on a single field row by its DB id (as shown by =mf metarecord get=):

- =get <id>= — prints the row (JSON);
- =set <id> <spec>= — sets the row to the field spec's name and value in place,
  *keeping its id*; a differing name renames the row, and the value type is
  validated against the (target) field name's established type;
- =delete <id>= — removes the row.

There is no =mf field add=: new rows are born at the name level (=mf metarecord
field … add=). =-f=/=--force= is required for reserved =mfr_*= fields.

** Log granularity

Every command maps to *one log operation per metarecord it changes* — the unit
the log shows and rollback reverses:

| Command                       | Operation(s)                                  |
|-------------------------------+-----------------------------------------------|
| =metarecord add= / =delete=   | one =CreateRecord= / =DeleteRecord=           |
| =metarecord set=              | one =SetRecord= (all fields before→after)     |
| =field set <name>= (n values) | one =SetField= (all rows of the name)         |
| =field add=                   | one =AppendField=                             |
| =field delete= (by value)     | one =DeleteField= (the removed rows)          |
| =field unset=                 | one =DeleteField= (all rows of the name)      |
| =field set <id>= / =delete=   | one row-scoped op (keeps the row id)          |

A query-scoped write is still *one revision* containing one such operation per
affected metarecord.
