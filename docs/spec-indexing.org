#+TITLE: Metafolder — Indexing & Query Performance Design
#+OPTIONS: toc:2 num:t

* Status

This is a *design* document, not an implemented specification. It records the
analysis and the chosen direction for making filtered + sorted + paginated
queries scale to large repositories (target: ~2 000 000 metarecords). Only the
two baseline indexes of [[*Baseline indexes (implemented)][Baseline indexes]]
are implemented today; everything from [[*The proposed architecture][The
proposed architecture]] onward is forward-looking and tagged =:v2:= or later.

Numbers below come from =crates/bench= run against the two persistent data
folders (=benchmarks/bench_data= ≈ 5 101 and =benchmarks/bench_data_big= ≈
50 201 metarecords); see =benchmarks/results-before-indexes.txt= and
=results-after-indexes.txt=.

* The problem

The data model is EAV (entity–attribute–value): the =field= table holds roughly
one row per field per metarecord, so a 2 M-metarecord repository is ~16 M field
rows. Two distinct costs emerge at scale.

** Pagination was super-linear (fixed)

=mf list= and =mf query= paginate by following a cursor (=PAGE_SIZE= 500), so a
full traversal of 50 k metarecords is 100 round-trips. Before the baseline
indexes, *each page* was itself O(n):

- the query path re-materialised the whole repository universe (the =_repo= CTE)
  per page and full-scanned =field= for the predicate;
- the list path sorted the whole repo in a temp b-tree per page (no keyset
  seek).

Total ≈ O(n²/PAGE_SIZE). Measured: ×10 data → ×64 (query) / ×63 (list) time.

** Bounded retrieval is still linear (open)

Even =LIMIT 100= over HTTP is ~linear in the DB size, not constant: every query
materialises the full =_repo= universe before the limit applies, and the
predicate still scans =field=. Measured: =mfr_path IS PRESENT LIMIT 100=
57 ms @ 5 k → 421 ms @ 50 k (≈ ×7–9). Point lookups (=get=/=path=/=create=)
are already constant.

* The complexity wall

The core theoretical limits, established by analysis (see the design discussion
that produced this document):

1. *Single indexed dimension + limit can be sub-linear.* =X LIMIT k= is
   O(log n + k) when one index yields =X= already in the requested order
   (e.g. =rate IS PRESENT ORDER BY rate=: only films carry =rate=, the index is
   the films pre-sorted).

2. *Conjunction breaks output-proportionality.* =X AND Y LIMIT k= has *no*
   index yielding =X ∩ Y= in order. With only single-attribute indexes,
   intersecting two index-accessible sets costs Ω(min(|X|, |Y|)) in the worst
   case — an adversary can interleave so almost everything must be visited. A
   *small result does not imply little work*: the cost is governed by
   =R = count(value ≥ T)= over the *whole* base (the sort-index scan length)
   and by the size of the filtered set, never by the result size.

3. *Sort independent of filter is the same wall.* =/A AND date ≥ T= where =/A=
   is half the base and =/B= is entirely newer: both access paths touch ~n/2
   (scan =/A= testing the date, or scan =date ≥ T= which includes all of =/B=
   testing membership). O(n/2) for 100 results.

4. *Exact COUNT of a large filtered set is O(m), irreducible in SQL.* A B-tree
   does not store sub-tree cardinalities, so counting a range means enumerating
   it. A top-k can stop early; a COUNT cannot. (Implemented choice: counts are
   *exact*, served O(1) from the result bitmap when the index accelerates the
   query, and only fall back to the O(n) SQL =COUNT(*)= otherwise — see
   [[*Counts are exact (implemented)]]. The "approximate / k+" route below is
   the theoretical escape hatch, not what is built.)

True O(log n + k) for an *independent* conjunction exists only with an index on
the *pair* (a composite). We cannot have one per pair (combinatorial blow-up).
The design therefore aims not to remove the wall but to (a) never hit it by
accident, (b) degrade gracefully and *knowingly* when a query is intrinsically
hard, and (c) shrink the constant of the unavoidable O(n) work by ~3–4 orders
of magnitude.

* Baseline indexes (implemented)

Two indexes, added in the daemon schema (=crates/daemon/src/db.rs=), created by
=init_schema= and back-filled on open for existing repositories
(=ensure_perf_indexes=, idempotent):

| Index                     | Definition                              | Fixes                                |
|---------------------------+-----------------------------------------+--------------------------------------|
| =idx_field_name=          | =field(field_name, metarecord_uuid)=    | predicate seeks the field_name range instead of scanning the EAV table; covers the =DISTINCT metarecord_uuid= projection |
| =idx_metarecord_db_uuid=  | =metarecord_db(db_id, metarecord_uuid)= | listing seeks by keyset, reads rows already ordered — no temp b-tree per page |

=list_entries_page= emits a *conditional* keyset (the cursor predicate is
omitted on the first page) so the composite index can seek; folding it into
=(?=NULL OR uuid > ?)= would defeat the seek.

Result: the LIST path is fixed (=list LIMIT 100= ≈ constant across sizes,
20 ms @ 50 k); the full paginated traversal is now ~linear instead of
quadratic. The QUERY path improved only marginally (still re-materialises =_repo=
per page, limit not pushed down) — addressed below.

* The proposed architecture

The shift in model: stop thinking "scan rows + filter one by one" and move to
*set algebra over compressed bitmaps*, which is the established way (Druid,
Pilosa/FeatureBase, Oracle bitmap indexes) to make ad-hoc filters + ranges
viable over millions of rows.

** Every field is indexed; the encoding is chosen by type

Indexing is *not* opt-in: every field is indexed automatically. Metafolder is
about arbitrary, ad-hoc metadata — which fields a user filters or sorts on is
not knowable in advance, so requiring a declaration would fight the model and
produce "why is this query slow?" surprises. There is no "sortable / filterable"
declaration to maintain.

Because there is no declaration to say how to index a field, the *encoding is
selected automatically from the field's value type* (the =value_type= already
recorded per row):

| Value type(s)                  | Encoding                          | See                                  |
|--------------------------------+-----------------------------------+--------------------------------------|
| =Int= / =Float= / =DateTime=   | bit-sliced index (BSI), range     | [[*Bit-sliced index (BSI) for ordered/range fields][BSI]]               |
| =Bool= / low-cardinality =String= | one bitmap per value           | [[*Bitmaps for categorical predicates][categorical bitmaps]]            |
| =Ref= / =TreeRef=              | reverse (inverted) bitmaps        | [[*Reverse bitmaps for Ref / TreeRef][reverse bitmaps]]                 |
| high-cardinality =String=      | inverted bitmaps for equality; =Matches= stays the REGEXP scan (FTS5 is the v2 acceleration, see [[file:spec-query.org][spec-query.org]]) | |

A field name has exactly one non-=Nothing= value type repository-wide (see
[[file:spec-data-model.org][spec-data-model.org]] → "One value type per field
name"), so the encoding is unambiguous — the =field_type= registry that enforces
the invariant *is* the encoding selector, with no dominant-type inference or
mixed-type fallback. A =retype= operation rebuilds the field's index under the
new encoding.

Since updates are cheap in memory and persistence is deferred (see
[[*Storage model: a derived in-memory cache][Storage model]]), the cost of "index
everything" is *not* write amplification but *resident memory*: ~16 M field
rows at 2 M metarecords, order ~100–200 MB of bitmaps (to be measured). The
gating mechanism is therefore the memory budget, with two fallbacks if it is
exceeded — *opt-out* (the schema may *exclude* a pathological field, e.g. a
near-unique string never usefully filtered), and the on-disk / mmap roaring
regime beyond the in-RAM range (≈ 100 M). Opt-out, never opt-in.

** Bitmaps for categorical predicates

For low-cardinality / categorical fields (=mfr_type=, =mfr_mime=, booleans,
tags, coarse buckets), keep one *roaring bitmap* per value: the set of
metarecord ids satisfying =field = v=. Then:

- =X AND Y= is a bitwise AND, O(n/64) worst case but in machine-word ops;
- =LIMIT k= is the first =k= set bits of the result;
- roaring skips empty blocks, so a small / clustered =X ∩ Y= is well under the
  worst case in practice.

** Bit-sliced index (BSI) for ordered/range fields

A bitmap-per-value is useless for =field ≥ v= over a large domain (date, size,
fine-grained rate). The right encoding is a *bit-sliced index*: store one
bitmap per *bit position* of the (order-preserving encoded) value — ~30 bitmaps
for an integer, ~20 for a day-resolution date, not one per value. Then:

- =field ≥ v= / =≤ v= / range is a comparison walked over the bit-slices,
  O(b · n/64) with b ≈ 30, yielding the matching bitmap;
- it composes with the other bitmaps as just another AND operand;
- bonus: BSI also answers aggregates (SUM, top-k, median) without touching rows,
  which folds part of the COUNT/sort work into the same structure.

Range-encoded bitmaps (cumulative =R_v = {field ≤ v}=, a range = 1–2 bitmaps)
are an alternative for bucketised ranges; BSI is preferred as the general tool
because it also serves aggregates and avoids one-bitmap-per-bucket.

Note that the encoding choice is about *queryability, not bit-width*: a value
that fits in few bits (e.g. a metarecord id in ~21 bits at 2 M) is *not* a BSI
candidate unless it is range-queried. Ids are queried only by equality /
traversal — see the next subsection.

** Reverse bitmaps for Ref / TreeRef

=Ref= and =TreeRef= fields are never range-queried: an id has no meaningful
order, so =target_id ≥ T= is meaningless. The only operation is equality /
traversal — =Follows X= ("which metarecords point to X?"). BSI is therefore the
*wrong* tool here (equality via BSI is ~b slice-ANDs to reconstruct, vs a single
fetch); the bit-width of the id is irrelevant.

The right encoding is a *reverse (inverted) bitmap index*: =target_id → set of
source ids=, i.e. one bitmap per target value, keyed on the referent (for =Ref=)
or the parent id (for =TreeRef=). This is the in-memory equivalent of the
existing SQL =idx_field_reverse=. The "up to 2 M distinct targets" is not a
storage problem: each bitmap is tiny (a node has few direct children; a
metarecord has few referrers) and the total set bits = number of edges ≈ O(n) —
many small bitmaps, not a storage explosion.

Consequence: the *direct* =Follows= ("direct children of =/A=" =
=parent = id(A)=) is a single =bitmap[id(A)]= fetch. The *transitive* query
=FollowsTransitive= is built from those direct fetches by iterative expansion —
see below; the contiguous-id range is only a later optimisation for one case.

** FollowsTransitive by iterative bitmap expansion

The default (v1-bitmap) way to compute all descendants of =/A=, with no
contiguous-id scheme: expand the reverse (direct-children) bitmaps level by
level, unioning into a result bitmap.

#+begin_example
  frontier = { id(A) };  result = ∅
  while frontier ≠ ∅:
      next = ⋃ bitmap[parent = n]  for n in frontier   (k-way union)
      result |= next
      frontier = next                                  (newly discovered nodes)
#+end_example

Each descendant is visited once → *O(sub-tree)*, the same cost as today's
tree-cache walk. Two wins over today: it stays in bitmap algebra — the
descendant set *is* a bitmap that ANDs natively with the other predicate
bitmaps, removing the current hybrid that injects a UUID =IN= clause into SQL —
and it reuses the =TreeRef= reverse index (the tree cache is then only needed to
resolve ="/A" → id(A)=). This is fine for the common case (browsing a normal
folder of thousands of files is sub-millisecond).

** The tree as a range dimension                                  :future:

An optimisation over iterative expansion for *one* case: a *large* sub-tree
combined with a *selective* other predicate. Iterative expansion materialises
the whole sub-tree before the =AND= (e.g. =/A= = half the base with few matching
files is still O(|/A|), even when the result is tiny), and the value predicate
cannot prune the walk — a non-file directory may still contain file descendants.

If metarecord ids are assigned in tree order (DFS pre-order) — or each node
carries a nested-set interval — a sub-tree =/A= becomes a *contiguous id range*,
i.e. a range bitmap. The transitive filter then never enumerates the sub-tree;
it is a range test the planner intersects with the most selective operand:

#+begin_example
  mfr_path ->* /A  AND  mfr_type = file  AND  date ≥ T
  = range_bitmap(/A) AND eq_bitmap(type, file) AND bsi_ge(date, T)
#+end_example

so the cost can drop to O(|files|) instead of O(|/A|).

The catch — and why this is deferred — is that tree-order ids are *not stable
under moves*: relocating a sub-tree breaks contiguity and would force
renumbering a range of ids, whereas the tree cache renames/moves in O(1) today
by repointing the parent. The viable compromise (pre-order ids with gaps, lazy
renumbering, or floating nested-set intervals) is unresolved; until then,
=FollowsTransitive= uses the iterative bitmap expansion above, and this is an
optimisation to prototype and measure later. See [[*Open questions][Open
questions]].

** Sort semantics in one orderable key

The spec sort is not a plain =ORDER BY column=: multi-map (a field with several
values sorts on its min/max), fixed type-group precedence, unknown/Nothing last.
Encode all of it into a single order-preserving BLOB *sort key* — one
type-group byte, then the order-preserving value, a high sentinel for
Nothing/unknown — and store, per (metarecord, field), the *min* and *max* key
(ascending reads min, descending reads max). A plain index / BSI on that key
then reproduces the total order, and =ORDER BY field LIMIT k= is a range scan
for the aligned / sparse cases (the two motivating examples: "latest modified
files", "films by rating").

*Implementation status (increment 3).* =RepoIndex::evaluate_sorted= reproduces
the full sort order (multi-map min/max representative, type-group precedence,
field-missing last, uuid tiebreak; multi-key; =LIMIT=) but as a *shortcut*
stores the per-(metarecord, field) min/max representative in a *separate* map,
rather than folding the order-preserving key into the field's BSI as above. It
sorts the filtered set in O(m log m) — at 57k, ~60 ms vs the SQL window-function
sort's 190 s (the SQL EAV sort is unusable at scale). The cost is *memory*: the
separate rep store is ~29 MB at 57k (vs ~5 MB for all filter bitmaps),
extrapolating to ~1 GB at 2 M — over budget. Folding the sort key into the BSI
(and deriving categorical/reference reps from the existing value partitions)
instead of a standalone store is the pending compaction before 2 M. Cursor
pagination over the sort key is also deferred.

** Counts are exact (implemented)

The original design proposed *approximate* / "k+" counts to dodge the COUNT wall.
That option was **not** adopted: =count= is always **exact**. The bitmap index
returns it in *O(1)* from the result bitmap (=RoaringBitmap::len()=,
=RepoIndex::count_with_roots=) — the COUNT wall does not apply because the result
set is already materialised as a bitmap; the SQL fallback returns an exact
=COUNT(*)= which is O(n) (the wall, for shapes the index does not accelerate).
Counts are opt-in per request (=count: true=, requires =limit=) but never
approximate. The "estimated count" idea is retained below only as the theoretical
escape hatch if a future shape is both unindexable and count-heavy.

** What we explicitly do NOT build

General per-pair composite indexes ("layer 3" for an arbitrary (filter, sort)
pair) — combinatorial blow-up, low value. The *one* narrow exception worth
considering is a tree-scoped sort index — keys =(parent_uuid | path-prefix,
sort_key, uuid)= — so "browse a folder sorted by date/name/size, limit k"
becomes a true O(log n + k) range scan. That single shape is plausibly the
central UX gesture, which is what justifies the composite there and nowhere
else.

*Spatial trees (kd-tree / BKD-tree) as the general primitive* — evaluated and
rejected. A kd-tree indexes points in a *fixed k-dimensional* space; our model
violates nearly every assumption it needs:

- *Open schema ⇒ high dimensionality.* A repo has hundreds of distinct
  =field_name=; which ones are filtered is not knowable in advance (the same
  reason indexing is not opt-in). A kd-tree must fix its axes at build time, and
  past ~10–20 dimensions it degenerates to a linear scan (the query box touches
  almost every node — the curse of dimensionality).
- *Sparse data + three-valued logic.* Each metarecord carries ~8 fields out of
  hundreds. A point needs a coordinate on *every* axis; =Unknown=/=Nothing= have
  no coordinate, and inventing a per-axis sentinel corrupts range queries.
- *Real queries are 1-D.* "latest modified", "films by =rate=" are single-field
  sorts/ranges; in one dimension a kd-tree adds nothing over a B-tree/BSI. The
  wall is *conjunction*, not the mono-field query.
- *Dynamic conjunction over arbitrary fields* is precisely what bitmap set
  algebra gives (intersect any subset of predicates at query time); a kd-tree
  fixes its axes, and putting every field on an axis re-triggers the curse.
- *Write-heavy.* The watcher mutates =mfr_path=/=mfr_mtime=/=mfr_size=
  continuously; classic kd-trees unbalance on insert and need periodic rebuilds,
  whereas bitmaps/BSI update per value in microseconds.
- *Ref / TreeRef are graph, not spatial* — reverse bitmaps are the fit; a
  kd-tree offers nothing there.

The *one* place a kd-tree derivative would genuinely win is a *correlated
multi-dimensional numeric box* (=date ∈ [a,b] AND size ∈ [c,d] AND rate ∈
[e,f]= as a single k-D query, vs intersecting three BSI ranges with intermediate
bitmaps). This is exactly how Lucene uses a *BKD-tree* (a write-friendly,
block-balanced variant that fixes the update problem) — for numeric/geo *point*
fields, *beside* its postings (≈ bitmaps), never as the general index. So a BKD
index stays a possible *specialised, opt-in secondary* for that one motif, to be
revisited only if the bench shows BSI intersection insufficient for it — the
same posture as the tree-scoped sort index above. It is not a candidate to
replace the bitmap/BSI/reverse-bitmap core.

* Storage model: a derived in-memory cache

The bitmaps/BSI are *not* SQL objects loaded per query and rewritten per edit.
They are a *derived in-memory accelerator*, exactly like the existing
=tree_cache=: authoritative data stays in the =field= table; the bitmaps are a
query cache that can always be rebuilt. This dissolves both performance
concerns.

*Implementation status (increments 5–6).* =RepoIndex= lives in =RepoState=
behind a mutex and carries the =log_head.op_id= it was built at. The query route
(=run_query_filter=) consults it only while it reflects the current HEAD, so it
never serves stale results; supported query shapes go through it (filter / sort /
count / paginate) and unsupported ones (=Matches=) fall back to the SQL engine,
cursors included. =Path=-target =Follows=/=FollowsTransitive= are *supported*:
=run_query_filter= resolves each path to its root metarecord through the
(eagerly populated) tree cache and passes the roots in as =PathRoots=, so the
bitmap expansion seeds from the resolved root — =mfr_path ->* "/dir"= + sort is
served fully in memory. The index is built at repository load (best-effort) by
the background warmup — an observable =load= task (phases =tree cache= then
=index=) so the GUI shows a load progress bar (see
[[file:spec-tasks.org][spec-tasks.org]]) — so the first query pays no build cost;
queries arriving before it completes are answered by the SQL fallback meanwhile.
Freshness is maintained by
=RepoIndex::refresh=: when the new HEAD is a *forward extension* of the built
HEAD (the common case — writes appended), it replays that forward delta
incrementally, recomputing only the touched =(metarecord, field)= cells from
their current DB rows (the log's before-snapshots supply the old values to clear
emptied buckets). A rewritten history (rollback / prune), an unrecognised op, or
an oversized delta falls back to a full rebuild — always correct. The replay
happens lazily on the read path, not on the write path, but the effect is the
spec's "single write path updates both": every write advances HEAD, and the next
query reconciles the index to it. Pagination is *keyset*: the cursor encodes the
last entry's sort key (one representative per key) and uuid, so the next page
resumes strictly after it — stable under concurrent inserts/deletes, matching the
SQL engine. Deleted metarecords leave dense-id tombstones on the incremental
path; when they exceed a fraction of the interned ids a refresh rebuilds and
reclaims them. For Int/Float/DateTime fields the sort representative is read
*from the BSI bit-slices* (the order-preserving key inverted back to its value),
not a separate store — so the high-cardinality numeric fields, which dominated
the sort memory, cost nothing extra; a small min/max store is kept only for
categorical and reference fields, which have no BSI. The residual cost is the
tree_ref (=mfr_path=) name representatives (≈ one string per file); folding those
needs the deferred tree-scoped sort index (see [[*What we explicitly do NOT
build]]).

** No per-query transfer (read)

The bitmaps live resident in =RepoState= (next to the tree cache). A query is
in-process bitmap algebra — zero SQLite I/O, zero deserialisation per query.
Size at 2 M: one bitmap ≈ 250 KB dense; a BSI field ≈ a few MB; indexing *all*
fields ≈ ~100–200 MB resident (≈ 16 M field rows) — the budget to measure (see
[[*Every field is indexed; the encoding is chosen by type][Every field is
indexed]]). The cost is at *repository load*, not per query.

** No per-write persistence (write)

- Editing a bitmap in memory is microseconds (a roaring container edit). A BSI
  value change touches ~b ≈ 30 slices → still microseconds, negligible against
  the current ~8 ms/write.
- The trap is *persisting* per write (re-serialising MB per edit). We avoid it:
  =field= is already written transactionally (the single =log::Writer= path), so
  *durability is already provided*. On write we update only the in-memory
  bitmaps (best-effort); a crash loses at most the bitmaps since the last
  snapshot, rebuilt on next load. No real data is ever at risk.

** Load via snapshot + event-log replay

Rebuilding from a full =field= scan at every start is too slow (field scans are
seconds at scale). Instead: snapshot the bitmaps to disk periodically with a
"current as of revision R" marker; on load, read the snapshot and *replay the
operations since R* from the event log
([[file:spec-event-log.org][spec-event-log.org]]), which already exists. Load
cost is bounded by the replay delta, tunable via snapshot frequency.

** The single write path updates both

Writes touch two places — =field= (transactional) and the in-memory bitmaps
(best-effort). The discipline "one write path updates all derived state" is
natural here because =log::Writer= is already that single chokepoint, alongside
the tree cache it already maintains.

* What to measure before committing

The design rests on assumptions that must be validated with =crates/bench=
(and a focused prototype) rather than assumed:

- *Write amplification* of maintaining every field's index in memory per write
  (all ~8 fields of a metarecord), vs the current ~8 ms/write — is the
  microsecond claim borne out end to end?
- *Resident memory budget* at 2 M with *all* fields indexed (the gating concern,
  ~100–200 MB estimated); the regime change toward on-disk / mmap roaring needed
  beyond (≈ 100 M), and whether any opt-out is forced in practice.
- *Load cost* of snapshot + event-log replay vs a cold rebuild, at realistic
  snapshot intervals.
- *Rust libraries*: maturity of =croaring= / =roaring-rs=; whether a BSI layer
  exists or must be built on top of roaring.
- *Real query distributions*: how often queries fall in the easy (aligned /
  sparse / small-m) regime vs the intrinsically hard one, to confirm the
  adaptive planner's default (filter-first) is the common path.

* Open questions

- Id assignment for the deferred tree-as-range trick (=FollowsTransitive=):
  stable tree-order ids vs nested-set intervals vs a separate path-prefix
  encoding — interaction with rename/move (which the tree cache handles in O(1)
  today). This is the unresolved blocker that keeps the optimisation future.
- High-cardinality string fields: an inverted-bitmap equality index is O(n) but
  rarely queried — confirm it is worth keeping vs. opt-out by heuristic.
- Whether the tree-scoped composite sort index is worth its write cost, or
  whether filter-first + BSI is fast enough for folder browsing at 2 M.
- (Resolved) Exact-count UX: counts are exact and opt-in (=count: true=),
  served O(1) from the result bitmap when the query is index-accelerated and by
  the O(n) SQL =COUNT(*)= otherwise. The approximate/background-count idea was
  not adopted.
