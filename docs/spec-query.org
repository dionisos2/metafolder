#+TITLE: Metafolder — Query Language Specification
#+OPTIONS: toc:2 num:t

* Overview

The query system allows filtering metadata metarecords by field values, presence,
and graph relationships. Queries are represented as a recursive JSON structure
and compiled to SQL on the daemon side. The CLI provides a human-friendly DSL
that compiles to this representation.

* Concepts

Queries are represented as a recursive JSON structure (the =Query= enum). The
CLI provides a DSL that compiles to this representation.

** Variants

| Variant             | JSON =type= key       | Description                                    |
|---------------------+-----------------------+------------------------------------------------|
| IsPresent           | =is_present=          | Field exists and is not =Nothing=              |
| IsAbsent            | =is_absent=           | Field exists and is =Nothing=                  |
| IsUnknown           | =is_unknown=          | Field does not exist at all                    |
| Eq / Neq            | =eq= / =neq=          | Equality / inequality comparison               |
| Lt / Lte / Gt / Gte | =lt= / =lte= / ...    | Ordered comparison                             |
| And                 | =and=                 | All operands must match                        |
| Or                  | =or=                  | At least one operand must match                |
| Not                 | =not=                 | Operand must not match                         |
| Follows             | =follows=             | =Ref= or =TreeRef= field: direct referent matches condition    |
| FollowsTransitive   | =follows_transitive=  | =TreeRef= only: metarecord is a descendant of the target (a path, or any metarecord matching a condition) |
| Matches             | =matches=             | String field matches a regex pattern                           |

* Daemon

** Compilation

Queries are compiled to a CTE chain in SQL. Each query node becomes one CTE;
the final SELECT is filtered via =metarecord_db= to isolate the repository and
exclude link metarecords by default.

Exception: =FollowsTransitive= on a =TreeRef= field is executed as a hybrid
operation — the daemon first collects the descendant UUID set iteratively via
the tree cache, then injects it as an =IN= clause into the surrounding SQL query
rather than emitting a CTE.

=Follows= takes either a condition (sub-query) or a path string as right-hand
side. With a condition, it matches metarecords whose =value_uuid= (the =Ref=
referent, or the =TreeRef= direct parent) points to a metarecord satisfying
that condition; this works on both =Ref= and =TreeRef= fields, e.g.
=mfr_path -> (mfr_path = "2021")= matches the direct children of every
directory named =2021=. With a path string (e.g. ="/music/jazz"=, =TreeRef=
semantics only), the path is resolved to a UUID via the tree cache and the
query matches metarecords whose direct parent (=value_uuid=) is that metarecord.
DSL syntax: =field -> "path"= / =field -> (predicate)=.

=FollowsTransitive= is restricted to =TreeRef= fields; applying it to a =Ref=
field is rejected with =400=. The right-hand side is either a path string
(e.g. ="/projects"=) that is resolved to a UUID via the tree cache, or a
condition (sub-query, executed on its own and repo-filtered) whose matches
form the root set — e.g. =mfr_path ->* (mfr_path = "2021")= matches
everything inside any directory named =2021=. All descendants of the root(s)
are then collected iteratively via the tree cache (fetching missing nodes
from the DB step by step, deduplicating across overlapping roots), and the
resulting UUID set is used to filter the query results. The roots themselves
are not part of the result.

String operations on a =TreeRef= row apply to =value_name= (the single name
component) — the same convention everywhere: comparisons (=Eq=/=Neq= and the
ordered operators) with a string operand, =Matches=, and sorting. So
=mfr_path = "2021"= matches every metarecord whose =mfr_path= name is exactly
=2021=, wherever it sits in the tree. Strict =(parent, name)= equality remains
available through =Eq= with a =tree_ref= JSON value (no DSL literal exists for
it; the same pair can be expressed in the DSL as
=field -> "/parent/path" AND field = "name"=).

=Matches= uses a SQLite user-defined function =REGEXP= (registered at connection
setup) backed by the =regex= crate. On a =TreeRef= field, =MATCHES= applies the
regex to =value_name= (the single name component).

** MATCHES performance

In v1, =MATCHES= performs a full scan of the candidate rows with the =REGEXP=
UDF. Benchmarks measured ~650 ms for 50 k metarecords; this is accepted for v1.

** MATCHES acceleration via FTS5                                          :v2:

To accelerate =MATCHES= on large repositories, an FTS5 virtual table with the
trigram tokenizer indexes the textual columns:

#+BEGIN_SRC sql
CREATE VIRTUAL TABLE field_text USING fts5(
    text,                       -- value_text or value_name
    content = '',               -- contentless: text is not stored twice
    tokenize = 'trigram'
);
#+END_SRC

One =field_text= row is maintained per =field= row of type =string=,
=datetime= or =tree_ref= (for =tree_ref=, the indexed text is =value_name=),
with =rowid= equal to =field.id=. The daemon keeps it in sync inside the same
transaction as every write to the =field= table (no triggers, so that the
event log remains the only side-channel).

Query plan for =MATCHES=:

1. Extract the literal substrings of length ≥ 3 that any match of the regex
   must contain (using the =regex-syntax= crate on the parsed pattern).
2. If at least one such literal exists, pre-filter via
   =SELECT rowid FROM field_text WHERE text MATCH '"<literal>"'= (trigram
   index lookup), then apply =REGEXP= only to the surviving rows.
3. If no literal can be extracted (e.g. pattern =^.*$=), fall back to the v1
   full scan.

The pre-filter is purely an optimisation: results are identical to the full
scan in all cases.

** HTTP API

*** =POST /repos/:repo_uuid/query=

#+BEGIN_SRC
POST /repos/:repo_uuid/query
#+END_SRC

Executes a query and returns matching metarecords.

Request body:
#+BEGIN_SRC json
{
  "query": <Query>,
  "select": ["path", "rating"],
  "sort": [{"field": "rating", "order": "desc"}],
  "limit": 500,
  "cursor": "..."
}
#+END_SRC

=query= is a =Query= JSON object (see [[*Query Language]]).

=sort=, =limit= and =cursor= are optional and enable sorted/paginated
results; their semantics (including the response envelope used when =limit=
is present) are defined in the Pagination section of
[[file:spec-data-model.org][spec-data-model.org]].

=select= is optional:
- Absent → response is =["uuid1", "uuid2", ...]=
- =["field1", "field2", ...]= → response is a list of metadata objects
  containing only the requested fields.
- ="*"= → response is a list of full metadata objects (all fields).

Example response with =select=:
#+BEGIN_SRC json
[
  {
    "uuid": "...",
    "version": 3,
    "fields": [
      {"id": 42, "name": "path", "value": {"type": "tree_ref", "value": {"parent": "<parent_uuid>", "name": "bar.mp3"}}},
      {"id": 43, "name": "rating", "value": {"type": "int", "value": 5}}
    ]
  }
]
#+END_SRC

Fields absent from the metarecord (not present in the DB for that metarecord) are simply
omitted from the response — they are not included as =Nothing=.

*** =POST /repos/:repo_uuid/set= (Batch set)

#+BEGIN_SRC
POST /repos/:repo_uuid/set
#+END_SRC

Runs a query server-side and sets a field on every matching metarecord in a single
transaction. More efficient than issuing individual PATCH requests from the
client.

Request body:
#+BEGIN_SRC json
{
  "query": <Query>,
  "name": "rating",
  "value": {"type": "int", "value": 5}
}
#+END_SRC

Writing a protected field name without =force= returns =400=. See [[*Protected fields]].

Response body: ={"updated": <count>}=

* CLI

** Query DSL

The CLI accepts queries as a human-friendly DSL string, parsed by a
hand-written lexer + recursive-descent parser and compiled to the =Query=
JSON representation.

The parser is shared code: it lives in the =core= crate
(=metafolder_core::dsl::parse_query=) and is used by both the CLI and the
GUI (the =metarecord-list= query editor, see [[file:spec-gui.org][spec-gui.org]]).

A higher-level, user-configurable surface that transpiles to this DSL — the
/simplified query language/ — is specified in [[*Simplified query language]]; it
is the default input of the GUI query editor.

*** Grammar

#+BEGIN_SRC
query     := or_expr
or_expr   := and_expr ( "OR" and_expr )*
and_expr  := unary ( "AND" unary )*
unary     := "NOT" unary | atom
atom      := "(" query ")" | predicate

predicate := field "IS" ( "PRESENT" | "ABSENT" | "UNKNOWN" )
           | field cmp_op literal
           | field "MATCHES" string
           | field "->"  ( string | "(" query ")" )
           | field "->*" ( string | "(" query ")" )

cmp_op    := "=" | "!=" | "<" | "<=" | ">" | ">="
literal   := int | float | string | "true" | "false"
field     := identifier
string    := '"' ... '"'
#+END_SRC

- Keywords (=AND=, =OR=, =NOT=, =IS=, =PRESENT=, =ABSENT=, =UNKNOWN=,
  =MATCHES=, =true=, =false=) are case-sensitive uppercase except the boolean
  literals.
- Precedence: =NOT= binds tighter than =AND=, which binds tighter than =OR=;
  parentheses override.
- =field -> "path"= compiles to =Follows= with a path right-hand side
  (=TreeRef= semantics); =field -> (predicate)= compiles to =Follows= with a
  sub-query right-hand side (the referent of a =Ref= row, or the direct
  parent of a =TreeRef= row, must satisfy the predicate).
- =field ->* "path"= / =field ->* (predicate)= compiles to
  =FollowsTransitive= (TreeRef only), with the same two right-hand-side
  forms as =->=.

Examples:

#+BEGIN_SRC
rating > 3 AND genre = "jazz"
mfr_path IS ABSENT
NOT (seen = true OR rating IS UNKNOWN)
mfr_path ->* "/music/jazz" AND title MATCHES "[Ll]ive"
author -> (name = "Coltrane")
mfr_type = "file" AND mfr_path -> (mfr_path = "2021")
#+END_SRC

** =mf query <predicate>=

Compiles the predicate and calls =POST /query=. By default prints matching
UUIDs, one per line. The CLI paginates internally (following =next_cursor=)
and streams the output.

| Flag                       | Description                                                          |
|----------------------------+----------------------------------------------------------------------|
| =--select a,b= / =--select '*'= | Print full metadata objects (JSON array) restricted to the listed fields, or all fields with ='*'= |
| =--sort field[:asc\vert{}desc]=  | Sort results; repeatable for secondary keys                          |
| =--limit N=                | Stop after N results                                                 |
| =--values=                 | With =--select <one field>=: print the field's raw values, one per line (multi-map: one line per row; =nothing= rows skipped; scalars bare, refs as 32-hex uuids, structured values as compact JSON) |
| =--simplified= / =-s=      | Treat =<predicate>= as [[*Simplified query language][simplified-language]] text and expand it to the normal DSL first (via =POST /query/expand=) before compiling |

* Simplified query language

The normal DSL ([[*Query DSL]]) is precise but verbose for everyday filtering
(=genre = "jazz"=, =AND=, =IS PRESENT=…). The /simplified query language/ is an
ergonomic surface that **transpiles to normal DSL text**, which is then parsed
by =parse_query= as usual. Its entire syntax is defined by a grammar the user
can edit — nothing is hard-coded; the shipped conveniences are default grammar
rules, not built-ins.

** Pipeline

#+BEGIN_SRC
simplified text → [ lexer → grammar engine ] → normal DSL text → parse_query → Query IR
#+END_SRC

The grammar engine lives in the =core= crate: a small hand-written
recursive-descent / PEG-style interpreter with output templates (no external
parser library — runtime-loadable grammars are rare and the template layer is
ours regardless). It emits normal DSL *text*; the existing =parse_query= turns
that text into the =Query= IR. The two stages have distinct error surfaces: a
/grammar error/ (the simplified input does not parse under the grammar) is
separate from a /DSL error/ (the produced text is rejected by =parse_query=).

The simplified language is **not** a superset of the normal DSL: a grammar may
accept neither, either, or a mix. Expansion is therefore **not idempotent** on
normal-DSL input. Consumers keep the two explicit — the GUI editor shows both
(see [[file:spec-gui.org][spec-gui.org]], =metarecord-list= panel), and the CLI gates simplified
input behind =mf query --simplified=.

** Configuration

The grammar is a single file in the daemon config directory,
=~/.config/metafolder/query-grammar=. It is installed with an editable default
grammar on first run and never overwritten once edited (same policy as the GUI
config files); deleting it restores the default on next start. It is **global**
(not per-repo) and read by the daemon at startup, so the CLI and GUI share one
grammar. An empty file = no rules = the simplified language accepts nothing. A
malformed grammar is reported and the simplified language is disabled; the
normal DSL is unaffected.

** Lexer (fixed)

A fixed tokenizer (not user-configurable) turns the input into a token stream,
discarding whitespace between tokens:

| Token     | Matches                                                                   |
|-----------+---------------------------------------------------------------------------|
| =WORD=    | =[A-Za-z_][A-Za-z0-9_]*=                                                   |
| =NUMBER=  | integer or float literal                                                  |
| =STRING=  | a double-quoted literal ='"' … '"'= (with escapes)                        |
| =SYMBOL=  | a maximal run of punctuation characters (=:=, =~=, =>=, =>==, =..=, =+=, …) |
| =(= =)=   | grouping, each its own token                                              |

Whitespace is insignificant except as a token separator: =genre:jazz= and
=genre : jazz= tokenize identically, and so do =100MB= and =100 MB=. Maximal
munch on =WORD= gives free word boundaries — a literal ="tag"= in the grammar
never matches the token =tags=.

** Grammar notation

A grammar is a list of named productions; the start production is =query=.

| Form                  | Meaning                                                                       |
|-----------------------+-------------------------------------------------------------------------------|
| =name = ...=          | production definition                                                         |
| =a b=                 | sequence                                                                      |
| =a \vert{} b=         | ordered choice — first alternative that succeeds wins, no backtracking afterwards |
| =a*= =a+= =a?=        | repetition: zero+, one+, optional (adjacent)                                  |
| =a ** s= =a ++ s=     | repetition separated by =s=: zero+, one+                                      |
| =( ... )=             | grouping                                                                       |
| ="lit"=               | matches one token whose text equals =lit=                                     |
| =WORD= =NUMBER= =STRING= | matches a token of that class                                              |
| =label:item=          | binds =item='s match to =label= (a single value, or the list for a repetition) |
| =item => template=    | the production's output template                                              |

A production **without** =>= returns the raw source text it matched (so =STRING=
keeps its quotes and =WORD= stays bare).

** Template language

Right of =>=, a template is literal text with two kinds of holes:

- =$name= — interpolate the capture =name= (its text) verbatim.
- ={ expr }= — evaluate =expr= and interpolate the result.

The base type is **text**; every capture is text. There is **no implicit
coercion** — converting text to a number is explicit, which keeps text fidelity
(e.g. ="007"= stays ="007"=) and leaves room for future text/date functions.
Inside ={ }=:

| Construct           | Notes                                                                            |
|---------------------+----------------------------------------------------------------------------------|
| =$name=             | the capture's text                                                               |
| =+ - * / %=         | arithmetic — operands must be numbers                                            |
| =num(x)=            | parse text =x= as a number (error if not numeric)                               |
| =str(x)=            | render =x= as a DSL string literal — idempotent quoting (quote a bare word, leave an already-quoted string) |
| =join(sep, list)=   | join a repetition's captured list with =sep=                                     |

Arithmetic is integer unless an operand is a float; the result is reformatted to
text. Text concatenation needs no ={ }= — adjacent literals and =$captures= in
the template already concatenate. The v1 set above is intentionally minimal;
more functions are localized engine additions that do not break existing
grammars (no migration; pre-v0.2).

** Default grammar

Installed and fully editable. It routes =:= by token class so typed values keep
their DSL type (=rating:4= → =rating = 4=, not ="4"=; =seen:true= → =seen = true=):

#+BEGIN_SRC
query = q:or                       => $q
or    = items:and ++ ("OR" | "+")  => {join(" OR ", $items)}
and   = items:unary +              => {join(" AND ", $items)}    # juxtaposition
unary = "!" u:unary                => NOT $u
      | a:atom                     => $a
atom  = "(" q:query ")"            => ($q)
      | p:predicate                => $p

# the part edited most of the time: the predicate alternatives
predicate =
    f:field ":"  v:STRING   => $f = $v
  | f:field ":"  n:NUMBER   => $f = $n
  | f:field ":"  "true"     => $f = true
  | f:field ":"  "false"    => $f = false
  | f:field ":"  w:WORD     => $f = {str($w)}
  | f:field "~"  v:value    => $f MATCHES {str($v)}
  | f:field ">=" n:number   => $f >= $n
  | f:field "?"             => $f IS PRESENT
  | "under" v:value         => mfr_path ->* {str($v)}
  | "tag"                   => tag = "test"      # bare-word alias, kept as a fallback
  | "fav"                   => rating >= 4

# field aliases: map a few names, pass the rest through
field  = "path" => mfr_path
       | "size" => mfr_size
       | w:WORD => $w
value  = STRING | WORD
number = n:NUMBER "MB"   => {num($n) * 1048576}
       | n:NUMBER "mins" => {num($n) * 60}
       | n:NUMBER        => $n
#+END_SRC

Ordering matters under ordered choice: the operator forms (=:=, =~=, =?=) precede
the bare-word aliases (=tag=, =fav=), so =tag:jazz= and =tag?= still parse as
predicates and only a lone =tag= triggers its alias.

** Validation

At load the grammar is checked: every referenced production exists, the start
production =query= is present, and **left recursion is rejected** (=or = or ...=
would loop forever) with an error naming the offending production. Grammar and
expansion errors carry the position in the simplified input.

** Dates                                                                  :v2:

Relative or absolute date filters (e.g. "modified in the last 10 minutes")
require a real datetime literal in the normal DSL first: =mfr_mtime=/=mfr_btime=
are stored as =datetime= text, and the DSL has no datetime literal, so a string
comparison (=mfr_mtime > "..."=) is typed =string= and matches no =datetime=
row. Until that lands (its own work item — datetime should stop being a plain
string), the default grammar ships **no** date rules. Numeric unit conversion
(sizes, durations as integers) is unaffected and available in v1.

** HTTP API

*** =POST /query/expand=

Expands simplified text to normal DSL text using the loaded grammar. Global
(not repo-scoped), since the grammar is global.

Request body:
#+BEGIN_SRC json
{ "simplified": "<text>" }
#+END_SRC

Response body:
#+BEGIN_SRC json
{ "dsl": "<normal DSL text>" }
#+END_SRC

On a grammar/expansion error, returns =400= with ={"error": "..."}= (the GUI
shows this under the simplified input; a subsequent =parse_query= failure on the
produced text is a separate error shown under the normal-DSL input).
