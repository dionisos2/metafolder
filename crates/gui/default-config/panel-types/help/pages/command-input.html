Command input

A single-line text input, always visible just above the status bar. It is how you run commands, type shell commands, and answer scripts. Press the configured key (by default :, bound to command-input:activate) to give it the keyboard focus; Escape removes the focus while keeping whatever you had typed.

There is one command input; it targets the focused slot's workspace. Its content (text and cursor position) is per-workspace: switching the focused slot to another workspace restores that workspace's own draft, so a half-typed command is never lost.

Two kinds of input

  • A command — a command name optionally followed by space-separated parameters (for example panel:set-type file).
  • A shell command — anything prefixed with ! (for example !ls -la). See Running shell commands below.

Commands and autocomplete

As you type a command name a suggestion list appears, filtered fuzzily and case-insensitively: the draft is split on whitespace and the terms must appear in the command name in order — so con def matches like the regular expression .*con.*def.*. Names that start with the first term rank first.

  • The first (best) suggestion is always highlighted; Up/Down move the highlight.
  • Tab writes the highlighted suggestion into the input without running it — handy before adding parameters.
  • Enter always runs exactly what is typed, never the highlighted suggestion (one is always highlighted, so running the selection would be surprising).
  • Parameters follow the name, separated by spaces. Because spaces are also the fuzzy term separators, the suggestion list simply empties once your typed parameters stop matching any command name — that is expected. Wrap a parameter in double quotes to keep its spaces, e.g. tab:rename "My music".

Editing keys

While the input has focus, the editing:* commands act on it (default bindings shown):

  • Enter — editing:confirm (run the command).
  • Escape — editing:unfocus (keep the draft, drop focus).
  • Ctrl+q — editing:discard (clear the draft and unfocus).
  • Ctrl+a / Ctrl+e — editing:goto-line-start / editing:goto-line-end.

These same keys also drive a focused text field inside a panel (a query box, an inline editor), not only the command input.

Running shell commands with !

A line starting with ! is run as a subprocess (via sh -c) in the background. Its stdout/stderr go to two places:

  • the message panel (the command is echoed as $ command, each output line timestamped), and
  • the terminal from which the GUI was launched.

So that you see the output immediately, running a ! command switches the focused slot to the message panel — unless a message panel is already visible in the workspace, in which case nothing moves.

Placeholders: passing the selection to scripts

A ! command line may reference the selected metarecord (the cursor metarecord in the metarecord list, file manager or detail panel — workspace.selected_metarecord) through %-tokens. They are expanded just before the command runs, so you can hand a metarecord's identity straight to your own scripts:

TokenExpands to
%uthe selected metarecord's UUID
%vits version number
%<field>the value of a single-valued scalar field
%<field>:patha TreeRef field resolved to its absolute path
%%a literal %

Examples (← is what actually runs):

!echo %u %v           ←  echo '8f3…a1' '7'
!vlc %mfr_path:path   ←  vlc '/music/My Track.flac'
!grep %artist notes   ←  grep 'Miles Davis' notes
!printf '100%%done'   ←  printf '100%done'

Placeholder rules

  • <field> is a field name: a letter or _ followed by letters, digits or _. A % that is not followed by % or a field name is left untouched, so 50% off stays literal.
  • %<field> works for scalar values only: string, integer, float, datetime (its ISO-8601 text) and boolean (true/false). For a TreeRef field such as mfr_path use the :path form.
  • Every substituted value is shell-quoted, so paths and values with spaces or quotes are passed safely as a single argument.
  • Substitution is strict: if a token cannot be resolved the command is not run and the reason is shown in the status bar. This happens when nothing is selected, the field is absent, the field is multi-valued, the field is empty (Nothing), or a bare %<field> names a TreeRef (use :path).
  • Only ! shell commands are expanded — named commands are passed through verbatim, so a literal % in a command parameter is safe.

The message log

Every command you dispatch is echoed to the message panel as > name args before it runs (basic editing primitives opt out, to avoid noise), and shell output as $ command followed by its lines. Clear the log with message:clear.

Answering scripts

A script driving the GUI can ask a question (POST /gui/prompt): the command input grabs the focus and shows the script's prompt, optionally with autocomplete. Type your answer and press Enter, or send it explicitly with answer:send.
